# AI-004 - Unsanctioned AI CLI Execution

> **Detection ID:** AI-004
> **Status:** Active
> **Risk Score:** 25 (user) + 15 (system) = 40 total
> **MITRE ATT&CK:** T1059.001 - Command and Scripting Interpreter: PowerShell (Execution)
> **Data Model:** Endpoint.Processes
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Exploitation
> **Last Updated:** 2026-02-13

---

## Overview

This detection identifies execution of AI CLI tools on endpoints. It covers a broad set of AI command-line tools including codex, claude, gemini, gh-copilot, aider, sgpt, llm, tgpt, mods, fabric, aichat, amazon-q, openai, and cline. CLI tools can send data to AI providers directly from the terminal, bypassing web proxy controls and DLP policies that typically monitor browser-based traffic.

## Detection Logic

### Plain English

1. Query `Endpoint.Processes` for process creation events matching known AI CLI process names.
2. Enrich via `ai_process_filter` macro to resolve the tool name, provider, and usage type from `ai_tool_processes.csv`.
3. For ambiguous process names (e.g., `claude.exe` which could be a desktop app), apply additional logic: filter to `usage_type="cli"` or validate the parent process is a shell.
4. Map unresolved process names to providers using eval logic.
5. Exclude sanctioned entities via `ai_unsanctioned_filter`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User who launched the CLI | `jsmith` |
| `src` | Endpoint hostname | `DEV-WS-42` |
| `process_name` | AI CLI tool name | `claude` |
| `parent_process_name` | Parent process | `zsh` |
| `provider` | AI provider | `Anthropic` |

## Triage Steps

> **Estimated triage time:** 5-10 minutes

1. **Validate the alert:** Confirm the process name and user.
2. **Identify the user:** Check their role and whether CLI tool usage is expected for their position.
3. **Review the command line:**
   ```spl
   | tstats summariesonly=t values(Processes.process) as command values(Processes.parent_process) as parent
     from datamodel=Endpoint.Processes where Processes.user="$user$" Processes.dest="$src$" Processes.process_name="$process_name$" earliest=-24h
   | table command parent
   ```
4. **Check for related detections:** Look for AI-008 (scripted invocation) and AI-010 (burst activity).
5. **Determine sanctioned status:** Verify if the user should be using this tool.

---

## True Positive Indicators

- User is not a developer or has no documented need for AI CLI tools.
- The CLI tool is not on the organization's approved software list.
- Large amounts of data are being piped into the CLI tool.
- The tool is being used from a production server or sensitive endpoint.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Developer using an approved AI CLI for daily work | High | Allowlist the user/group for the specific provider |
| Process name collision (e.g., `llm` could be a different tool) | Low | Validate via process path and original_file_name |
| Evaluation/trial usage during an approved AI pilot | Medium | Allowlist pilot group with expiration |

---

## Response Actions

### If confirmed True Positive

1. Notify the user and their manager about the policy violation.
2. Verify what data the user may have sent via the CLI.
3. If the CLI tool is not approved, request removal from the endpoint via IT.
4. Document the finding.

### If Benign True Positive

1. Add the user to `ai_sanctioned_entities.csv` with the specific provider and an expiration date.
