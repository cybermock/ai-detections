# AI-040 - User Peer Group AI Usage Anomaly

> **Detection ID:** AI-040
> **Status:** Active
> **Risk Score:** 30 (user)
> **MITRE ATT&CK:** T1567.002 - Exfiltration Over Web Service: Exfiltration to Cloud Storage (Exfiltration)
> **Data Model:** Web
> **Schedule:** `30 4 * * *` (daily at 4:30 AM)
> **Lookback Window:** `-7d@d` to `now`
> **Kill Chain Phase:** Exfiltration
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies users whose AI service usage significantly deviates from their department peer group. Rather than comparing a user against a global baseline (which would flag every developer as anomalous compared to HR staff), this detection normalizes usage within each department, surfacing individuals whose AI activity is unusually high compared to colleagues in the same role and function.

Peer group analysis is a powerful UEBA technique because it automatically accounts for department-specific usage patterns. An engineering department may average 500 AI requests per day, while the legal department averages 10. A legal team member making 100 AI requests would be a major anomaly within their peer group even though it falls far below the engineering average. This normalization catches insider threats and policy violations that absolute thresholds would miss.

The detection uses z-score analysis to measure how many standard deviations a user's daily AI requests and bytes uploaded fall from their department's mean. When a user exceeds the configurable z-score threshold (default: 2.5 standard deviations), the anomaly fires. The 7-day lookback window provides enough data to establish a stable peer group baseline while remaining responsive to recent behavioral changes. The detection runs daily at 4:30 AM to capture the previous day's complete activity.

## Detection Logic

### Plain English

1. Query the `Web` data model for AI service traffic over the past 7 days, aggregated by user, URL domain, and day.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter`.
3. Calculate daily request counts and byte volumes per user and provider.
4. Look up each user's department from `identity_lookup_expanded`.
5. Calculate department-level statistics using `eventstats`: average requests, standard deviation, average bytes, and standard deviation -- grouped by department and day.
6. Load the peer group z-score threshold from `ai_detection_config.csv` (default: 2.5).
7. Calculate per-user z-scores for both request count and byte volume.
8. Filter for users exceeding the z-score threshold on either metric.
9. Filter to only the most recent day's results.
10. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
11. Tag the event with `detection_id=AI-040`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User with anomalous peer group AI usage | `lwright@corp.com` |
| `department` | User's department | `legal` |

| `dept_avg_requests` | Department average request count | `12` |

## Triage Steps

> **Estimated triage time:** 10-15 minutes

### Step 1: Validate the alert and note the anomaly severity

```spl
index=risk search_name="AI RBA - AI-040: User Peer Group AI Usage Anomaly" risk_object="$user$" earliest=-7d
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the z-scores for requests and bytes. Higher z-scores indicate more extreme deviation.
- Check if this user has triggered the peer group anomaly on multiple days.

### Step 2: Identify the user and their department

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Note the department -- this determines the peer group.
- Is the user in a role that would justify significantly higher AI usage than peers?

### Step 3: Compare the user against their peer group

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.url_domain=* earliest=-7d latest=now by Web.user Web.url_domain _time span=1d
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(count) as daily_requests sum(bytes_out) as daily_bytes by _time user
| lookup identity_lookup_expanded identity as user OUTPUTNEW department
| where department=lower("$user_department$")
| stats avg(daily_requests) as avg_req stdev(daily_requests) as stdev_req max(daily_requests) as max_req by user
| sort -avg_req
```

- Where does the flagged user rank within their department?
- Is the user clearly an outlier, or is the entire department's usage variable?

### Step 4: Examine the user's AI usage trend

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.user="$user$" earliest=-30d latest=now by Web.url_domain _time span=1d
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| timechart span=1d sum(count) as requests sum(bytes_out) as bytes_out
| eval mb_out=round(bytes_out/1024/1024,2)
```

- Is the anomalous day a sudden spike or part of a gradual increase?
- A sudden spike is more suspicious than a gradual ramp-up.

### Step 5: Determine what the user was doing with AI

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" earliest=-2d latest=now
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(count) as requests sum(bytes_out) as total_bytes values(urls) as urls values(user_agents) as agents by provider
| eval mb_out=round(total_bytes/1024/1024,2)
```

- Which AI providers drove the anomalous usage?
- Was data being uploaded (high bytes_out) or just browsing (low bytes_out with high request count)?
- API access or web interface?

### Step 6: Check for related detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-040 + AI-007 (high volume) = anomalous usage is also volume-breaching.
- AI-040 + AI-041 (time-of-day anomaly) = usage pattern is anomalous in both volume and timing.
- AI-040 + AI-017 (individual volume anomaly) = anomalous against both peer group and self-baseline.

### Step 7: Make a determination

- **True Positive:** User's AI usage significantly exceeds their department peers, indicating potential data exfiltration or unauthorized AI adoption.
- **Benign True Positive:** User has a unique role within their department that legitimately requires more AI usage (e.g., the only data analyst in the legal department).
- **False Positive:** Small department size causes high z-scores from normal variation; department assignment is incorrect in identity lookup.

---

## True Positive Indicators

- Z-score exceeds 4.0, indicating extreme deviation from peer group.
- User's AI usage is 10x or more above the department average.
- High bytes_out indicates the user is uploading significant data, not just browsing.
- No business justification for AI usage at this level compared to peer roles.
- Combined with other anomaly detections (AI-041, AI-017) in the same timeframe.
- User recently changed roles or gained access to new data sources.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Small department (fewer than 5 people) where one user's normal usage skews statistics | High | Require minimum department size of 5+ for peer group analysis |
| User has a unique AI-dependent role within an otherwise non-AI department | Medium | Create role-specific peer groups or allowlist unique roles |
| Department assignment is incorrect in identity_lookup_expanded | Medium | Audit identity lookup department assignments quarterly |
| New employee's first-week exploration inflates relative to peers' routine | Low | Exclude users with fewer than 7 days of activity from peer group comparison |
| Seasonal project causes legitimate temporary AI usage spike | Medium | Check for project context; increase z-score threshold if seasonal spikes are common |

## Response Actions

### If confirmed True Positive

1. **Contain:** If data exfiltration is suspected, restrict the user's AI service access via proxy policy.
2. **Notify:** Contact the user's manager to discuss the anomalous AI usage pattern and verify business justification.
3. **Investigate:** Determine what data was being processed through AI services and whether it falls within the user's authorized scope.
4. **Remediate:** If unauthorized, enforce AI usage policies and require approval for continued AI access. If the usage reflects a new legitimate need, formally approve and document it.
5. **Document:** Record the peer group comparison, z-scores, and business justification assessment.

### If Benign True Positive

1. If the user's role justifies higher AI usage, document the exception and consider adjusting the peer group.
2. Add the user to `ai_sanctioned_entities.csv` if they need ongoing AI access.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Increase the `peer_group_zscore` threshold from 2.5 to 3.0 if too many benign anomalies fire.
- Enforce a minimum department size (the detection already requires sufficient datapoints from `eventstats`).
- Create sub-department peer groups for large departments with diverse roles (e.g., separate "legal-analysts" from "legal-attorneys").

### Reviewing peer group z-score distribution

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.url_domain=* earliest=-7d latest=now by Web.user Web.url_domain _time span=1d
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(count) as daily_requests sum(bytes_out) as daily_bytes by _time user
| lookup identity_lookup_expanded identity as user OUTPUTNEW department
| eval department=lower(coalesce(department,"general"))
| eventstats avg(daily_requests) as dept_avg stdev(daily_requests) as dept_stdev by department _time
| eval z_score=if(dept_stdev>0,round((daily_requests-dept_avg)/dept_stdev,2),0)
| where _time>=relative_time(now(),"-1d@d")
| stats max(z_score) as max_z avg(z_score) as avg_z by department
| sort -max_z
```

This shows the z-score distribution across departments, helping calibrate the threshold.

### Adjusting thresholds

Update `lookups/ai_detection_config.csv`:
```csv
control_name,value,notes
peer_group_zscore,3.0,Adjusted from 2.5 to reduce false positives in small departments
```
