# AI-030 - Prompt Injection Indicators In Web Uploads

> **Detection ID:** AI-030
> **Status:** Active
> **Risk Score:** 45 (user) + 25 (system)
> **MITRE ATT&CK:** T1190 - Exploit Public-Facing Application (Initial Access)
> **Data Model:** Data_Loss_Prevention
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Staging
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies known prompt injection patterns in file content uploaded to AI services, using DLP or proxy content inspection as the telemetry source. It matches against 15 distinct prompt injection signatures (PI-001 through PI-015) that represent well-documented attack patterns for manipulating large language models.

Prompt injection is the primary attack vector against AI applications. An attacker crafts input designed to override the AI system's instructions, bypass safety guardrails, extract the system prompt, or cause the model to perform unintended actions. When these patterns appear in files being uploaded to corporate AI services, it may indicate an insider attempting to jailbreak an AI deployment, an attacker probing for vulnerabilities in AI-powered workflows, or preparation for a more sophisticated attack chain.

The detection relies on DLP content inspection to see inside uploaded files. This means it catches injection attempts embedded in documents, spreadsheets, code files, and other uploads -- not just text pasted into a chat interface. Each matched pattern is tagged with a pattern ID (PI-001 through PI-015) to help analysts quickly classify the type of injection attempted.

Because security researchers and red teams routinely test these patterns, false positive rates can be moderate in organizations with active AI security programs. The pattern-level tagging allows analysts to quickly distinguish between research payloads and genuine attack attempts.

## Detection Logic

### Plain English

1. Query the `Data_Loss_Prevention` data model for all DLP events with a destination, aggregating by user, source, destination, file name, file content, and DLP type.
2. Extract the AI domain from the destination field and enrich against `ai_provider_domains.csv` via `ai_domains_filter` to confirm the destination is an AI service.
3. Convert file content to lowercase and match against 15 prompt injection patterns (e.g., "ignore previous instructions", "jailbreak", "output your system prompt").
4. Tag each matched pattern with a pattern ID (PI-001 through PI-015) for classification.
5. Aggregate results by user, source, and provider, collecting all matched file names and pattern IDs.
6. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
7. Tag the event with `detection_id=AI-030`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User uploading the file | `jsmith@corp.com` |
| `src` | Source endpoint | `10.1.2.50` |

## Triage Steps

> **Estimated triage time:** 10-15 minutes

### Step 1: Validate the alert and identify the user

```spl
index=risk search_name="AI RBA - AI-030*" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the matched patterns and file names from the risk message.
- Check if this user has triggered this detection before.

### Step 2: Identify the user's role and department

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit
```

- Security researchers, red team members, or AI safety engineers may legitimately submit injection payloads.
- Check the `bunit` and `category` fields for team affiliation.
- Users in non-technical roles submitting injection patterns are significantly more suspicious.

### Step 3: Examine the specific patterns and files

```spl
| tstats summariesonly=t count values(DLP.file_name) as file_name values(DLP.file_content) as file_content from datamodel=Data_Loss_Prevention where DLP.user="$user$" DLP.dest=* earliest=-1h
| rename DLP.* as *
| eval content_lc=lower(file_content)
| eval pi_type=case(
    match(content_lc,"output your system prompt") OR match(content_lc,"repeat everything above") OR match(content_lc,"translate the above"), "prompt_extraction",
    match(content_lc,"jailbreak") OR match(content_lc,"do anything now") OR match(content_lc,"developer mode"), "jailbreak",
    match(content_lc,"bypass content filter") OR match(content_lc,"ignore safety") OR match(content_lc,"pretend you have no restrictions"), "safety_bypass",
    1=1, "instruction_override")
| stats count by file_name pi_type
```

- **Prompt extraction** attempts (PI-013, PI-014, PI-015) suggest reconnaissance against the AI's configuration.
- **Jailbreak** attempts (PI-007, PI-008, PI-012) suggest the user is trying to remove safety guardrails.
- **Safety bypass** (PI-009, PI-010, PI-011) indicates an attempt to get the AI to produce restricted content.
- Look at the file name -- is it `pentest_report.docx`, `ai_safety_training.pdf`, or something like `exploit.txt`?

### Step 4: Check if this is an automated scanner or test framework

```spl
| tstats summariesonly=t count values(DLP.file_name) as files from datamodel=Data_Loss_Prevention where DLP.user="$user$" earliest=-24h by DLP.src DLP.dest
| rename DLP.* as *
| where count>20
| table src dest count files
```

- High volume from a single source with many distinct files suggests an automated security scanner.
- Security tools like Burp Suite, OWASP ZAP, or custom AI security scanners will generate many injection patterns.

### Step 5: Check if the AI service processed the injection

```spl
| tstats summariesonly=t count sum(Web.bytes_in) as bytes_in sum(Web.bytes_out) as bytes_out from datamodel=Web where Web.user="$user$" earliest=-1h by Web.url_domain Web.http_method Web.status
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| table provider url_domain http_method status bytes_out bytes_in count
```

- A successful POST (status=200) with substantial bytes_in means the AI service likely processed the injected content and returned a response.
- A blocked or failed request (status=403, 429) means the injection was stopped upstream.

### Step 6: Correlate with other AI risk events

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-030 + AI-020 (DLP violation) = injection embedded in sensitive document upload.
- AI-030 + AI-007 (high volume) = bulk injection testing campaign.
- AI-030 + AI-015 (after-hours) = potential evasion behavior.

### Step 7: Make a determination

- **True Positive:** Non-researcher user uploading injection patterns targeting a production AI service.
- **Benign True Positive:** Security researcher or red team testing AI guardrails with known payloads.
- **False Positive:** AI safety training materials, blog posts about prompt injection, or security tool documentation.

---

## True Positive Indicators

- User is not a member of security, red team, or AI safety teams.
- Multiple distinct pattern types matched (e.g., both jailbreak and prompt extraction in the same file).
- File names suggest intentional crafting (not documentation or training materials).
- Injection patterns appear in otherwise normal-looking business documents.
- The AI service returned a successful response after the upload (injection may have been processed).
- User has no prior history of security research or testing activities.
- Activity occurred outside normal business hours.
- Correlated with other AI risk events (DLP violations, high upload volume).

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Security researcher testing AI guardrails with known injection payloads | High | Allowlist red team members with provider scope and expiration in `ai_sanctioned_entities.csv` |
| Red team exercise targeting corporate AI deployments | High | Coordinate with red team lead; add time-bounded allowlist entries for exercise duration |
| AI safety training materials containing example injection patterns | Medium | Identify and exclude known training document hashes or file names |
| Blog posts or articles about prompt injection copied into AI services | Medium | Check file content context -- educational content typically includes surrounding explanation |
| Automated security scanner (Burp, ZAP) testing AI endpoints | Medium | Allowlist scanner source IPs or service accounts |
| User pasting security advisory content that mentions injection techniques | Low | Look at the full file content for advisory formatting and CVE references |

## Response Actions

### If confirmed True Positive

1. **Block:** Immediately block the upload if the DLP system supports inline blocking. If the upload already succeeded, contact the AI service provider about the injection attempt.
2. **Investigate intent:** Determine what the user was trying to extract or accomplish. Prompt extraction attempts may indicate reconnaissance for a larger attack. Jailbreak attempts may indicate an attempt to generate harmful content.
3. **Check AI response:** If the AI service processed the injection, assess whether the injection succeeded. Did the AI reveal its system prompt? Did it produce restricted content?
4. **Contain:** Suspend the user's access to AI services pending investigation. If API keys were used, revoke them.
5. **Escalate:** Notify the AI security team and the user's manager. For successful injections against production AI systems, notify the application owner.
6. **Document:** Record the specific patterns matched, the target AI service, and whether the injection was successful.

### If Benign True Positive

1. Add the user or team to `ai_sanctioned_entities.csv` with appropriate scope and expiration.
2. If the user is a security researcher, recommend using a designated test environment rather than production AI services.
3. Document the business justification and close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Allowlist security and red team groups at the team level for all AI providers during authorized testing windows.
- Consider creating a separate allowlist for users who routinely handle AI security documentation.
- If your organization has an AI security testing program, coordinate with that team to establish pre-approved testing windows.

### Identifying your baseline pattern match rate

```spl
| tstats summariesonly=t count from datamodel=Data_Loss_Prevention where DLP.dest=* earliest=-7d by DLP.user DLP.dest _time span=1d
| rename DLP.* as *
| eval dest_lc=lower(dest)
| lookup ai_provider_domains domain as dest_lc OUTPUTNEW provider enabled
| where enabled=1
| eval content_lc=lower(file_content)
| stats count by user _time
| timechart span=1d dc(user) as unique_users sum(count) as total_events
```

### Adding pattern-specific suppression

If a specific pattern generates excessive false positives (e.g., PI-012 "developer mode" appearing in legitimate developer documentation), consider adding a pattern-level suppression. This requires modifying the detection search to exclude that pattern.
