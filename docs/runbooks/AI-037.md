# AI-037 - AI Agent Framework Execution

> **Detection ID:** AI-037
> **Status:** Active
> **Risk Score:** 40 (user) + 25 (system)
> **MITRE ATT&CK:** T1059 - Command and Scripting Interpreter (Execution)
> **Data Model:** Endpoint.Processes
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Execution
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies the execution of autonomous AI agent frameworks on corporate endpoints. Unlike simple AI CLI tools that relay a single prompt to a language model, autonomous agent frameworks can decompose goals into multi-step plans, execute code, browse the web, read and write files, call external APIs, spawn sub-agents, and iterate without human oversight. This behavior represents a qualitatively higher risk profile than passive AI tool usage.

The frameworks covered include AutoGPT, CrewAI, LangChain (serve mode), AutoGen, MetaGPT, BabyAGI, SuperAGI, AgentGPT, OpenDevin/OpenHands, SWE-agent, Devika, GPT-Engineer, and Smol Developer. These tools are designed to autonomously accomplish objectives — which means a single invocation can result in dozens of downstream actions: filesystem traversal, outbound network connections, subprocess execution, and credential use — all without additional user interaction.

The detection fires on process name match against the `Endpoint.Processes` data model, suppressed per user/system/process combination for one hour. It is additive to the RBA risk score rather than generating a standalone notable, meaning severity accumulates when agents are run alongside other suspicious behaviors.

## Detection Logic

### Plain English

1. Query `Endpoint.Processes` via `tstats` with `summariesonly=t` for processes whose `process_name` matches any of the 28 tracked agent binary names (both POSIX and `.exe` variants).
2. Aggregate by `user`, `dest` (host), and `process_name`, collecting `firstTime`, `lastTime`, raw `process` command line, and `parent_process_name`.
3. Rename `Processes.*` fields to flat field names and coalesce nulls in `user` and `src`.
4. Map each matching process name to its canonical `provider` label (AutoGPT, CrewAI, etc.) using a `case()` expression; unrecognized matches fall back to `"AI Agent"`.
5. Pass through `ai_unsanctioned_filter(user, src, provider)` to exclude allowlisted users and systems.
6. Tag the event with `ai_activity="agent_framework_execution"`, `detection_id="AI-037"`, `mitre_attack="T1059"`, and `ai_kill_chain_phase="execution"`.
7. Assign risk: 40 to the `user` risk object, 25 to the `src` (system) risk object.

## Triage Steps

> **Estimated triage time:** 15-25 minutes

### Step 1: Validate the alert and confirm execution

```spl
index=risk search_name="AI RBA - AI-037: AI Agent Framework Execution" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score search_name process_name provider src
| sort -_time
```

- Confirm the alert fired for the expected user and host.
- Note `process_name` and `provider` — some frameworks (BabyAGI, AutoGPT with `--continuous`) are far more dangerous than others because they operate without per-step human approval.
- Check `lastTime` vs `firstTime`; a long-running agent may have been executing for minutes before suppression.

### Step 2: Retrieve full command-line arguments and parent process

```spl
| tstats summariesonly=t allow_old_summaries=t count min(_time) as firstTime max(_time) as lastTime
    values(Processes.process) as process values(Processes.parent_process_name) as parent_process_name
    values(Processes.parent_process) as parent_process
  from datamodel=Endpoint.Processes
  where Processes.user="$user$" Processes.dest="$src$"
    Processes.process_name IN ("autogpt","crewai","langchain","autogen","metagpt","babyagi",
      "superagi","agentgpt","opendevin","openhands","swe-agent","devika","gpt-engineer","smol-developer")
  earliest=-1h
| rename Processes.* as *
| table firstTime lastTime process_name parent_process_name parent_process process
```

- **Parent is a terminal or IDE** (`bash`, `zsh`, `code`, `pycharm`): likely developer activity; investigate further for sandboxing.
- **Parent is a CI/CD runner** (`jenkins`, `github-actions-runner`, `buildkite-agent`): likely an automated pipeline — confirm the pipeline is authorized.
- **Parent is a scheduled task or cron**: unattended autonomous execution; treat as higher risk.
- **Command-line flags to note:** `--continuous`, `--no-user-input`, `--skip-reprompt`, `--gpt4`, or long context prompts embedded in shell arguments indicate hands-off autonomous operation.

### Step 3: Identify the user and assess authorization

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit watchlisted
```

- Is the user an engineer, data scientist, or researcher with a plausible use case?
- Is the user on a watchlist, a privileged account, or associated with a current HR/security investigation?
- Does the user's business unit have a documented AI agent evaluation program?

### Step 4: Determine whether the agent had internet access

```spl
| tstats summariesonly=t count sum(Network_Traffic.bytes_out) as bytes_out
    values(Network_Traffic.dest) as remote_ips values(Network_Traffic.dest_port) as dest_ports
  from datamodel=Network_Traffic
  where Network_Traffic.src="$src$"
  earliest=-1h
| rename Network_Traffic.* as *
| eval bytes_out_mb=round(bytes_out/1024/1024,2)
| table count bytes_out_mb remote_ips dest_ports
```

- Outbound traffic during or immediately after agent execution indicates the agent successfully made network calls.
- Common agent traffic destinations: `api.openai.com`, `api.anthropic.com`, GitHub, npm/PyPI registries, Wikipedia, and arbitrary web content.
- Large `bytes_out_mb` concurrent with agent execution is a significant escalation indicator — combine with AI-007 investigation.

### Step 5: Check for file system or code execution side effects

```spl
| tstats summariesonly=t count values(Processes.process_name) as child_processes
  from datamodel=Endpoint.Processes
  where Processes.parent_process_name IN ("autogpt","crewai","langchain","autogen","metagpt",
    "babyagi","superagi","agentgpt","opendevin","openhands","swe-agent","devika","gpt-engineer","smol-developer")
  Processes.dest="$src$" Processes.user="$user$"
  earliest=-1h
| rename Processes.* as *
| table count child_processes
```

- If the agent spawned child processes (`bash`, `python`, `git`, `curl`, `npm`, `docker`), it executed code on the host.
- `git clone`, `git push`, or `gh` invocations may indicate the agent interacted with source code repositories.
- Shell interpreters spawned by the agent indicate it executed LLM-generated code without human review.

### Step 6: Check for accumulated risk from correlated detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections count by risk_object
| sort -total_risk
```

- **AI-037 + AI-007 (High Volume Upload):** Agent exfiltrated data to an AI service endpoint.
- **AI-037 + AI-035 (Lateral Movement):** Agent was used as a pivot vehicle after reconnaissance.
- **AI-037 + AI-008 (Scripted CLI):** Layered automation suggests attacker tooling, not developer experimentation.
- **AI-037 + AI-036 (Compromised Plugin):** Agent may have been installed via a compromised supply chain package.
- **AI-037 + AI-016 (Privileged Account):** Privileged user running an autonomous agent with network access is a critical escalation.
- Total risk >= 80 within 24 hours should trigger escalation to Tier 2 regardless of other context.

### Step 7: Make a determination

- **True Positive:** Unrecognized or unsanctioned user running an agent with `--continuous` or similar flags, with evidence of outbound network activity, child process execution, or file writes outside expected project directories.
- **Benign True Positive:** Developer on an approved agent evaluation team, running a sandboxed or air-gapped instance, confirmed by manager attestation and absence of outbound traffic.
- **False Positive:** CI/CD pipeline test against known-safe agent code with no outbound network access; confirm via parent process chain and network telemetry.

## True Positive Indicators

- Agent executed with `--continuous`, `--no-user-input`, or equivalent flags that disable the human approval loop.
- Agent spawned shell interpreters, package managers, or git clients as child processes.
- Significant outbound network bytes from the host concurrent with agent execution.
- Parent process is a scheduled task, cron job, or CI system with no change management record.
- User is not in engineering or a documented AI evaluation program.
- Agent execution occurred outside business hours or on a non-development endpoint (server, jump host, shared workstation).
- Risk score accumulation >= 80 within 24 hours combining AI-037 with other detections.
- Execution on a host that previously triggered AI-036 (compromised AI plugin installation).

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Developer evaluating an agent framework in a local virtual environment with no external network access | High | Allowlist the user in `ai_sanctioned_entities.csv` with `provider=*` and document the evaluation scope and end date |
| CI/CD pipeline running agent integration tests against a mock API or local stub | High | Allowlist the pipeline service account and the specific CI host; confirm no outbound traffic in the pipeline definition |
| Data science team using LangChain or AutoGen in non-serve mode (library import only, not process execution) | Medium | This detection fires on process execution; pure library imports do not generate matching events — investigate if a CLI wrapper was added |
| Security researcher running agent frameworks in an air-gapped lab VM | Low | Allowlist the lab host range; verify via network telemetry that no traffic egressed the segment |
| Legitimate internal AI platform that wraps one of the monitored binaries under the same process name | Low | Add the internal platform's service account to the allowlist and open a detection gap ticket for a custom process name exclusion |

## Response Actions

### If confirmed True Positive

1. **Contain:** If the agent is still running, isolate the endpoint from the network via EDR or NAC policy to halt autonomous outbound activity. Do not terminate the process immediately — capture a process dump and running state for forensics.
2. **Preserve Evidence:** Collect the agent's working directory (typically contains `ai_settings.yaml`, task logs, output files, and any code the agent wrote or modified), network connection logs, and child process artifacts before isolation.
3. **Assess Blast Radius:** Determine which external services the agent contacted (API keys used, repositories cloned or pushed to, files read from sensitive paths). Check VCS audit logs if the agent had git credentials.
4. **Revoke Credentials:** Rotate any API keys, tokens, or credentials that the agent may have accessed, including those stored in environment variables, `.env` files, or the user's shell profile.
5. **Notify:** Inform the user's manager. If external data exposure is confirmed or suspected, engage the data governance team and follow the incident response plan for potential data loss. Notify the CISO if the agent had access to production systems.
6. **Remediate:** Remove agent framework binaries and associated virtual environments. Audit `pip`, `npm`, and system packages on the host for related installations. Enforce endpoint policy to block unapproved interpreter-based tool installation.
7. **Document:** Record the framework used, command-line arguments, duration of execution, confirmed or suspected network destinations, files written, and child processes spawned.

### If Benign True Positive

1. Add the user and host to `ai_sanctioned_entities.csv` with an explicit `provider` value, a documented business justification, an expiration date aligned with the evaluation period, and manager approval.
2. Request that the user's team document network isolation controls (proxy allowlist, firewall rule, or air-gapped VM) for the evaluation environment.
3. Close the notable as "Benign True Positive" and tag the ticket with the evaluation program name for trend tracking.

## Tuning Guidance

### Reducing false positives from development teams

- Allowlist entire development teams by group rather than individual users when a formal agent evaluation program exists:
```
entity_type,entity,provider,scope,expires,owner,reason,enabled,allow_key
group,ai-platform-team,*,group,2026-12-31,ciso,approved_agent_eval_program,1,ai-platform-team|*
```

- Allowlist CI/CD service accounts with their specific runner hosts:
```
entity_type,entity,provider,scope,expires,owner,reason,enabled,allow_key
user,svc-jenkins,*,user,2026-06-30,platform-eng,ci_agent_integration_tests,1,svc-jenkins|*
```

### Validating coverage — find all agent framework executions in the past 30 days

```spl
| tstats summariesonly=t count values(Processes.process) as process
  from datamodel=Endpoint.Processes
  where Processes.process_name IN ("autogpt","crewai","langchain","autogen","metagpt","babyagi",
    "superagi","agentgpt","opendevin","openhands","swe-agent","devika","gpt-engineer","smol-developer")
  earliest=-30d latest=now
  by Processes.user Processes.dest Processes.process_name
| rename Processes.* as *
| stats count as execution_count values(dest) as hosts by user process_name
| sort -execution_count
```

Use this query to identify high-volume users who may warrant pre-emptive allowlisting versus users who ran an agent once and are low risk.

### Extending the monitored process list

Add new agent framework binaries to the `process_name IN (...)` list in the saved search when new frameworks emerge. Also update the `case()` provider mapping to assign a human-readable label. Common candidates to watch: `aider` (already covered by AI-008), `open-interpreter`, `agno`, `magentic-one`.

### Adjusting suppression for high-frequency evaluation environments

If a team is running repeated agent tests and the 1-hour suppression is generating stale risk events rather than fresh ones, increase the suppression period to `4h` for that team's host range using a local override in `savedsearches.conf`:

```ini
alert.suppress.period = 4h
```
