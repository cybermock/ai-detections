# AI-041 - Time-Of-Day Usage Pattern Anomaly

> **Detection ID:** AI-041
> **Status:** Active
> **Risk Score:** 25 (user)
> **MITRE ATT&CK:** T1567.002 - Exfiltration Over Web Service: Exfiltration to Cloud Storage (Exfiltration)
> **Data Model:** Web
> **Schedule:** `45 4 * * *` (daily at 4:45 AM)
> **Lookback Window:** `-30d@d` to `now`
> **Kill Chain Phase:** Exfiltration
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies users whose AI service usage at a specific hour of the day significantly deviates from their own historical pattern for that same hour. Unlike AI-015 (which flags any after-hours usage), AI-041 builds a per-user, per-hour baseline over 30 days and alerts only when usage at a particular hour is statistically anomalous for that specific individual.

Time-of-day behavioral profiling is valuable because most users have consistent daily routines. A developer who typically uses AI tools between 9 AM and 5 PM might suddenly start heavy AI usage at 2 AM, which could indicate account compromise, data exfiltration attempts during reduced monitoring windows, or a user deliberately timing sensitive activities for periods of lower oversight.

The detection uses per-user hourly histogram analysis: for each user and each hour (0-23), it calculates the historical average request count and standard deviation over 30 days. It then applies a z-score test (default threshold: 3.0) to identify hours where the current day's usage is a significant outlier. A minimum of 7 data points per hour-slot ensures the baseline is stable before alerting. The detection runs daily and evaluates the previous day's activity, making it a retrospective analysis that catches subtle timing shifts.

## Detection Logic

### Plain English

1. Query the `Web` data model for AI service traffic over the past 30 days, grouped by user, URL domain, and 1-hour time spans.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter`.
3. Extract the hour (0-23) and date from each time span.
4. Calculate hourly request totals per user, hour, and date.
5. Use `eventstats` to calculate per-user, per-hour averages, standard deviations, and data point counts over the 30-day window.
6. Load the z-score threshold from `ai_detection_config.csv` (default: 3.0).
7. Require at least 7 data points per hour-slot for baseline stability.
8. Calculate the z-score for each user-hour combination.
9. Filter for z-scores exceeding the threshold on the most recent day.
10. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
11. Tag the event with `detection_id=AI-041`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User with anomalous time-of-day usage | `tjackson@corp.com` |
| `hour` | Hour of anomalous activity (0-23) | `3` |
| `hourly_requests` | Request count during the anomalous hour | `145` |
| `user_hour_avg` | User's historical average for this hour | `2.3` |
| `user_hour_stdev` | User's historical standard deviation for this hour | `4.1` |
| `z_score` | Z-score measuring the anomaly severity | `34.8` |
| `datapoints` | Number of historical data points for baseline | `22` |

### Thresholds

| Parameter | Default Value | Config Lookup Key | Notes |
|---|---|---|---|
| Time anomaly z-score | 3.0 | `time_anomaly_zscore` | Standard deviations above user's hour-specific mean |
| Minimum data points | 7 | N/A | Hardcoded; ensures stable baseline per hour-slot |
| Lookback window | 30 days | N/A | Set via dispatch.earliest_time |

## Triage Steps

> **Estimated triage time:** 10-15 minutes

### Step 1: Validate the alert and note the anomalous hour

```spl
index=risk search_name="AI RBA - AI-041: Time-Of-Day Usage Pattern Anomaly" risk_object="$user$" earliest=-7d
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the specific hour that was anomalous and the z-score.
- Very late-night/early-morning anomalies (midnight to 5 AM) are higher priority.

### Step 2: Identify the user and their typical schedule

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Is the user in a time zone that would make the anomalous hour normal working time?
- Does the user's role involve shift work, on-call duties, or international collaboration?

### Step 3: Visualize the user's historical hourly AI usage pattern

```spl
| tstats summariesonly=t count from datamodel=Web where Web.user="$user$" earliest=-30d latest=now by Web.url_domain _time span=1h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| eval hour=tonumber(strftime(_time,"%H"))
| stats avg(count) as avg_requests max(count) as max_requests by hour
| sort hour
```

- This shows the user's typical hourly activity profile over 30 days.
- Compare the anomalous hour's activity against the historical pattern.
- Hours with zero historical average that suddenly show activity are the most suspicious.

### Step 4: Examine what the user was doing during the anomalous hour

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" earliest=-2d latest=now
| rename Web.* as *
| eval ai_domain=lower(url_domain), hour=strftime(_time,"%H")
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(count) as requests sum(bytes_out) as total_bytes values(urls) as urls values(user_agents) as agents by hour provider
| eval mb_out=round(total_bytes/1024/1024,2)
| sort hour
```

- What AI providers were used during the anomalous hour?
- Was data being uploaded (high bytes_out) or just browsing?
- What user agent was used? Automation tools running at unusual hours are suspicious.

### Step 5: Check for authentication anomalies at the same time

```spl
index=risk risk_object="$user$" earliest=-24h
| search NOT search_name="AI RBA*"
| table _time risk_object risk_score search_name risk_message
| sort -_time
```

- Are there authentication anomalies (impossible travel, anomalous IP, MFA bypass) around the same time?
- Concurrent authentication and AI usage anomalies strongly suggest account compromise.

### Step 6: Check the user's overall risk posture

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-041 + AI-015 (after-hours) = confirms out-of-pattern timing.
- AI-041 + AI-040 (peer group anomaly) = anomalous in both timing and volume.
- AI-041 + AI-007 (high volume upload) = unusual timing with significant data upload.

### Step 7: Determine if this is a one-time event or emerging pattern

```spl
index=risk search_name="AI RBA - AI-041*" risk_object="$user$" earliest=-30d
| table _time risk_object risk_score risk_message
| sort -_time
```

- A single occurrence may be a one-off event (travel, personal emergency, insomnia).
- Multiple occurrences suggest a sustained behavioral change that requires further investigation.

### Step 8: Make a determination

- **True Positive:** User's AI usage at an unusual hour indicates account compromise or deliberate evasion of monitoring.
- **Benign True Positive:** User traveled to a different time zone, worked late on a project, or changed work schedule.
- **False Positive:** Time zone misconfiguration causes incorrect hour assignment; daylight saving time shift causes temporary anomalies.

---

## True Positive Indicators

- Anomalous hour is between midnight and 5 AM local time with no historical precedent.
- High data upload volume during the anomalous hour.
- Authentication anomalies (new IP, new device, MFA bypass) at the same time.
- User's manager confirms no known schedule change or travel.
- Automation tools or API access during the anomalous hour.
- Z-score exceeds 5.0, indicating extreme deviation from personal baseline.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| User traveled to a different time zone and maintained normal AI usage | High | Cross-reference with VPN/authentication logs for travel indicators |
| User working late on a deadline or project sprint | High | Check with the user's manager; document if confirmed |
| Daylight saving time transitions creating temporary 1-hour offset anomalies | Low | Account for DST transitions in baseline calculation |
| Cron job or automation running at scheduled times that change infrequently | Low | Exclude known automation service accounts |
| Weekend usage from users who work flexible schedules | Medium | Consider separate weekday/weekend baselines |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** If account compromise is suspected, force a password reset and revoke active sessions. If insider threat, restrict AI service access.
2. **Notify:** Alert the user's manager and the security operations team. For account compromise indicators, initiate incident response.
3. **Investigate:** Correlate the anomalous timing with authentication logs, VPN connections, and endpoint activity to determine who was using the account.
4. **Remediate:** If compromised, remediate the account and investigate the scope of exposure. If insider threat, enforce monitoring and policy restrictions.
5. **Document:** Record the anomalous hour, z-score, and investigation findings.

### If Benign True Positive

1. Document the reason for the schedule change (travel, project deadline, flexible work).
2. If the schedule change is permanent, the baseline will naturally adjust over 30 days.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Increase the `time_anomaly_zscore` threshold from 3.0 to 4.0 if too many benign schedule variations trigger alerts.
- Increase the minimum data points requirement from 7 to 14 to require 2+ weeks of stable baseline.
- Consider separate weekday and weekend baselines for organizations with flexible work policies.

### Reviewing hourly usage distribution across users

```spl
| tstats summariesonly=t count from datamodel=Web where Web.url_domain=* earliest=-7d latest=now by Web.user Web.url_domain _time span=1h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| eval hour=tonumber(strftime(_time,"%H"))
| stats sum(count) as requests by user hour
| stats avg(requests) as avg_requests stdev(requests) as stdev_requests by hour
| sort hour
```

This shows the organization-wide hourly AI usage pattern, helping identify normal vs. unusual hours.

### Adjusting thresholds

Update `lookups/ai_detection_config.csv`:
```csv
control_name,value,notes
time_anomaly_zscore,4.0,Adjusted from 3.0 to accommodate flexible work schedules
```
