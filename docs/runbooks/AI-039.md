# AI-039: AI Agent Autonomous Action Detection

## Overview
| Field | Value |
|-------|-------|
| **Detection ID** | AI-039 |
| **MITRE ATT&CK** | T1059 — Command and Scripting Interpreter |
| **Kill Chain Phase** | Execution |
| **Risk Score** | User: 50, System: 30 |
| **Data Model** | Endpoint.Processes |
| **Cron** | */5 * * * * |

## Description
Detects AI agent frameworks (AutoGPT, CrewAI, LangChain, AutoGen, MetaGPT, OpenHands, SWE-agent, etc.) spawning 3+ child processes, indicating the agent is performing autonomous actions. Unlike AI-037 which detects agent framework execution itself, this detection specifically identifies agents that are actively performing work — creating files, running commands, making API calls, or modifying system state without direct human oversight.

## Alert Rationale
AI agents that autonomously execute commands pose significant security risks including unauthorized file access, data exfiltration, privilege escalation, and supply chain compromise. An agent spawning multiple child processes indicates it has moved beyond simple inference into autonomous action mode, which warrants immediate security review.

## Triage Steps

### 1. Identify the Agent and Its Actions
```spl
| tstats count values(Processes.process) as commands values(Processes.process_name) as child_procs
  from datamodel=Endpoint.Processes
  where Processes.parent_process_name="$parent_process_name$" Processes.dest="$src$"
  by Processes.user Processes.parent_process_name _time span=5m
| sort _time
```

### 2. Classify the Autonomous Actions
Look for high-risk child processes:
- **File operations**: `git`, `cp`, `mv`, `rm`, `mkdir`, `touch`, `cat`, `echo`
- **Network access**: `curl`, `wget`, `ssh`, `scp`, `nc`
- **System modification**: `chmod`, `chown`, `pip install`, `npm install`, `apt-get`
- **Credential access**: `aws`, `gcloud`, `az`, `vault`, `ssh-keygen`

### 3. Check Scope of Agent Activity
```spl
| tstats count values(Processes.process) as all_commands
  from datamodel=Endpoint.Processes
  where Processes.dest="$src$" Processes.user="$user$"
  earliest=-1h latest=now
  by Processes.process_name _time span=5m
| sort _time
```

### 4. Verify Authorization
- Was this agent execution authorized by the user?
- Is the agent operating within expected boundaries?
- Are there guardrails (sandboxing, restricted permissions) in place?

## Known False Positives
- CI/CD pipelines that use AI agents for automated code review or testing
- Development environments where engineers are actively testing agent frameworks
- Jupyter notebooks or data science workflows that shell out to agent CLIs
- Containerized agent workloads in sandboxed environments

## Tuning Recommendations
- Increase `autonomous_actions` threshold (default: 3) for development environments
- Exclude specific source systems known to run authorized AI agent workloads
- Add sanctioned entity entries for authorized agent service accounts
- Consider reducing cron frequency for environments with heavy agent usage

## Escalation Criteria
- **Critical**: Agent spawning network tools (curl, wget, ssh) or credential access commands
- **High**: Agent modifying system files or installing packages
- **Medium**: Agent performing file operations in user directories
- **Low**: Agent running read-only commands (ls, cat, grep)

## Response Actions
1. Immediately assess what actions the agent has taken (file changes, network connections)
2. If unauthorized actions detected, terminate the agent process
3. Review and revert any file system changes made by the agent
4. Check for any data exfiltration via network connections
5. Audit the agent's configuration and permissions
6. Document the incident and update AI acceptable use policy
