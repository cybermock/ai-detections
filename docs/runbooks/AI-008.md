# AI-008 - Scripted AI CLI Invocation

> **Detection ID:** AI-008
> **Status:** Active
> **Risk Score:** 32 (user) + 18 (system) = 50 total
> **MITRE ATT&CK:** T1059.004 - Command and Scripting Interpreter: Unix Shell (Execution)
> **Data Model:** Endpoint.Processes
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Exploitation / Actions on Objectives
> **Last Updated:** 2026-02-13

---

## Overview

This detection identifies when AI CLI tools are launched by scripting interpreters or automation frameworks rather than by direct user interaction. Specifically, it fires when a known AI CLI process (codex, claude, gemini, aider, sgpt, llm, etc.) has a parent process that is a scripting interpreter (python, bash, powershell, node, etc.) or build tool (make).

This is a significant indicator because it means AI tools are being used programmatically, which has several security implications:

1. **Automated data exfiltration:** An attacker or malicious script could systematically feed sensitive data to an AI service via CLI, bypassing interactive controls.
2. **CI/CD pipeline abuse:** AI CLIs embedded in build pipelines may send source code, configuration files, or secrets to external AI providers without human review.
3. **Lateral movement tooling:** Adversaries may use AI CLIs to generate malicious payloads, decode obfuscated data, or automate reconnaissance after initial compromise.
4. **Shadow automation:** Users may create unsanctioned automation workflows that continuously send data to AI services without governance oversight.

The parent-child process relationship (interpreter -> AI CLI) is the key distinguishing factor from AI-004 (direct CLI execution). AI-008 focuses on the automation signal, which carries higher risk (32 + 18 = 50 points vs. AI-004's 25 + 15 = 40 points).

## Detection Logic

### Plain English

1. Query the `Endpoint.Processes` data model for process creation events where:
   - The process name matches a known AI CLI tool (codex, claude, gemini, gh-copilot, aider, sgpt, llm, tgpt, mods, fabric, aichat, amazon-q, openai, cline).
   - The parent process is a scripting interpreter or build tool (python, python3, bash, zsh, sh, pwsh, powershell, cmd, node, npm, npx, make, osascript, wscript, cscript, ruby, perl).
2. Enrich with the `ai_process_filter` macro to resolve the AI tool name and provider.
3. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
4. Tag the event with `detection_id=AI-008`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User account running the script | `jsmith` |
| `src` (dest) | Endpoint where the script ran | `DEV-WORKSTATION-42` |
| `process_name` | AI CLI tool that was invoked | `claude` |
| `parent_process_name` | Scripting interpreter or build tool | `python3` |
| `process` | Full command line of the AI CLI | `claude --prompt "analyze this code"` |
| `parent_process` | Full command line of the parent process | `python3 /home/jsmith/scripts/ai_helper.py` |
| `provider` | Resolved AI service provider | `Anthropic` |

### Thresholds

This detection has no configurable thresholds. Any instance of an interpreter spawning an AI CLI triggers the alert. The presence of the parent-child relationship is itself the indicator.

## Triage Steps

> **Estimated triage time:** 15-25 minutes

### Step 1: Validate the alert

```spl
index=risk search_name="AI RBA - Scripted AI CLI Invocation" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score search_name
| sort -_time
```

- Note the frequency: a single occurrence may be a one-off test, while repeated occurrences suggest an automated workflow.

### Step 2: Identify the user and their role

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit
```

- Is this user a developer? Developers may legitimately use AI CLIs in scripts.
- Is this user in a non-technical role? Scripted AI CLI usage from non-technical users is highly suspicious.

### Step 3: Examine the parent-child process chain

```spl
| tstats summariesonly=t count values(Processes.process) as cli_command values(Processes.parent_process) as parent_command values(Processes.process_path) as cli_path
  from datamodel=Endpoint.Processes
  where Processes.user="$user$" Processes.dest="$src$" Processes.process_name="$process_name$" Processes.parent_process_name="$parent_process_name$" earliest=-24h
| table count cli_command parent_command cli_path
```

- **What script is calling the AI CLI?** The `parent_command` reveals the calling script path and arguments.
- **What arguments are being passed to the AI CLI?** The `cli_command` may show prompts, file paths, or data being piped.
- **Where is the script located?** Unusual paths (temp directories, user downloads, hidden folders) are red flags.

### Step 4: Investigate the calling script

```spl
| tstats summariesonly=t values(Processes.process) as process values(Processes.process_name) as child_processes
  from datamodel=Endpoint.Processes
  where Processes.parent_process_name IN ("python","python3","bash","zsh","node") Processes.dest="$src$" Processes.user="$user$" earliest=-24h
| mvexpand child_processes
| stats count by child_processes
```

- Does the script call other suspicious tools besides the AI CLI?
- Is the script reading from sensitive files, databases, or environment variables?
- Is the script sending AI responses to external services or writing to network paths?

### Step 5: Check if this is a CI/CD pipeline

```spl
| tstats summariesonly=t values(Processes.process) as process values(Processes.parent_process_name) as parent
  from datamodel=Endpoint.Processes
  where Processes.dest="$src$" Processes.parent_process_name IN ("jenkins","gitlab-runner","github-actions","buildkite-agent","drone","circleci") earliest=-24h
| table process parent
```

- If the parent process chain includes a CI/CD agent, the AI CLI is embedded in a build pipeline.
- CI/CD usage may be sanctioned but should still be reviewed for data handling.

### Step 6: Check for related detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-004 (CLI execution) often fires alongside AI-008 for the same event.
- AI-007 (high upload) or AI-010 (burst) suggest the script is sending large/frequent data.
- AI-020 (DLP) would confirm sensitive data involvement.

### Step 7: Make a determination

- **True Positive:** Unsanctioned script automating AI CLI usage without approval, especially if sensitive data is being processed.
- **Benign True Positive:** Developer has an approved workflow using AI CLIs in scripts. Add to allowlist.
- **False Positive:** The parent process happens to be an interpreter but the AI CLI was launched interactively through a terminal emulator. Check if the parent is actually a terminal app (e.g., bash as login shell).

---

## True Positive Indicators

- Parent script reads from sensitive files or environment variables containing secrets.
- AI CLI is called in a loop or with `--batch` / `--no-interactive` flags.
- Script is located in a temporary directory, download folder, or hidden path.
- User is not a developer or has no documented need for AI CLI automation.
- The AI CLI is passing `--api-key` on the command line (credential exposure).
- Multiple AI CLIs are called from the same script (tool chaining).
- The script also exfiltrates AI responses to external services.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Developer running AI CLI from their shell (bash -> claude is normal terminal usage) | High | Check if parent is a login shell vs. a script; consider allowlisting developers |
| CI/CD pipeline that uses AI for code review or documentation generation | Medium | Allowlist CI/CD service accounts or specific build agents |
| Package manager or IDE extension spawning interpreter that calls AI CLI | Low | Investigate the full process chain to identify the actual initiator |
| Cron job or scheduled task running an approved AI automation | Medium | Add the service account and endpoint to the allowlist |

## Response Actions

### If confirmed True Positive

1. **Contain:**
   - Kill the running script/process if it is actively sending data.
   - Block the endpoint from accessing AI service APIs at the proxy/firewall.
   - Disable or quarantine the script file.

2. **Notify:**
   - Inform the user's manager.
   - If this is a CI/CD pipeline, notify the DevOps/platform engineering team.
   - If the script handles sensitive data, notify the data governance team.

3. **Investigate:**
   - Obtain and review the full source code of the calling script.
   - Determine what data the script reads and sends to the AI CLI.
   - Check for exfiltration of AI responses to external destinations.
   - Review the user's command history and recent file access.

4. **Remediate:**
   - Remove or disable unauthorized scripts.
   - If legitimate need exists, channel the automation through an approved AI gateway with content filtering.
   - Implement controls to prevent AI CLI tools from being installed on non-developer endpoints.

5. **Document:** Record the script path, process chain, data flow, and remediation steps.

### If Benign True Positive

1. Add the user and/or system to `ai_sanctioned_entities.csv` with the specific provider.
2. Document the automation workflow and its data handling practices.
3. Set an expiration date for periodic review.
4. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- The most common false positive is developers using AI CLIs from their login shell (bash/zsh). To filter these, consider adding a condition that excludes parent processes that are login shells (check for `-bash` or process with `--login`).
- Allowlist approved CI/CD service accounts and build agents.
- If a specific AI CLI tool is widely sanctioned (e.g., gh-copilot for all developers), allowlist the tool at the group level.

### Analyzing scripted AI CLI usage patterns

```spl
| tstats summariesonly=t count values(Processes.process) as commands values(Processes.parent_process) as parent_commands
  from datamodel=Endpoint.Processes
  where Processes.process_name IN ("codex","claude","gemini","aider","sgpt","llm","openai")
    AND Processes.parent_process_name IN ("python","python3","bash","zsh","sh","pwsh","powershell","cmd","node","make")
  earliest=-7d latest=now by Processes.user Processes.dest Processes.process_name Processes.parent_process_name
| rename Processes.* as *
| stats sum(count) as total_invocations dc(dest) as unique_endpoints values(process_name) as ai_tools values(parent_process_name) as interpreters by user
| sort -total_invocations
```

### Adding allowlist entries

To allowlist a CI/CD service account:
```
entity_type,entity,provider,scope,expires,owner,reason,enabled,allow_key
system,BUILD-AGENT-01,*,global,2026-12-31,devops_lead,approved_cicd_ai_integration,1,build-agent-01|*
```
