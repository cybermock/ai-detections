# AI-024 - Browser Extension High-Volume Data Transmission

> **Detection ID:** AI-024
> **Status:** Active
> **Risk Score:** 30 (user) + 15 (system)
> **MITRE ATT&CK:** T1176 - Browser Extensions (Persistence)
> **Data Model:** Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Exfiltration
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies AI browser extensions transmitting unusually high volumes of data to their backend services. While AI-021 detects the presence or installation of AI browser extensions, AI-024 focuses specifically on the data volume being transmitted, catching extensions that are actively exfiltrating page content, clipboard data, or other browsing context to AI services in bulk.

Browser extensions have broad access to page content, form data, clipboard contents, and browsing history. When an AI extension (such as ChatGPT, Copilot, or Claude browser extensions) transmits large volumes of data, it may indicate that the extension is scraping entire pages of sensitive internal applications -- such as wikis, code repositories, CRM systems, or financial dashboards -- and sending that content to external AI services for processing.

The detection monitors traffic to known AI extension backend domains (e.g., `extensions.chat.openai.com`, `browser.claude.ai`) as well as traffic from browser extension user agents (`chrome-extension`, `moz-extension`). When the outbound data volume exceeds the configured threshold (default 10 MB), it fires with risk scores that reflect the bulk data exposure. This complements AI-021 by adding a volume dimension that distinguishes passive extension presence from active data exfiltration.

## Detection Logic

### Plain English

1. Query the `Web` data model for all web traffic, aggregating by user, source IP, URL domain, and HTTP user agent.
2. Filter for traffic destined to known AI extension backend domains (`extensions.chat.openai.com`, `browser.claude.ai`, `copilot.microsoft.com`) or traffic where the user agent contains browser extension identifiers (`chrome-extension`, `moz-extension`).
3. Enrich matching domains against `ai_provider_domains.csv` via `ai_domains_filter`.
4. Calculate outbound data volume in megabytes.
5. Load the extension bytes threshold from `ai_detection_config.csv` (default: 10 MB).
6. Filter for users exceeding the threshold.
7. Aggregate request counts, total MB, and AI domains per user, source, and provider.
8. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
9. Tag the event with `detection_id=AI-024`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User whose browser extension is transmitting data | `mgarcia@corp.com` |
| `src` | Source endpoint | `10.1.5.22` |
| `provider` | AI extension provider | `OpenAI` |
| `total_mb` | Total megabytes transmitted by extension | `18.7` |
| `request_count` | Number of extension requests | `312` |
| `ai_domains` | Extension backend domains | `extensions.chat.openai.com` |

### Thresholds

| Parameter | Default Value | Config Lookup Key | Notes |
|---|---|---|---|
| Extension bytes threshold | 10 MB | `extension_bytes_threshold_mb` | Minimum outbound volume to trigger detection |

## Triage Steps

> **Estimated triage time:** 10-15 minutes

### Step 1: Validate the alert and note the data volume

```spl
index=risk search_name="AI RBA - AI-024: Browser Extension High-Volume Data Transmission" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the total MB transmitted and the AI extension provider.
- Check if this user has triggered this detection repeatedly.

### Step 2: Identify the user and their role

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Users in roles with access to sensitive internal systems (finance, HR, legal, engineering) pose higher risk.
- Determine if the user has approved AI extension usage.

### Step 3: Examine the extension traffic in detail

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" Web.src="$src$" earliest=-4h
  (Web.url_domain IN ("extensions.chat.openai.com","browser.claude.ai","copilot.microsoft.com") OR Web.http_user_agent="*chrome-extension*" OR Web.http_user_agent="*moz-extension*")
| rename Web.* as *
| stats sum(bytes_out) as total_bytes sum(count) as requests values(urls) as urls values(user_agents) as agents by url_domain
| eval mb_out=round(total_bytes/1024/1024,2)
| sort -mb_out
```

- Which extension backend is receiving the data?
- Are the URLs suggesting content submission (POST endpoints) or configuration sync?
- What does the user agent string reveal about the specific extension?

### Step 4: Identify what internal sites the user was browsing

```spl
| tstats summariesonly=t count values(Web.url) as urls from datamodel=Web
  where Web.user="$user$" Web.src="$src$" earliest=-4h
  NOT (Web.url_domain IN ("extensions.chat.openai.com","browser.claude.ai","copilot.microsoft.com"))
| rename Web.* as *
| stats count by url_domain
| sort -count
| head 20
```

- Internal wikis, code repositories, CRM systems, or financial dashboards accessed alongside the extension activity indicate what data may have been captured.
- Sensitive internal applications accessed during the same session significantly increase the risk assessment.

### Step 5: Compare to baseline extension usage

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.user="$user$" earliest=-30d latest=now
  (Web.url_domain IN ("extensions.chat.openai.com","browser.claude.ai","copilot.microsoft.com") OR Web.http_user_agent="*chrome-extension*")
  by _time span=1d
| eval mb_out=round(bytes_out/1024/1024,2)
| timechart span=1d sum(mb_out) as daily_extension_mb
```

- Is today's volume abnormal compared to the user's typical extension data transmission?
- A sudden spike suggests the extension captured unusual amounts of content.

### Step 6: Check for related detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-024 + AI-021 (extension activity) = extension installed and actively exfiltrating.
- AI-024 + AI-007 (high volume upload) = multi-channel high-volume AI data transfer.
- AI-024 + AI-020 (DLP) = confirmed sensitive data reaching AI via extension.

### Step 7: Make a determination

- **True Positive:** Browser extension is bulk-transmitting internal application content to an AI service without authorization.
- **Benign True Positive:** Approved AI extension used heavily by a power user for legitimate summarization or analysis.
- **False Positive:** Extension syncing configuration or preferences produces inflated byte counts.

---

## True Positive Indicators

- Extension data volume exceeds 20+ MB in a 15-minute window.
- User was browsing sensitive internal applications (code repos, wikis, financial systems) during the same session.
- Extension backend domain receives POST requests with large payloads.
- User does not have an approved AI extension usage exception.
- Extension is not on the organization's approved extension list.
- Combined with DLP violations (AI-020) for the same timeframe.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Approved AI coding assistant extension (Copilot) with large codebase context | High | Allowlist approved extensions for development teams; adjust threshold upward for dev groups |
| Extension syncing user preferences, history, or configuration data | Medium | Identify sync-specific URL patterns and exclude from byte counting |
| Proxy logs double-counting extension traffic due to preflight/CORS requests | Low | Validate byte counting accuracy with known test uploads |
| User researching public information and using AI extension for summarization | Medium | Assess data sensitivity based on browsed sites, not just volume |
| Browser update or extension auto-update inflating traffic counts | Low | Filter out known update/manifest URLs from byte aggregation |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** Disable or remove the AI browser extension via endpoint management (GPO, MDM, or browser policy). Block extension backend domains at the proxy.
2. **Notify:** Inform the user's manager and data governance team about the scope of data potentially exfiltrated.
3. **Investigate:** Review the internal applications the user browsed during the extension's active period to assess data sensitivity.
4. **Remediate:** Deploy a browser extension allowlist policy. Implement content-aware proxy rules that inspect extension traffic payloads.
5. **Document:** Record the extension name, data volume, internal applications accessed, and data exposure assessment.

### If Benign True Positive

1. Add the user/extension combination to `ai_sanctioned_entities.csv` with appropriate scope and expiration.
2. If the extension is approved organizationally, adjust the threshold for users in the approved group.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Increase the `extension_bytes_threshold_mb` from 10 MB to 25 MB if your environment has many approved AI extension users.
- Create extension-specific allowlists for approved development tools like GitHub Copilot.
- Exclude known extension update/sync URLs from byte counting in the proxy configuration.

### Measuring extension data volumes to calibrate thresholds

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out from datamodel=Web
  where (Web.url_domain IN ("extensions.chat.openai.com","browser.claude.ai","copilot.microsoft.com") OR Web.http_user_agent="*chrome-extension*" OR Web.http_user_agent="*moz-extension*")
  earliest=-7d latest=now by Web.user _time span=15m
| rename Web.* as *
| eval mb_out=round(bytes_out/1024/1024,2)
| stats perc50(mb_out) as p50_mb perc90(mb_out) as p90_mb perc95(mb_out) as p95_mb perc99(mb_out) as p99_mb max(mb_out) as max_mb by user
| stats avg(p90_mb) as avg_p90 avg(p95_mb) as avg_p95 avg(p99_mb) as avg_p99 max(max_mb) as env_max
```

Set the threshold above p95 to capture only truly anomalous extension data transmission.

### Adjusting thresholds

Update `lookups/ai_detection_config.csv`:
```csv
control_name,value,notes
extension_bytes_threshold_mb,25,Adjusted from 10 MB based on approved extension usage patterns
```
