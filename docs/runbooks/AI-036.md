# AI-036 - Compromised AI Plugin/Extension Installation

> **Detection ID:** AI-036
> **Status:** Active
> **Risk Score:** 35 (user) + 20 (system)
> **MITRE ATT&CK:** T1195.002 - Supply Chain Compromise: Compromise Software Supply Chain (Initial Access)
> **Data Model:** Endpoint.Processes + Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Persistence
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies the installation of AI-related plugins, packages, or browser extensions through package managers (npm, pip) or browser extension stores. The AI ecosystem's rapid growth has created a fertile ground for supply chain attacks: malicious or compromised packages masquerade as legitimate AI tools to gain access to development environments, steal credentials, or exfiltrate code. Typosquatting attacks on AI package names (e.g., `chatgpt-helper` vs. `chatgpt-heIper`) are increasingly common.

Supply chain compromise through AI plugins represents a dual threat. First, the plugin itself may contain malware that exfiltrates data, installs backdoors, or modifies source code. Second, AI browser extensions gain privileged access to all web page content the user views, effectively becoming a surveillance tool that can capture sensitive internal application data and transmit it to attacker-controlled infrastructure.

The detection monitors two installation vectors. On the endpoint side, it watches for package manager processes (npm, pip) and IDE extension managers (VS Code, Cursor) executing install commands that reference AI-related keywords (ai, gpt, llm, copilot, claude). On the web side, it monitors POST/PUT requests to browser extension stores (Chrome Web Store, Firefox Add-ons, VS Code Marketplace) with AI-related URL parameters. Any installation event triggers the detection, reflecting the high risk of even a single compromised plugin gaining persistent access.

## Detection Logic

### Plain English

1. Query the `Endpoint.Processes` data model for package manager processes (`npm`, `pip`, `pip3`, `code`, `cursor`) executing install commands that contain AI-related keywords (`ai`, `gpt`, `llm`, `copilot`, `claude`).
2. Query the `Web` data model for POST/PUT requests to browser extension stores (`chrome.google.com/webstore`, `addons.mozilla.org`, `marketplace.visualstudio.com`) with AI-related keywords in the URL.
3. Union the two datasets by user and source.
4. Aggregate install events, methods, commands, and URLs per user and source.
5. Filter for at least 1 installation event.
6. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
7. Tag the event with `detection_id=AI-036`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User installing AI plugins | `pchen@corp.com` |
| `src` | Source endpoint | `10.1.6.15` |
| `install_events` | Number of installation events | `3` |
| `install_methods` | Package managers or stores used | `npm, code` |
| `install_commands` | Full install commands captured | `npm install chatgpt-api` |
| `install_urls` | Extension store URLs accessed | `chrome.google.com/webstore/detail/chatgpt...` |

### Thresholds

| Parameter | Default Value | Notes |
|---|---|---|
| Minimum install events | 1 | Any AI plugin installation triggers detection |
| AI keyword match | `ai\|gpt\|llm\|copilot\|claude` | Matched against process arguments and URL parameters |

## Triage Steps

> **Estimated triage time:** 15-20 minutes

### Step 1: Validate the alert and identify the installation method

```spl
index=risk search_name="AI RBA - AI-036: Compromised AI Plugin/Extension Installation" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the installation method (npm, pip, browser extension) and any commands/URLs.
- Multiple installations in a short period increase suspicion.

### Step 2: Identify the user

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Developers are more likely to install packages legitimately.
- Non-technical users installing AI packages or extensions are more suspicious.

### Step 3: Examine the specific packages or extensions installed

```spl
| tstats summariesonly=t count values(Processes.process) as commands values(Processes.parent_process_name) as parent_processes
  from datamodel=Endpoint.Processes where Processes.user="$user$" Processes.dest="$src$"
  Processes.process_name IN ("npm","pip","pip3","code","cursor")
  (Processes.process="*install*")
  earliest=-4h
| rename Processes.* as *
| table _time process_name commands parent_processes count
```

- What specific packages were installed? Look for typosquatting (character substitution in popular package names).
- Was the install initiated interactively or from a script (check parent process)?
- Are the packages from known/trusted publishers?

### Step 4: Check the package against known threat intelligence

```spl
| tstats summariesonly=t values(Processes.process) as commands from datamodel=Endpoint.Processes
  where Processes.user="$user$" Processes.dest="$src$"
  Processes.process_name IN ("npm","pip","pip3")
  Processes.process="*install*" earliest=-4h
| rename Processes.* as *
| rex field=commands "install\\s+(?<package_name>[a-zA-Z0-9_-]+)"
| table package_name commands
```

- Research the extracted package name on pypi.org or npmjs.com.
- Check the package's download count, publication date, and maintainer.
- Recently published packages with low download counts are higher risk.
- Check for security advisories against the package name.

### Step 5: Examine browser extension installation activity

```spl
| tstats summariesonly=t count values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" Web.src="$src$"
  Web.url IN ("*chrome.google.com/webstore*","*addons.mozilla.org*","*marketplace.visualstudio.com*")
  Web.http_method IN ("POST","PUT") earliest=-4h
| rename Web.* as *
| table _time urls user_agents count
```

- What extension store was accessed?
- What extension was installed (extract from URL)?
- Research the extension's ratings, reviews, and publisher credibility.

### Step 6: Check for post-installation suspicious behavior

```spl
| tstats summariesonly=t count values(Processes.process) as processes values(Processes.parent_process_name) as parents
  from datamodel=Endpoint.Processes where Processes.dest="$src$" earliest=-4h
  (Processes.parent_process_name="node" OR Processes.parent_process_name="python*")
| rename Processes.* as *
| where match(lower(processes),"curl|wget|nc|ncat|powershell|base64|eval|exec")
| table _time processes parents count
```

- After installation, did any suspicious child processes spawn?
- Network callbacks, encoded commands, or shell spawns from the installed package are strong indicators of compromise.

### Step 7: Correlate with other detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-036 + AI-024 (extension high volume) = installed extension now exfiltrating data.
- AI-036 + AI-021 (browser extension activity) = extension installed and active.
- AI-036 + AI-033 (API key exposure) = installed package may have exposed API keys.

### Step 8: Make a determination

- **True Positive:** User installed a compromised or malicious AI package/extension that poses a security risk.
- **Benign True Positive:** Developer installing a legitimate, well-known AI package for approved development work.
- **False Positive:** Package name contains AI keyword but is not AI-related (e.g., "repair-ai-file" utility).

---

## True Positive Indicators

- Package name is a typosquat of a popular AI package.
- Package was recently published with few downloads or reviews.
- Post-installation suspicious behavior (network callbacks, encoded commands, shell spawns).
- User is not a developer or does not have approval to install software.
- Multiple AI packages installed in rapid succession from different sources.
- Extension has been flagged or removed from the store (but cached version installed).

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Developer installing well-known AI packages (openai, langchain, transformers) | High | Maintain an allowlist of approved AI packages; match against package name |
| Automated CI/CD pipeline installing AI packages as part of build process | Medium | Exclude CI/CD service accounts from this detection |
| User browsing extension store without installing (GET requests counted as POST) | Low | Ensure detection filters strictly on POST/PUT methods |
| Package name coincidentally contains AI keyword (e.g., "repair" tool) | Low | Review and tune keyword matching for precision |
| IT team deploying approved extensions via GPO or MDM | Medium | Exclude managed deployment service accounts |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** Immediately remove the compromised package/extension from the endpoint. If the package had network access, isolate the endpoint for forensic analysis.
2. **Notify:** Alert the security engineering team and the user's manager. If the package is a known supply chain threat, escalate to incident response.
3. **Investigate:** Analyze the package source code for malicious indicators. Check if credentials, API keys, or source code were exfiltrated. Review all systems where the package may have been installed.
4. **Remediate:** Rotate any credentials that may have been exposed. Deploy package allowlisting policies. Block known-compromised packages at the package manager level.
5. **Document:** Record the package name, source, installation method, and any confirmed malicious behavior.

### If Benign True Positive

1. If the package is approved, add it to the approved AI software list.
2. If the user needs the package for legitimate work, ensure it is from a trusted source.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Maintain an allowlist of approved AI packages (e.g., `openai`, `anthropic`, `langchain`, `transformers`) and exclude them from this detection.
- Exclude CI/CD service accounts and automated deployment pipelines.
- Refine the AI keyword list to reduce false matches from non-AI packages.

### Building an approved AI package allowlist

```spl
| tstats summariesonly=t count values(Processes.process) as commands from datamodel=Endpoint.Processes
  where Processes.process_name IN ("npm","pip","pip3") Processes.process="*install*"
  earliest=-30d latest=now by Processes.user Processes.dest
| rename Processes.* as *
| mvexpand commands
| rex field=commands "install\\s+(?<package_name>[a-zA-Z0-9_-]+)"
| where match(lower(package_name),"ai|gpt|llm|copilot|claude|openai|anthropic|langchain|transformers")
| stats count dc(user) as unique_users values(user) as users by package_name
| sort -count
```

Review the top AI-related packages being installed across your environment. Validate and add legitimate ones to an allowlist.
