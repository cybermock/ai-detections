# AI-015 - After-Hours AI Service Usage

> **Detection ID:** AI-015
> **Status:** Active
> **Risk Score:** 15 (user) + 8 (system) = 23 total
> **MITRE ATT&CK:** T1567.002 - Exfiltration Over Web Service: Exfiltration to Cloud Storage (Exfiltration)
> **Data Model:** Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Actions on Objectives
> **Last Updated:** 2026-02-13

---

## Overview

This detection identifies AI service usage outside business hours. The default after-hours window is 8 PM to 6 AM local time and all day on weekends (Saturday/Sunday). After-hours AI usage may indicate reduced oversight conditions where a user can more freely exfiltrate data, or it may simply reflect a user working late with AI tools. The risk score is low (23 points), serving as a risk multiplier when combined with other detections.

## Detection Logic

### Plain English

1. Query the `Web` data model for AI domain traffic, bucketed by 1-hour spans.
2. Enrich via `ai_domains_filter`.
3. Extract the hour of day and day of week from the event timestamp.
4. Load after-hours start/end from `ai_detection_config.csv` (defaults: start=20, end=6).
5. Filter to events where the hour is >= start OR < end, or the day of week is Saturday/Sunday (dow > 5).
6. Aggregate by user/source/provider.
7. Exclude sanctioned entities.

### Thresholds

| Parameter | Default Value | Config Lookup Key |
|---|---|---|
| After-hours start | 20 (8 PM) | `after_hours_start` |
| After-hours end | 6 (6 AM) | `after_hours_end` |
| Weekend | Saturday & Sunday | N/A (hardcoded `dow > 5`) |

## Triage Steps

> **Estimated triage time:** 5 minutes

1. **Is the user known to work late or in a different time zone?**
   ```spl
   | inputlookup identity_lookup_expanded | search identity="$user$"
   | table identity bunit category
   ```
2. **Check what AI services they accessed and how much data was uploaded.**
3. **Check for AI-007 (high upload) and AI-020 (DLP) co-occurrence.** After-hours + data upload is a higher-priority combination.
4. **Review the user's typical working hours over the past 30 days to determine if this is normal behavior.**

---

## True Positive Indicators

- User does not typically work late (no historical after-hours web activity).
- After-hours access combined with high upload volume (AI-007).
- DLP violation also fired (AI-020).
- Activity occurs at unusual hours (e.g., 2-4 AM).

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| User working late on a deadline | High | Accept as risk signal; low risk on its own |
| User in a different time zone than the configured after-hours window | High | Adjust after-hours window or create per-timezone configurations |
| Automated system/cron job running during maintenance window | Medium | Allowlist the system |

---

## Response Actions

### If confirmed True Positive

1. This detection alone rarely warrants immediate action.
2. If combined with high-severity detections (AI-007, AI-020), follow those runbooks with elevated urgency.
3. Document the after-hours pattern for the user's risk profile.

### If Benign True Positive

1. For shift workers or remote employees in other time zones, consider adjusting the after-hours window or adding allowlist entries.

## Tuning Guidance

### Calibrating after-hours windows

```spl
| tstats summariesonly=t count from datamodel=Web where Web.url_domain=* earliest=-30d by Web.user _time span=1h
| rename Web.* as *
| eval hour=tonumber(strftime(_time,"%H"))
| stats sum(count) as total_requests by user hour
| xyseries user hour total_requests
```

Use this to visualize each user's activity by hour. Set the after-hours window to exclude typical working hours for your population.
