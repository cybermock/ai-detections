# AI-026 - AI Usage From Personal Account / OAuth Mismatch

> **Detection ID:** AI-026
> **Status:** Active
> **Risk Score:** 28 (user) + 12 (system)
> **MITRE ATT&CK:** T1078.004 - Valid Accounts: Cloud Accounts (Defense Evasion, Persistence)
> **Data Model:** Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Collection
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies users who access AI services using personal accounts (Gmail, Yahoo, Hotmail, iCloud, Proton) rather than their corporate identity. When an employee logs into ChatGPT or Claude with a personal email address from a corporate endpoint, it creates a significant governance blind spot: data uploaded to the AI service is associated with the personal account, making it invisible to corporate data retention, audit, and compliance controls.

Personal account usage on AI services is particularly dangerous because the conversation history, uploaded files, and AI-generated outputs belong to the personal account holder -- not the organization. If the employee leaves the company, all data submitted through the personal AI account goes with them. Additionally, personal accounts typically lack the enterprise security features (SSO, audit logging, admin controls) that corporate AI subscriptions provide.

The detection works by inspecting cookie data in proxy logs for indicators of personal email domain authentication (gmail.com, yahoo.com, hotmail.com, etc.) and comparing against the user's corporate email from `identity_lookup_expanded`. When the authenticated session contains personal email indicators that do not match the corporate email, the mismatch is flagged. This approach requires proxy infrastructure that logs cookie data, which not all environments support.

## Detection Logic

### Plain English

1. Query the `Web` data model for all web traffic, including cookie data, aggregated by user, source IP, URL domain, and cookie.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter` to identify AI provider traffic.
3. Look up the user's corporate email from `identity_lookup_expanded`.
4. Inspect cookie data for personal email domain patterns (gmail.com, yahoo.com, hotmail.com, outlook.com, icloud.com, proton).
5. Filter for cases where the cookie contains a personal email domain AND does not contain the corporate email.
6. Aggregate request counts, AI domains, and timing per user, source, provider, and corporate email.
7. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
8. Tag the event with `detection_id=AI-026`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | Corporate user identity | `jmiller@corp.com` |
| `src` | Source endpoint | `10.1.4.30` |
| `provider` | AI service provider | `OpenAI` |
| `corp_email` | User's corporate email | `jmiller@corp.com` |
| `request_count` | Number of requests with personal account | `45` |
| `ai_domains` | AI domains accessed | `chatgpt.com` |

### Thresholds

| Parameter | Default Value | Notes |
|---|---|---|
| Personal email domains | `gmail.com\|yahoo.com\|hotmail.com\|outlook.com\|icloud.com\|proton` | Matched in cookie data |
| Corporate email mismatch | Required | Cookie must NOT contain the user's corporate email |

## Triage Steps

> **Estimated triage time:** 10-15 minutes

### Step 1: Validate the alert and note the personal account indicators

```spl
index=risk search_name="AI RBA - AI-026: AI Usage From Personal Account / OAuth Mismatch" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the corporate email and the AI provider involved.
- Check if the user has triggered this detection for multiple AI providers.

### Step 2: Identify the user and confirm corporate email

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Confirm the corporate email address matches the user's identity.
- Check if the user has a known secondary or alias email that could explain the mismatch.

### Step 3: Examine the AI traffic with cookie context

```spl
| tstats summariesonly=t count values(Web.url) as urls values(Web.cookie) as cookies
  from datamodel=Web where Web.user="$user$" Web.src="$src$" earliest=-4h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(count) as requests values(urls) as urls values(cookies) as cookies by provider
```

- Review the cookie data for personal email indicators.
- What specific personal email domain appears in the cookies?
- Which AI providers are being accessed with personal accounts?

### Step 4: Check if the user also uses a corporate account for AI

```spl
| tstats summariesonly=t count values(Web.cookie) as cookies from datamodel=Web
  where Web.user="$user$" earliest=-7d latest=now
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(count) as requests values(cookies) as cookies by provider
```

- If the user sometimes uses a corporate SSO account and sometimes a personal account, investigate why they switch.
- Exclusively personal account usage is more concerning than occasional personal account usage.

### Step 5: Assess data upload volume through personal account

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out count from datamodel=Web
  where Web.user="$user$" Web.src="$src$" earliest=-24h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as total_bytes sum(count) as requests by provider
| eval mb_out=round(total_bytes/1024/1024,2)
```

- Large data volumes through a personal account compound the governance risk.
- Even small volumes are concerning if the data is sensitive.

### Step 6: Correlate with other detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-026 + AI-023 (unmanaged device) = personal account on personal device = complete bypass.
- AI-026 + AI-007 (high volume) = personal account with large data uploads.
- AI-026 + AI-022 (VPN evasion) = personal account via anonymized channel.

### Step 7: Check organizational AI account policy

```spl
| inputlookup ai_sanctioned_entities
| search entity="$user$"
| table entity provider scope expires owner reason enabled
```

- Does the organization provide corporate AI accounts that the user should be using instead?
- Is there an enterprise SSO integration for the AI provider that the user is bypassing?

### Step 8: Make a determination

- **True Positive:** User deliberately uses a personal account to access AI services, bypassing corporate governance controls.
- **Benign True Positive:** User has both personal and corporate accounts; personal account usage is incidental (e.g., auto-login from saved browser session).
- **False Positive:** Cookie parsing incorrectly identifies personal email indicators; proxy data quality issue.

---

## True Positive Indicators

- Personal email domain clearly visible in cookie/session data.
- Organization provides corporate AI accounts that the user should be using.
- User uses personal account exclusively, never the corporate account.
- Data uploads through the personal account contain work-related content.
- Combined with unmanaged device (AI-023) or VPN evasion (AI-022).
- User recently received notification about corporate AI account availability.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| User's browser has saved personal account credentials that auto-login | Medium | Educate users about browser profile separation; enforce corporate browser profiles |
| Cookie data contains personal email references from non-authentication contexts | Medium | Refine cookie parsing to focus on authentication cookies specifically |
| Organization does not provide corporate AI accounts; personal accounts are implicitly allowed | High | Establish clear AI account policy before enforcing this detection |
| SSO integration sends corporate email in a non-standard cookie format | Low | Validate cookie format against your SSO provider's implementation |
| Shared workstation where previous user's cookies persist | Low | Enforce browser session cleanup on shared workstations |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** Contact the user and require them to switch to their corporate AI account. If no corporate account exists, restrict AI access until one is provisioned.
2. **Notify:** Inform the user's manager about the personal account usage and the governance implications.
3. **Investigate:** Assess what data was uploaded through the personal account. Coordinate with the AI provider if enterprise admin access allows reviewing account activity.
4. **Remediate:** Deploy SSO enforcement for AI services at the proxy level. Implement browser policies that block personal account logins on corporate endpoints.
5. **Document:** Record the personal account indicators, data exposure assessment, and remediation steps.

### If Benign True Positive

1. If no corporate AI account exists, provision one and guide the user to migrate.
2. If personal account usage was incidental, help the user clear saved personal credentials from the corporate browser.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Only enable this detection after your organization has deployed corporate AI accounts. Without an alternative, personal account usage may be the only option.
- Refine cookie parsing rules to match your specific proxy's cookie logging format.
- Exclude users who have been granted explicit personal account exceptions.

### Validating cookie data quality

```spl
| tstats summariesonly=t count values(Web.cookie) as cookies from datamodel=Web
  where Web.url_domain=* earliest=-1d latest=now by Web.user Web.url_domain
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| eval has_cookie=if(isnotnull(cookies) AND cookies!="","yes","no")
| stats count by has_cookie provider
```

If cookie data is sparse or absent, this detection will have limited effectiveness. Consider alternative personal account detection methods such as SSO bypass detection at the IdP level.
