# AI-035 - AI-Assisted Lateral Movement

> **Detection ID:** AI-035
> **Status:** Active
> **Risk Score:** 45 (user) + 25 (system)
> **MITRE ATT&CK:** T1021 - Remote Services (Lateral Movement)
> **Data Model:** Endpoint.Processes
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Collection
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies when AI CLI tools and remote access/lateral movement tools are executed on the same host by the same user within a 15-minute window. The premise is that an attacker or malicious insider may use an AI CLI to generate exploit code, reconnaissance scripts, or attack payloads, then immediately pivot to other hosts using remote access tools like SSH, PsExec, or RDP.

The detection uses a dual `tstats` + `append` architecture. The first query identifies AI CLI tool executions (Codex, Claude, Gemini, GitHub Copilot, Aider, and 12 other AI CLI tools). The second query identifies lateral movement tool executions (SSH, PsExec, WMI, WinRM, net.exe, SMBClient, RDP clients). Both queries are aggregated into 1-hour time buckets by user and host, then joined. An alert fires when both conditions are true: at least one AI CLI execution and at least one lateral movement tool execution on the same host by the same user.

This is a behavioral correlation detection, not a signature detection. The co-occurrence of AI CLI and lateral movement tools is not inherently malicious -- DevOps engineers and system administrators routinely use both. The detection's value is in surfacing this pattern for human analysis, especially when the user is not a known administrator or when the lateral movement targets are sensitive systems.

The risk score is moderate (45 user + 25 system) because the correlation alone does not confirm malicious intent. However, when combined with other AI risk detections (especially AI-004 for AI CLI usage and AI-008 for scripted CLI invocations), the cumulative risk can quickly escalate a user past the notable threshold.

## Detection Logic

### Plain English

1. **First tstats query:** Search `Endpoint.Processes` for AI CLI tool executions (16 tools, both Linux and Windows binary names). Collect process name, command line, user, and host. Group by user and host in 1-hour buckets.
2. **Second tstats query (appended):** Search `Endpoint.Processes` for remote access/lateral movement tool executions (14 tools including SSH, PsExec, WMI, WinRM, net.exe, SMBClient, RDP clients). Group by user and host in 1-hour buckets.
3. **Join:** Merge both result sets by user and host. Count AI CLI events and lateral movement events separately. Collect the specific tools used in each category.
4. **Filter:** Only return results where `ai_cli_count >= 1 AND lateral_count >= 1` (both must be present).
5. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
6. Tag the event with `detection_id=AI-035`.

## Triage Steps

> **Estimated triage time:** 15-25 minutes

### Step 1: Validate the alert and understand the correlation

```spl
index=risk search_name="AI RBA - AI-035*" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note which AI CLI tools and which lateral movement tools were correlated.
- The risk message contains both `ai_tools` and `lateral_tools` for quick context.

### Step 2: Identify the user and their role

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit
```

- **System administrators and DevOps engineers:** Routine SSH + AI CLI usage is expected. Focus on whether the targets are unusual.
- **Developers:** May use AI CLI + SSH for deployments. Check if targets match their assigned infrastructure.
- **Non-technical users:** AI CLI + lateral movement tools is highly anomalous. Escalate immediately.

### Step 3: Examine the AI CLI activity in detail

```spl
| tstats summariesonly=t count values(Processes.process) as process values(Processes.process_path) as process_path from datamodel=Endpoint.Processes where Processes.user="$user$" Processes.dest="$src$" Processes.process_name IN ("codex","codex.exe","claude","claude.exe","gemini","gemini.exe","gh-copilot","gh-copilot.exe","aider","aider.exe","sgpt","sgpt.exe","llm","llm.exe","tgpt","tgpt.exe","mods","mods.exe","fabric","fabric.exe","aichat","aichat.exe","amazon-q","amazon-q.exe","openai","openai.exe","cline","cline.exe") earliest=-2h by Processes.process_name _time span=5m
| rename Processes.* as *
| table _time process_name process process_path count
| sort _time
```

- What did the AI CLI command include? Look at the full process command line.
- Was the user asking the AI to generate exploit code, scripts, or credentials?
- Is the process path a standard installation or a suspicious temp directory?

### Step 4: Examine the lateral movement activity in detail

```spl
| tstats summariesonly=t count values(Processes.process) as process from datamodel=Endpoint.Processes where Processes.user="$user$" Processes.dest="$src$" Processes.process_name IN ("ssh","ssh.exe","psexec","psexec.exe","wmic","wmic.exe","winrm","winrm.cmd","net","net.exe","smbclient","xfreerdp","rdesktop","mstsc","mstsc.exe") earliest=-2h by Processes.process_name _time span=5m
| rename Processes.* as *
| table _time process_name process count
| sort _time
```

- What hosts were targeted? Extract destination IPs or hostnames from the command line.
- Was PsExec used with credentials on the command line (extremely suspicious)?
- Was SSH used with key-based auth (normal) or password on the command line (suspicious)?
- How many distinct targets were accessed? Many targets in a short window indicates network scanning or lateral spread.

### Step 5: Calculate the time delta between AI CLI and lateral movement

```spl
| tstats summariesonly=t min(_time) as event_time values(Processes.process) as process from datamodel=Endpoint.Processes where Processes.user="$user$" Processes.dest="$src$" earliest=-2h by Processes.process_name
| rename Processes.* as *
| eval tool_type=case(
    match(process_name,"codex|claude|gemini|gh-copilot|aider|sgpt|llm|tgpt|mods|fabric|aichat|amazon-q|openai|cline"), "ai_cli",
    match(process_name,"ssh|psexec|wmic|winrm|net|smbclient|xfreerdp|rdesktop|mstsc"), "lateral",
    1=1, "other")
| where tool_type IN ("ai_cli","lateral")
| stats min(event_time) as first_event max(event_time) as last_event by tool_type
| eval first_ai=if(tool_type="ai_cli",first_event,null()), first_lateral=if(tool_type="lateral",first_event,null())
| stats values(first_ai) as first_ai values(first_lateral) as first_lateral
| eval delta_minutes=round(abs(first_lateral-first_ai)/60,1)
| table first_ai first_lateral delta_minutes
```

- A tight delta (under 5 minutes) between AI CLI usage and lateral movement is more suspicious.
- A longer delta (30+ minutes) may indicate unrelated activities that happened to occur in the same window.
- Check if AI CLI usage preceded lateral movement (attack preparation) or followed it (post-exploitation analysis).

### Step 6: Check if the lateral movement targets are sensitive

```spl
| inputlookup asset_lookup_by_str
| search ip IN ($lateral_targets$) OR dns IN ($lateral_targets$) OR nt_host IN ($lateral_targets$)
| table ip dns nt_host owner priority category bunit
```

- Lateral movement to domain controllers, jump servers, database servers, or financial systems is highest priority.
- Movement to development or test servers is less concerning but still warrants investigation.
- Check if the user has authorized access to the target systems.

### Step 7: Correlate with other risk events

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-035 + AI-004 (AI CLI usage) = confirms the AI CLI activity; check if it was sanctioned.
- AI-035 + AI-008 (scripted CLI) = automated AI-to-lateral-movement pipeline is very high risk.
- AI-035 + AI-RISK-003 (kill chain) = lateral movement as part of a confirmed attack chain.
- AI-035 + AI-015 (after-hours) = lateral movement outside business hours increases severity.

### Step 8: Make a determination

- **True Positive:** Non-administrator user using AI CLI to generate attack tools, then immediately pivoting to other hosts. Or any user targeting sensitive systems with no documented access need.
- **Benign True Positive:** DevOps engineer using Claude CLI for infrastructure automation while managing servers via SSH. Developer using Copilot while deploying code via SSH.
- **False Positive:** Two unrelated activities (AI coding and server management) happening to occur in the same time window.

---

## True Positive Indicators

- The AI CLI command line contains references to exploits, vulnerabilities, credential extraction, or network reconnaissance.
- Lateral movement immediately follows AI CLI usage (under 5 minutes delta).
- The lateral movement targets are sensitive systems the user does not normally access.
- The user is not a system administrator, DevOps engineer, or developer.
- PsExec or WMI was used (more commonly associated with attacks than SSH in most environments).
- Multiple distinct lateral movement targets were accessed in rapid succession.
- The activity occurred outside business hours.
- The AI CLI was launched from a scripted context (parent process is bash/PowerShell/python) rather than an interactive terminal.

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| DevOps engineer using AI CLI + SSH as normal workflow (e.g., Claude CLI for script generation, SSH for deployment) | Very High | Allowlist DevOps team members; track their normal target hosts for baseline comparison |
| Developer using AI coding assistant while managing remote dev servers | High | Allowlist developer teams for specific AI CLI tools; focus investigation on target sensitivity |
| Security team using AI CLI to analyze threats while investigating active incidents on remote hosts | Medium | Coordinate with SOC; allowlist security team with time-bounded entries |
| Sysadmin using AI CLI for troubleshooting while performing routine remote maintenance | Medium | Verify targets are within the admin's normal scope; allowlist if consistent |
| Two unrelated activities coinciding in the same time window (user happens to use both tools) | Medium | Check the time delta; longer gaps reduce correlation significance |

## Response Actions

### If confirmed True Positive

1. **Contain:** Immediately disable the user's remote access capabilities. If active lateral movement to sensitive systems is confirmed, isolate both the source and destination hosts.
2. **Investigate AI CLI usage:** Determine what the AI CLI was used for. If process arguments are available, analyze the queries submitted to the AI. If not, check command history on the endpoint.
3. **Investigate destination hosts:** On each lateral movement target, check for:
   - New processes or services started after the connection.
   - Files created or modified.
   - Data accessed or exfiltrated.
   - Credential access (LSASS, SAM, credential stores).
4. **Check for persistence:** Look for new scheduled tasks, services, SSH keys, or user accounts created on the destination hosts.
5. **Assess scope:** Determine the full extent of lateral movement. The detected tools may be only part of the activity.
6. **Escalate:** Notify the IR team and the CISO. AI-assisted lateral movement may indicate a sophisticated threat actor.
7. **Preserve evidence:** Capture process execution logs, command history, network connections, and any AI-generated content from the source host.

### If Benign True Positive

1. Add the user to `ai_sanctioned_entities.csv` with scope limited to "AI CLI" and an appropriate expiration.
2. Document the legitimate workflow that triggered the correlation.
3. Consider whether the user's normal workflow should be baselined to prevent recurring alerts.
4. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Allowlist DevOps and SRE teams at the group level, as they are the primary source of benign correlations.
- If SSH generates excessive noise, consider requiring a minimum `lateral_count` of 2+ or 3+ for SSH-only correlations.
- Consider removing `net.exe` from the lateral movement tool list if it generates too much noise -- `net use` for mapped drives is extremely common.

### Measuring your baseline correlation rate

```spl
| tstats summariesonly=t count from datamodel=Endpoint.Processes where Processes.process_name IN ("codex","claude","gemini","gh-copilot","aider","sgpt","llm","tgpt","mods","fabric","aichat","amazon-q","openai","cline") earliest=-7d by Processes.user Processes.dest _time span=1h
| rename Processes.* as *
| eval _source="ai_cli"
| append [| tstats summariesonly=t count from datamodel=Endpoint.Processes where Processes.process_name IN ("ssh","psexec","wmic","winrm","net","smbclient","xfreerdp","rdesktop","mstsc") earliest=-7d by Processes.user Processes.dest _time span=1h | rename Processes.* as * | eval _source="lateral"]
| stats dc(eval(if(_source="ai_cli",_time,null()))) as ai_hours dc(eval(if(_source="lateral",_time,null()))) as lateral_hours by user dest
| where ai_hours>0 AND lateral_hours>0
| sort -ai_hours
```

This shows which users routinely trigger both conditions. Allowlist the consistent users and focus investigation on new or intermittent correlations.
