# AI-023 - AI Access From Unmanaged Device

> **Detection ID:** AI-023
> **Status:** Active
> **Risk Score:** 30 (user) + 25 (system)
> **MITRE ATT&CK:** T1078.004 - Valid Accounts: Cloud Accounts (Defense Evasion, Persistence)
> **Data Model:** Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Collection
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies users accessing AI services from endpoints that are not registered in the organization's asset inventory or that are flagged as unexpected. Unmanaged devices -- including personal laptops, BYOD phones, guest network endpoints, and shadow IT systems -- represent a significant blind spot for security controls because they typically lack endpoint protection, DLP agents, and certificate inspection capabilities.

When a user accesses an AI service from an unmanaged device, the organization loses visibility into what data is being copied, pasted, or uploaded. Corporate DLP policies, endpoint monitoring, and browser security extensions are absent, meaning sensitive data can flow to AI providers without any content inspection. This makes unmanaged device access one of the most dangerous vectors for AI data exfiltration.

The detection works by looking up each source IP against `asset_lookup_by_str`. If the IP has no asset ID (unregistered) or is marked as `is_expected=false`, the traffic is flagged. The system risk score of 25 is notably high because the unmanaged device itself represents a control gap that amplifies the risk of any AI interaction originating from it.

## Detection Logic

### Plain English

1. Query the `Web` data model for all web traffic, aggregating by user, source IP, and URL domain.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter` to identify AI provider traffic.
3. Look up the source IP against `asset_lookup_by_str` to retrieve asset ID, category, and expected status.
4. Filter for traffic where no asset ID exists (unregistered device) or `is_expected=false`.
5. Aggregate request counts, AI domains, and timing per user, source, and provider.
6. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
7. Tag the event with `detection_id=AI-023`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User accessing AI from unmanaged device | `asmith@corp.com` |
| `src` | Unmanaged device IP address | `192.168.50.99` |
| `provider` | AI service provider accessed | `Anthropic` |
| `request_count` | Number of requests from unmanaged device | `34` |
| `ai_domains` | AI domains accessed | `claude.ai` |

### Thresholds

| Parameter | Default Value | Notes |
|---|---|---|
| Minimum requests | 1 | Any AI access from unmanaged device triggers detection |
| Asset lookup | `asset_lookup_by_str` | IP must be absent or `is_expected=false` |

## Triage Steps

> **Estimated triage time:** 10-15 minutes

### Step 1: Validate the alert and confirm the device is unmanaged

```spl
index=risk search_name="AI RBA - AI-023: AI Access From Unmanaged Device" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the source IP and request count from the risk message.
- Confirm this is a recurring pattern versus a one-time event.

### Step 2: Verify the source IP is truly unmanaged

```spl
| inputlookup asset_lookup_by_str
| search ip="$src$" OR dns="$src$"
| table ip dns nt_host mac owner category is_expected priority bunit
```

- If no results are returned, the device is genuinely unregistered.
- If `is_expected=false`, the device is known but flagged as unexpected on the network.

### Step 3: Identify the user

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Check if the user is a contractor, visitor, or temporary employee who may legitimately use personal devices.
- Privileged or high-priority users on unmanaged devices are especially concerning.

### Step 4: Examine the AI traffic from the unmanaged device

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" Web.src="$src$" earliest=-4h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as total_bytes sum(count) as requests values(urls) as urls values(user_agents) as agents by provider
| eval mb_out=round(total_bytes/1024/1024,2)
```

- What AI providers are being accessed?
- What user agent is being used? Mobile user agents suggest a personal phone or tablet.
- How much data was uploaded from the unmanaged device?

### Step 5: Determine the network segment

```spl
| tstats summariesonly=t count from datamodel=Network_Traffic where All_Traffic.src="$src$" earliest=-4h by All_Traffic.dest All_Traffic.dest_port
| rename All_Traffic.* as *
| stats count by dest dest_port
| sort -count
| head 20
```

- Is the unmanaged device on a guest network, employee Wi-Fi, or a production VLAN?
- An unmanaged device on the corporate VLAN is more concerning than one on a guest network.

### Step 6: Check if the user also accesses AI from managed devices

```spl
| tstats summariesonly=t count from datamodel=Web where Web.user="$user$" earliest=-7d latest=now by Web.src Web.url_domain
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| lookup asset_lookup_by_str ip as src OUTPUTNEW asset_id category is_expected
| eval managed=if(isnotnull(asset_id) AND is_expected!="false","managed","unmanaged")
| stats count as requests by src managed provider
| sort managed -requests
```

- If the user accesses the same AI service from both managed and unmanaged devices, the unmanaged access may be incidental.
- If AI access occurs only from the unmanaged device, this is highly suspicious.

### Step 7: Correlate with other detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
| sort -total_risk
```

- AI-023 + AI-022 (VPN evasion) = layered evasion strategy.
- AI-023 + AI-007 (high volume) = data exfiltration from uncontrolled device.
- AI-023 + AI-026 (personal account) = personal device with personal account = complete control bypass.

### Step 8: Make a determination

- **True Positive:** User deliberately uses an unmanaged device to access AI services, bypassing endpoint security controls.
- **Benign True Positive:** New employee whose device has not yet been registered in asset inventory.
- **False Positive:** Asset inventory is incomplete or stale, causing managed devices to appear unmanaged.

---

## True Positive Indicators

- The source IP is on a guest or BYOD network segment, not the corporate VLAN.
- The user agent indicates a personal device (mobile browser, non-corporate OS version).
- AI access occurs exclusively from the unmanaged device, never from managed endpoints.
- High data upload volume from the unmanaged device to AI services.
- Combined with personal account OAuth mismatch (AI-026) on the same device.
- User has no approved BYOD exception on file.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| New employee or contractor whose device has not been added to asset inventory | High | Implement automated asset registration; review asset onboarding SLA |
| DHCP reassignment causes IP to become unregistered temporarily | Medium | Ensure asset lookup is updated frequently; use MAC or hostname for secondary lookup |
| VM or container with dynamic IP not tracked in asset inventory | Medium | Register dynamic IP ranges or use hostname-based asset tracking |
| Guest Wi-Fi user authenticated via corporate SSO for legitimate AI research | Low | Create separate policy for guest network AI access with appropriate controls |
| Asset inventory stale data -- device was decommissioned but IP reassigned | Medium | Audit asset inventory quarterly; implement automated deprovisioning |

## Response Actions

### If confirmed True Positive

1. **Contain:** Block the unmanaged device's network access or restrict it to a quarantine VLAN. If the user is using a personal device, restrict their corporate account access until the device is enrolled.
2. **Notify:** Contact the user's manager. Explain the security gap created by unmanaged device AI access.
3. **Investigate:** Assess what data may have been uploaded from the unmanaged device. Since no endpoint telemetry exists, rely on proxy logs, DLP alerts, and the user's recent data access patterns.
4. **Remediate:** Enroll the device in MDM if it will be used for work, or enforce a policy that AI access is only permitted from managed endpoints.
5. **Document:** Record the unmanaged device details, network segment, and data exposure assessment.

### If Benign True Positive

1. Register the device in asset inventory with appropriate classification.
2. If BYOD is permitted, ensure the device meets minimum security requirements (MDM enrollment, OS patching).
3. Close the notable as "Benign True Positive" with documentation.

## Tuning Guidance

### Reducing false positives

- Ensure asset inventory is current. Run a weekly reconciliation between DHCP leases and `asset_lookup_by_str`.
- If your organization permits BYOD, create a BYOD asset category and exclude it from this detection or reduce the risk score.
- Consider adding a minimum request count threshold (e.g., 5+ requests) to avoid alerting on single accidental page loads.

### Auditing asset inventory completeness

```spl
| tstats summariesonly=t dc(Web.user) as unique_users count from datamodel=Web where Web.url_domain=* earliest=-7d latest=now by Web.src
| rename Web.* as *
| lookup asset_lookup_by_str ip as src OUTPUTNEW asset_id category
| where isnull(asset_id)
| stats sum(count) as requests max(unique_users) as users by src
| sort -requests
| head 50
```

This identifies the top unregistered IPs generating web traffic. Register legitimate devices to reduce false positives.

### Adjusting risk scores for BYOD environments

If your organization has a formal BYOD program, consider reducing the system risk score from 25 to 10 for devices on the BYOD VLAN while maintaining the full score for devices on production or sensitive network segments.
