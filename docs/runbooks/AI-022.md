# AI-022 - AI Access Via VPN/Proxy Evasion

> **Detection ID:** AI-022
> **Status:** Active
> **Risk Score:** 35 (user) + 20 (system)
> **MITRE ATT&CK:** T1090.003 - Proxy: Multi-hop Proxy (Command and Control)
> **Data Model:** Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Collection
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies users who access AI services through VPN, proxy, or Tor exit nodes rather than the corporate network's standard egress path. Routing AI traffic through anonymization infrastructure is a strong indicator of intentional policy evasion, as it suggests the user is aware that direct AI access is monitored or blocked and is actively circumventing controls.

Unlike accidental policy violations where a user simply visits an AI website out of curiosity, VPN/proxy evasion requires deliberate technical effort. The user must configure a VPN client, browser proxy, or Tor connection before accessing the AI service. This premeditation significantly increases the likelihood that the activity involves sensitive data exfiltration or unauthorized AI usage that the user wants to hide from security monitoring.

The detection works by enriching source IP addresses against the asset inventory to identify endpoints categorized as VPN, proxy, Tor, or anonymizer nodes. It also inspects HTTP user agent strings for tunnel or proxy indicators. When AI provider traffic originates from these evasion vectors, it generates elevated risk scores that reflect the deliberate nature of the bypass attempt.

## Detection Logic

### Plain English

1. Query the `Web` data model for all web traffic, aggregating by user, source IP, URL domain, and HTTP user agent.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter` to identify AI provider traffic.
3. Look up the source IP against `asset_lookup_by_str` to retrieve asset category and expected status.
4. Filter for traffic where the source asset category matches VPN, proxy, Tor, or anonymizer patterns, or where the user agent contains tunnel/proxy/VPN/Tor indicators.
5. Aggregate request counts, AI domains, and evasion methods per user, source, and provider.
6. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
7. Tag the event with `detection_id=AI-022`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User accessing AI via evasion vector | `jdoe@corp.com` |
| `src` | Source endpoint (VPN/proxy IP) | `198.51.100.42` |
| `provider` | AI service provider accessed | `OpenAI` |
| `request_count` | Number of HTTP requests through evasion path | `87` |
| `ai_domains` | AI domains accessed via evasion | `chatgpt.com` |
| `evasion_methods` | Asset category indicating evasion type | `vpn, proxy` |

### Thresholds

| Parameter | Default Value | Notes |
|---|---|---|
| Minimum requests | 1 | Any AI access via VPN/proxy triggers detection |
| Asset category match | `vpn\|proxy\|tor\|anonymizer` | Matched against `asset_lookup_by_str` category field |
| User agent match | `tunnel\|proxy\|vpn\|tor` | Matched against HTTP user agent string |

## Triage Steps

> **Estimated triage time:** 10-15 minutes

### Step 1: Validate the alert and identify the evasion method

```spl
index=risk search_name="AI RBA - AI-022: AI Access Via VPN/Proxy Evasion" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the evasion method (VPN, proxy, Tor) from the risk message.
- Tor usage is significantly more suspicious than corporate VPN usage.

### Step 2: Identify the user and their role

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Determine if the user has a legitimate reason for VPN usage (remote worker, traveler).
- Security researchers or red team members may use VPN/proxy tools as part of their duties.

### Step 3: Examine the source IP and asset classification

```spl
| inputlookup asset_lookup_by_str
| search ip="$src$" OR dns="$src$"
| table ip dns nt_host owner category is_expected priority bunit
```

- Confirm the source IP is truly categorized as a VPN/proxy/Tor endpoint.
- Check if this is a corporate VPN endpoint (less suspicious) versus a personal or commercial VPN service (more suspicious).

### Step 4: Review the AI traffic details

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" Web.src="$src$" earliest=-4h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as total_bytes sum(count) as requests values(urls) as urls values(user_agents) as agents by provider
| eval mb_out=round(total_bytes/1024/1024,2)
```

- What AI providers are being accessed through the evasion path?
- How much data was uploaded? Large volumes through VPN/proxy compound the risk.
- What user agent is being used? A Tor browser user agent is a strong evasion indicator.

### Step 5: Check for normal VPN usage patterns

```spl
| tstats summariesonly=t count from datamodel=Web where Web.user="$user$" earliest=-30d latest=now by Web.src _time span=1d
| rename Web.* as *
| lookup asset_lookup_by_str ip as src OUTPUTNEW category as src_category
| where match(lower(coalesce(src_category,"")),"vpn|proxy")
| timechart span=1d count as vpn_requests
```

- Does this user routinely use VPN for all web traffic (remote worker)?
- A sudden shift to VPN specifically for AI traffic is more suspicious than consistent VPN usage for all activities.

### Step 6: Correlate with other AI detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
| sort -total_risk
```

- AI-022 + AI-007 (high volume upload) = evasion with data exfiltration.
- AI-022 + AI-015 (after-hours) = double evasion attempt.
- AI-022 + AI-020 (DLP) = sensitive data upload through anonymized channel.

### Step 7: Check if VPN/proxy is blocking corporate DLP inspection

```spl
| tstats summariesonly=t count from datamodel=Data_Loss_Prevention where DLP.user="$user$" earliest=-24h by DLP.dlp_type DLP.action
| rename DLP.* as *
| stats sum(count) as dlp_events by dlp_type action
```

- If no DLP events exist for a user uploading significant data through VPN, it confirms the evasion is bypassing content inspection.

### Step 8: Make a determination

- **True Positive:** User deliberately routed AI traffic through anonymization to bypass monitoring or DLP controls.
- **Benign True Positive:** Remote worker using corporate VPN; AI traffic happens to traverse the VPN tunnel.
- **False Positive:** Asset inventory misclassification labels a legitimate corporate egress point as VPN/proxy.

---

## True Positive Indicators

- Source IP belongs to a commercial VPN provider (NordVPN, ExpressVPN, Mullvad) rather than corporate VPN.
- Tor browser user agent string detected.
- User does not normally use VPN for daily work activities.
- AI traffic volume through VPN is significantly higher than through normal egress.
- DLP inspection is absent for the evasion path, suggesting intentional bypass.
- Combined with after-hours access or other evasion indicators.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Remote worker using corporate VPN for all traffic including AI sites | High | Exclude corporate VPN gateway IPs from detection or tag them separately in asset inventory |
| Asset inventory incorrectly classifies a legitimate egress proxy as anonymizer | Medium | Audit asset_lookup_by_str categories for accuracy; correct misclassified entries |
| Security researcher or red team using VPN/Tor for authorized testing | Low | Add to sanctioned entities with provider scope and time-limited expiration |
| Split-tunnel VPN where only AI traffic routes through VPN (rare legitimate case) | Low | Investigate split-tunnel configurations; this pattern is inherently suspicious |
| Cloud-based proxy for compliance in certain regions | Medium | Tag regional compliance proxies in asset inventory with appropriate category |

## Response Actions

### If confirmed True Positive

1. **Contain:** Block the user's VPN/proxy egress path via network controls. If using a personal VPN client, work with endpoint management to remove it.
2. **Notify:** Alert the user's manager and information security team. Tor usage should trigger immediate escalation.
3. **Investigate:** Determine what data was uploaded through the evasion path. Since DLP may have been bypassed, rely on endpoint telemetry, file access logs, and clipboard history.
4. **Remediate:** Enforce network policies that prevent personal VPN/proxy usage on corporate endpoints. Deploy certificate inspection to detect tunnel protocols.
5. **Document:** Record the evasion method, data exposure assessment, and whether DLP controls were successfully circumvented.

### If Benign True Positive

1. If the user is a legitimate remote worker, update asset inventory to properly categorize the corporate VPN gateway.
2. Add the user to `ai_sanctioned_entities.csv` with appropriate provider scope if AI usage is approved.
3. Close the notable as "Benign True Positive" with documentation of the VPN usage context.

## Tuning Guidance

### Reducing false positives

- Exclude known corporate VPN gateway IPs by ensuring they are categorized correctly in `asset_lookup_by_str` (e.g., category="corporate_vpn" rather than "vpn").
- If your organization has a large remote workforce, consider adding a minimum request count threshold to avoid alerting on incidental AI page loads through VPN.
- Allowlist security teams or red team members who use VPN/Tor operationally.

### Identifying VPN/proxy traffic patterns in your environment

```spl
| tstats summariesonly=t count from datamodel=Web where Web.url_domain=* earliest=-7d latest=now by Web.user Web.src Web.url_domain
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| lookup asset_lookup_by_str ip as src OUTPUTNEW category as src_category
| where match(lower(coalesce(src_category,"")),"vpn|proxy|tor|anonymizer")
| stats count as requests dc(user) as unique_users values(src_category) as categories by src
| sort -requests
```

Use this to identify which VPN/proxy IPs are generating the most AI traffic and whether they are corporate or personal.

### Adjusting asset classifications

Update `asset_lookup_by_str` to distinguish corporate VPN from personal/commercial VPN:
- Corporate VPN gateways: `category=corporate_vpn`
- Known commercial VPN exit nodes: `category=vpn`
- Tor exit nodes: `category=tor`
