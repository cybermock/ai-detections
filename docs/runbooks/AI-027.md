# AI-027 - Personal Email To AI Service Upload Correlation

> **Detection ID:** AI-027
> **Status:** Active
> **Risk Score:** 30 (user) + 15 (system)
> **MITRE ATT&CK:** T1048.002 - Exfiltration Over Alternative Protocol: Exfiltration Over Asymmetric Encrypted Non-C2 Protocol (Exfiltration)
> **Data Model:** Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Exfiltration
> **Last Updated:** 2026-02-22

---

## Overview

This detection correlates file uploads to personal email services (Gmail, Outlook.com, Yahoo Mail, ProtonMail, WhatsApp Web) with subsequent uploads to AI services within a one-hour window. This two-stage pattern -- sending data to a personal email account and then uploading to an AI service -- is a well-known data staging technique that allows employees to move corporate data outside the organization's control boundary before processing it with AI tools.

The personal-email-to-AI pipeline is particularly dangerous because it creates two independent exfiltration channels. Even if the organization blocks direct AI service access, an employee can email sensitive documents to their personal account, then access the AI service from a personal device or network using the emailed data. This detection catches the initial staging step while the user is still on the corporate network.

The detection monitors POST/PUT requests to major personal email service domains and correlates them with AI service web traffic from the same user and source within a 1-hour time span. When both personal email upload activity and AI service upload activity are detected with non-zero byte counts, the correlation fires. This dual-channel detection is uniquely valuable because it reveals the intent to stage data through personal channels before AI processing.

## Detection Logic

### Plain English

1. Query the `Web` data model for POST/PUT requests to personal email domains (`mail.google.com`, `outlook.live.com`, `mail.yahoo.com`, `mail.proton.me`, `web.whatsapp.com`) within 1-hour windows, grouped by user and source.
2. Query the `Web` data model for AI service traffic within the same 1-hour windows, grouped by user and source.
3. Enrich AI traffic domains against `ai_provider_domains.csv` via `ai_domains_filter`.
4. Join the two datasets by user and source, keeping only records where both personal email uploads and AI web requests exist.
5. Calculate byte volumes for both channels.
6. Filter for correlations where both personal email MB and AI MB are greater than zero.
7. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
8. Tag the event with `detection_id=AI-027`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User performing the two-stage upload | `rjones@corp.com` |
| `src` | Source endpoint | `10.1.3.44` |
| `provider` | AI service provider | `OpenAI` |
| `personal_mb` | Megabytes uploaded to personal email | `3.5` |
| `ai_mb` | Megabytes uploaded to AI service | `2.1` |
| `email_count` | Number of personal email upload events | `8` |
| `ai_count` | Number of AI web requests | `15` |
| `ai_domains` | AI domains accessed | `chatgpt.com` |

### Thresholds

| Parameter | Default Value | Notes |
|---|---|---|
| Time window | 1 hour | Personal email and AI uploads must fall within same 1-hour span |
| Minimum personal email MB | > 0 | Any upload to personal email triggers the correlation |
| Minimum AI MB | > 0 | Any upload to AI service completes the correlation |

## Triage Steps

> **Estimated triage time:** 15-20 minutes

### Step 1: Validate the alert and note the data volumes

```spl
index=risk search_name="AI RBA - AI-027: Personal Email To AI Service Upload Correlation" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the personal email MB and AI MB from the risk message.
- High volumes in both channels indicate significant data movement.

### Step 2: Identify the user

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Determine the user's role and department.
- Users with access to sensitive data (source code, financial records, customer data) pose higher risk.

### Step 3: Examine the personal email upload traffic

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls
  from datamodel=Web where Web.user="$user$" Web.src="$src$"
  Web.url_domain IN ("mail.google.com","outlook.live.com","mail.yahoo.com","mail.proton.me","web.whatsapp.com")
  Web.http_method IN ("POST","PUT") earliest=-4h
| rename Web.* as *
| stats sum(bytes_out) as total_bytes sum(count) as requests values(urls) as urls by url_domain
| eval mb_out=round(total_bytes/1024/1024,2)
```

- Which personal email service received the data?
- ProtonMail usage is particularly concerning because it provides end-to-end encryption that makes content inspection impossible.
- What are the upload URLs? Attachment upload endpoints vs. plain text composition?

### Step 4: Examine the AI service upload traffic

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" Web.src="$src$" earliest=-4h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as total_bytes sum(count) as requests values(urls) as urls values(user_agents) as agents by provider
| eval mb_out=round(total_bytes/1024/1024,2)
```

- What AI providers received data in the same window?
- Are the AI upload volumes similar to the personal email upload volumes?
- Is the user accessing AI via API or web interface?

### Step 5: Check DLP events for content context

```spl
| tstats summariesonly=t count values(DLP.dlp_type) as dlp_types values(DLP.file_name) as file_names values(DLP.dest) as destinations
  from datamodel=Data_Loss_Prevention where DLP.user="$user$" DLP.src="$src$" earliest=-4h
| rename DLP.* as *
| table _time dlp_type file_name dest count
```

- Were DLP policies triggered during the personal email upload?
- File names and DLP types reveal the sensitivity of data being staged.

### Step 6: Analyze the temporal sequence

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.user="$user$" Web.src="$src$" earliest=-4h
  by Web.url_domain _time span=5m
| rename Web.* as *
| eval channel=case(
  match(url_domain,"mail\\.google\\.com|outlook\\.live\\.com|mail\\.yahoo\\.com|mail\\.proton\\.me|web\\.whatsapp\\.com"),"personal_email",
  true(),"other")
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| eval channel=if(enabled=1,"ai_service",channel)
| where channel IN ("personal_email","ai_service")
| timechart span=5m sum(bytes_out) as bytes by channel
```

- Did the personal email uploads happen before the AI uploads? This sequence suggests data staging.
- Simultaneous activity suggests parallel workflows.

### Step 7: Correlate with other detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-027 + AI-026 (personal account) = personal email + personal AI account = complete offloading.
- AI-027 + AI-007 (high volume) = significant data staged through personal email to AI.
- AI-027 + AI-043 (data access spike) = file access, email staging, then AI upload.

### Step 8: Make a determination

- **True Positive:** User is staging corporate data through personal email before uploading to AI services, indicating deliberate exfiltration.
- **Benign True Positive:** User emailing personal notes and coincidentally browsing an AI service in the same window.
- **False Positive:** Personal email browsing (reading, not uploading) generates false POST events from AJAX/polling.

---

## True Positive Indicators

- Upload volumes to personal email and AI service are proportional, suggesting the same data.
- Personal email upload precedes AI upload in temporal sequence.
- DLP violations triggered during the personal email upload window.
- User accessed sensitive file shares or databases before the email upload.
- ProtonMail or encrypted email service used (indicates intent to hide content).
- User does not normally use personal email for work purposes.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| User checking personal email and browsing AI service independently | High | Require minimum byte thresholds for both channels; assess temporal correlation |
| AJAX/polling requests to personal email domains counted as POST uploads | Medium | Filter for minimum bytes_out threshold on personal email POSTs |
| User emailing AI-generated content to personal account for personal projects | Medium | Assess content sensitivity; lower risk if no corporate data involved |
| Marketing team using AI to draft email campaigns and sending via personal email | Low | Allowlist marketing team members with documented workflow |
| Automated email notifications from SaaS tools coinciding with AI browsing | Low | Exclude known SaaS notification domains from personal email list |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** Block the user's access to personal email services on the corporate network. If data was confirmed staged, restrict AI service access as well.
2. **Notify:** Alert the user's manager and the data governance team. Include the volume and timing correlation evidence.
3. **Investigate:** Determine what data was emailed by reviewing DLP content logs, email gateway logs, and the user's recent file access activity.
4. **Remediate:** Implement policies that block or inspect personal email attachments on the corporate network. Deploy DLP rules for email-to-AI staging patterns.
5. **Document:** Record the two-channel correlation, data volumes, temporal sequence, and data sensitivity assessment.

### If Benign True Positive

1. If personal email and AI usage are coincidental, document the finding and close.
2. If the user has a legitimate workflow involving personal email, document the business justification.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Add a minimum byte threshold for personal email uploads (e.g., 100 KB) to exclude AJAX polling and small read operations.
- Narrow the correlation window from 1 hour to 30 minutes for tighter temporal correlation.
- Exclude users in roles that routinely use personal email for legitimate external communications (e.g., recruiting).

### Measuring personal email upload patterns

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out count from datamodel=Web
  where Web.url_domain IN ("mail.google.com","outlook.live.com","mail.yahoo.com","mail.proton.me","web.whatsapp.com")
  Web.http_method IN ("POST","PUT") earliest=-7d latest=now
  by Web.user _time span=1h
| rename Web.* as *
| eval mb_out=round(bytes_out/1024/1024,2)
| where mb_out>0
| stats perc50(mb_out) as p50 perc90(mb_out) as p90 perc95(mb_out) as p95 max(mb_out) as max_mb count as windows by user
| sort -max_mb
| head 50
```

Use this to understand baseline personal email upload patterns and set appropriate thresholds.
