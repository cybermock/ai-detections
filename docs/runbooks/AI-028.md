# AI-028 - Unsanctioned AI Web Access

> **Detection ID:** AI-028
> **Status:** Active
> **Risk Score:** Dynamic via multiplier (base 15 user x severity_weight x risk_multiplier, 10 system x severity_weight)
> **MITRE ATT&CK:** T1071.001 - Application Layer Protocol: Web Protocols (Command and Control)
> **Data Model:** Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Reconnaissance
> **Last Updated:** 2026-02-22

---

## Overview

This is the foundational detection for unsanctioned web-based AI service access. It consolidates the previously separate AI-001 (ChatGPT), AI-002 (Claude), and AI-003 (Gemini) detections into a single, dynamically driven rule that covers all AI providers defined in `ai_provider_domains.csv`. Any web traffic to an AI provider domain that is not explicitly sanctioned generates a risk event.

AI-028 serves as the broadest net in the detection suite. While other detections focus on specific risk factors (volume, timing, device posture, evasion methods), AI-028 simply answers the question: "Is this user accessing an unsanctioned AI service via the web?" This makes it the most frequently firing detection and the foundation that other detections layer upon. Its risk score is intentionally low to avoid overwhelming the risk index, but it accumulates meaningfully when combined with other signals.

The detection uses a dynamic risk scoring model with two multipliers. The `severity_weight` from `ai_provider_domains.csv` adjusts risk based on provider sensitivity (e.g., a general-purpose AI chatbot scores lower than a specialized code generation service). The `ai_risk_multiplier` macro further adjusts based on contextual factors like user risk history, department sensitivity, or data classification. This dual-multiplier approach ensures that the same base detection produces appropriately calibrated risk scores across different users and providers.

## Detection Logic

### Plain English

1. Query the `Web` data model for all web traffic, aggregating by user, source IP, destination, and URL domain.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter` to identify AI provider traffic.
3. Filter for web-type usage (`usage_type="web"`).
4. Extract the `severity_weight` from the provider lookup (default: 1.0).
5. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
6. Apply the `ai_risk_multiplier` macro for contextual risk adjustment.
7. Calculate final risk scores: user risk = `15 * _final_multiplier * severity_weight`, system risk = `10 * severity_weight`.
8. Tag the event with `detection_id=AI-028`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User accessing unsanctioned AI service | `bwilson@corp.com` |
| `src` | Source endpoint | `10.1.2.88` |
| `provider` | AI service provider | `OpenAI` |
| `ai_domain` | Specific AI domain accessed | `chatgpt.com` |
| `severity_weight` | Provider-specific severity multiplier | `1.5` |
| `_final_multiplier` | Contextual risk multiplier | `1.2` |
| `_mult_user_risk` | Calculated user risk score | `27` |
| `_weighted_sys_risk` | Calculated system risk score | `15` |

### Thresholds

| Parameter | Default Value | Notes |
|---|---|---|
| Base user risk | 15 | Multiplied by severity_weight and risk_multiplier |
| Base system risk | 10 | Multiplied by severity_weight |
| Minimum requests | 1 | Any unsanctioned AI web access triggers detection |

## Triage Steps

> **Estimated triage time:** 5-10 minutes

### Step 1: Validate the alert and note the risk multipliers

```spl
index=risk search_name="AI RBA - AI-028: Unsanctioned AI Web Access" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the severity_weight and risk_multiplier from the risk message.
- Count how many times this detection has fired for the user in the past 24 hours.
- Frequent firing suggests sustained unsanctioned AI usage, not a one-time visit.

### Step 2: Identify the user

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Determine the user's department and role.
- Check if the user is in a department that has pending AI sanctioning (e.g., engineering evaluating new tools).

### Step 3: Review the specific AI providers accessed

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" Web.src="$src$" earliest=-4h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled severity_weight
| where enabled=1
| stats sum(count) as requests sum(bytes_out) as total_bytes values(urls) as urls values(user_agents) as agents by provider ai_domain severity_weight
| eval mb_out=round(total_bytes/1024/1024,2)
| sort -requests
```

- Which AI providers is the user accessing?
- How many distinct providers? Multiple providers suggest tool shopping or experimentation.
- Are they accessing the web interface or API endpoints?

### Step 4: Assess the frequency and duration

```spl
| tstats summariesonly=t count from datamodel=Web where Web.user="$user$" earliest=-7d latest=now by Web.url_domain _time span=1d
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| timechart span=1d count as daily_ai_requests by provider
```

- Is this a one-time exploration or consistent daily usage?
- Daily sustained usage indicates the user has adopted the AI tool for work.
- A one-time visit may be curiosity-driven.

### Step 5: Check the user's sanctioning status

```spl
| inputlookup ai_sanctioned_entities
| search entity="$user$" OR entity="*"
| table entity_type entity provider scope expires owner reason enabled
| sort entity
```

- Is the user in any sanctioned group?
- Has a previous sanctioning request been submitted but not yet processed?
- Is the provider sanctioned for some groups but not the user's group?

### Step 6: Correlate with other AI detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections dc(search_name) as unique_detections by risk_object
```

- AI-028 alone at low volumes is typically low severity.
- AI-028 + any other detection significantly increases the investigation priority.
- Multiple unique detections (3+) indicate the user should be prioritized.

### Step 7: Make a determination

- **True Positive:** User is actively using an unsanctioned AI service for work purposes without authorization.
- **Benign True Positive:** User briefly visited an AI service out of curiosity; no data was uploaded.
- **False Positive:** Embedded AI widgets on non-AI websites trigger domain matches; ad tracking redirects through AI domains.

---

## True Positive Indicators

- Sustained daily access over multiple days.
- Upload data (bytes_out) indicates the user is submitting content, not just browsing.
- User agent indicates active interaction (not embedded iframes or prefetch).
- The AI provider is not in any pending sanctioning review.
- Combined with other detections (upload volume, after-hours, DLP).
- User's department has no approved AI tooling.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Embedded AI widget on a non-AI website triggers domain match | Medium | Review `ai_provider_domains.csv` for overly broad domain entries |
| Ad network or analytics redirect through an AI provider domain | Low | Validate URL paths; pure domain matches may include non-AI subdomains |
| User clicked a shared link to an AI service but did not interact | Medium | Consider adding minimum request count or bytes_out threshold |
| Browser prefetch or DNS prefetch resolves AI domains without user intent | Low | Filter for minimum request count (3+) to exclude prefetch |
| IT team testing AI service accessibility as part of evaluation | Medium | Allowlist IT evaluation accounts with time-limited scope |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** If data was uploaded, restrict the user's access to the AI service via proxy policy.
2. **Notify:** Contact the user to inform them of the AI usage policy. For first-time violations with no data upload, a user education notification may be sufficient.
3. **Investigate:** If bytes_out is significant, determine what data was submitted using proxy content logs or endpoint telemetry.
4. **Remediate:** Offer the user an approved AI alternative if one exists. Enforce AI usage awareness training.
5. **Document:** Record the provider, frequency, and disposition.

### If Benign True Positive

1. If the user's department is approved for the AI provider, add them to `ai_sanctioned_entities.csv`.
2. If the user explored briefly with no data upload, close with a note.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Add a minimum request count (e.g., 3+) to filter out accidental clicks and browser prefetch.
- Review `ai_provider_domains.csv` entries to ensure subdomains are specific enough to avoid matching non-AI traffic.
- Adjust `severity_weight` per provider in `ai_provider_domains.csv` to reflect your organization's risk posture.

### Reviewing the risk multiplier effect

```spl
index=risk search_name="AI RBA - AI-028: Unsanctioned AI Web Access" earliest=-7d
| stats count avg(risk_score) as avg_risk max(risk_score) as max_risk by risk_object
| sort -avg_risk
| head 25
```

If average risk scores are too high, reduce the base user risk from 15 or lower the severity_weight for low-risk providers. If too low, increase the base or add contextual multiplier rules.

### Adjusting provider severity weights

Update `lookups/ai_provider_domains.csv` to calibrate risk per provider:
```csv
domain,provider,usage_type,enabled,severity_weight
chatgpt.com,OpenAI,web,1,1.0
claude.ai,Anthropic,web,1,1.0
deepseek.com,DeepSeek,web,1,2.0
```

Higher-risk providers (e.g., foreign-operated services, services without enterprise agreements) should have higher severity weights.
