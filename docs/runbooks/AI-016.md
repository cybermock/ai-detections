# AI-016 - Privileged Account AI Service Usage

> **Detection ID:** AI-016
> **Status:** Active
> **Risk Score:** 40 (user) + 20 (system) = 60 total
> **MITRE ATT&CK:** T1078.004 - Valid Accounts: Cloud Accounts (Privilege Escalation)
> **Data Model:** Web + Identity (identity_lookup_expanded)
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Actions on Objectives
> **Last Updated:** 2026-02-13

---

## Overview

This detection identifies when privileged accounts (administrators, service accounts, critical-priority users) access AI services through web traffic. Privileged accounts typically have elevated access to sensitive systems, data, and infrastructure. When these accounts interact with external AI services, the risk of inadvertent data exposure is significantly amplified because:

1. Privileged users often have access to production databases, secrets management systems, infrastructure configurations, and customer data.
2. Prompts and uploads from privileged users are more likely to contain sensitive operational details (server names, IP ranges, security configurations, credentials).
3. A compromised privileged account using AI services could be systematically exfiltrating data under the guise of normal AI usage.

This detection leverages Splunk ES's `identity_lookup_expanded` to identify users categorized as "privileged" or with "critical" priority. A single firing generates 60 risk points, immediately reaching the medium risk threshold.

## Detection Logic

### Plain English

1. Query the `Web` data model for all web traffic to any domain.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter` to identify AI provider traffic.
3. Lookup the user in `identity_lookup_expanded` to retrieve their `category` and `priority`.
4. Filter to users where `category="privileged"` OR `priority="critical"`.
5. Aggregate by user, source, and provider: count requests, sum bytes uploaded, collect AI domains.
6. Calculate megabytes uploaded.
7. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
8. Tag the event with `detection_id=AI-016`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | The privileged user account | `admin.jsmith@corp.com` |
| `src` | Source endpoint | `10.1.2.50` |
| `provider` | AI service provider | `Anthropic` |
| `user_category` | Identity category from ES | `privileged` |
| `user_priority` | Identity priority from ES | `critical` |
| `request_count` | Number of requests to AI service | `47` |
| `mb_out` | Megabytes uploaded | `2.3` |
| `ai_domains` | AI domains accessed | `claude.ai, api.anthropic.com` |

### Thresholds

This detection has no configurable thresholds. Any AI service access by a privileged/critical user triggers the alert. The determination of "privileged" status is controlled by Splunk ES's identity framework.

## Triage Steps

> **Estimated triage time:** 10-20 minutes

### Step 1: Validate the alert and the user's privilege level

```spl
index=risk search_name="AI RBA - Privileged Account AI Service Usage" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score search_name
| sort -_time
```

Confirm that the alert is real and note how frequently it has fired.

### Step 2: Verify the user's identity and privilege classification

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit
```

- Confirm that `category=privileged` or `priority=critical` is correctly assigned.
- Identify the user's department, role, and manager.
- Determine what systems/data this user has elevated access to.

### Step 3: Review the AI service activity details

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" earliest=-24h
| eval ai_domain=lower(Web.url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider
| where isnotnull(provider)
| stats sum(count) as requests sum(bytes_out) as total_bytes values(urls) as urls by provider
| eval mb_out=round(total_bytes/1024/1024,2)
| table provider requests mb_out urls
```

- What AI services are they using? Web interface or API?
- How much data is being uploaded? Large uploads from privileged accounts are high risk.
- What user agent is being used? (Browser vs. programmatic access)

### Step 4: Determine if the user is uploading sensitive data

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.user="$user$" Web.http_method="POST" earliest=-24h
| eval mb_out=round(bytes_out/1024/1024,2)
```

- POST requests with large payloads to AI services indicate data submission.
- Cross-reference with DLP events (see AI-020) for content classification.

### Step 5: Check for related detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections dc(search_name) as detection_count by risk_object
| sort -total_risk
```

- AI-016 (60 points) + any other detection = exceeded medium threshold.
- Check for AI-007 (upload volume), AI-020 (DLP), or AI-015 (after-hours).

### Step 6: Review the user's recent access patterns

```spl
| tstats summariesonly=t count from datamodel=Web where Web.user="$user$" by Web.url_domain _time span=1h earliest=-7d
| timechart span=1h count by Web.url_domain
```

- Is this a new behavior or a longstanding pattern?
- Did the user recently change roles or receive new access?

### Step 7: Make a determination

- **True Positive:** Privileged user is accessing unsanctioned AI services with no business justification. Escalate.
- **Benign True Positive:** Privileged user has legitimate need (e.g., security researcher evaluating AI tools). Add to allowlist with tight scope and expiration.
- **False Positive:** User is incorrectly classified as privileged in identity_lookup_expanded. Correct the identity data.

---

## True Positive Indicators

- User holds domain admin, database admin, or infrastructure admin credentials.
- AI service access is to a web interface (not API) -- suggests manual data entry.
- Large upload volume (bytes_out > 1 MB) accompanies the activity.
- User also triggered AI-020 (DLP violation) or AI-007 (high volume upload).
- Activity occurs outside business hours (AI-015 also fired).
- User recently accessed sensitive systems (PAM, secrets vault, production databases) before AI usage.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| User is correctly privileged but has approved AI access for their role | Medium | Add to `ai_sanctioned_entities.csv` with specific provider scope |
| Identity lookup has stale data; user is no longer in a privileged role | Low | Coordinate with the identity management team to update ES identity data |
| Shared/service account that appears privileged but is used for approved automation | Low | Add system-level allowlist entry for the service account's endpoint |
| User visits AI website without uploading data (browsing only, 0 bytes out) | Medium | Consider adding a bytes_out threshold if browsing-only alerts are noisy |

## Response Actions

### If confirmed True Positive

1. **Contain:**
   - Notify the user immediately and request they cease AI service usage until reviewed.
   - If data exposure is suspected, work with the proxy team to block AI domains for this user.
   - If the account may be compromised, initiate a privileged access review (password reset, MFA verification, session termination).

2. **Notify:**
   - Inform the user's manager.
   - Notify the Privileged Access Management (PAM) team.
   - If the user holds admin credentials for critical systems, notify the relevant system owners.

3. **Investigate:**
   - Determine what data the user could have shared based on their access level.
   - Review PAM/vault logs for recent credential checkouts.
   - Check if the user accessed production databases or sensitive file shares before AI usage.
   - Review the AI provider's data handling policies for retention and training usage.

4. **Remediate:**
   - Enforce AI usage policy training.
   - Consider implementing dedicated AI tools with data governance controls for privileged users.
   - Evaluate proxy policies to block or warn privileged accounts accessing AI services.

5. **Document:** Record the incident with details about the user's privilege level and potential data exposure.

### If Benign True Positive

1. Add the user to `ai_sanctioned_entities.csv` with the specific AI provider, a short expiration (30-90 days), and documented business justification.
2. Consider adding monitoring conditions (e.g., approve web access but flag API access).
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Review the identity_lookup_expanded to ensure privilege classifications are current and accurate.
- If certain privileged users have approved AI access, add them to the allowlist with tight provider scope.
- Consider adding a `bytes_out` minimum threshold if zero-byte "browsing only" alerts are too noisy.

### Reviewing privileged user AI activity baseline

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.url_domain=* earliest=-30d latest=now by Web.user Web.url_domain
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| lookup identity_lookup_expanded identity as user OUTPUT category as user_category priority as user_priority
| where user_category="privileged" OR user_priority="critical"
| stats sum(count) as total_requests sum(bytes_out) as total_bytes dc(provider) as unique_providers values(provider) as providers by user user_category user_priority
| eval mb_out=round(total_bytes/1024/1024,2)
| sort -total_requests
```

Use this to understand which privileged users are actively using AI services and at what volume, so you can make informed allowlist decisions.

### Adding allowlist entries

To allowlist a privileged user for a specific AI provider:
```
entity_type,entity,provider,scope,expires,owner,reason,enabled,allow_key
user,admin.jsmith@corp.com,Anthropic,provider,2026-06-30,ciso,approved_claude_for_security_research,1,admin.jsmith@corp.com|anthropic
```
