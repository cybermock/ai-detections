# AI-043 - AI Usage Following Data Access Spike

> **Detection ID:** AI-043
> **Status:** Active
> **Risk Score:** 40 (user) + 20 (system)
> **MITRE ATT&CK:** T1567.002 - Exfiltration Over Web Service: Exfiltration to Cloud Storage (Exfiltration)
> **Data Model:** Endpoint.Filesystem + Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Collection
> **Last Updated:** 2026-02-22

---

## Overview

This detection correlates file access spikes on endpoints with subsequent data uploads to AI services, identifying the classic collection-to-exfiltration kill chain pattern. When a user accesses an unusually high number of files and then uploads data to an AI provider within the same time window, it strongly suggests that the user is collecting corporate data and feeding it into an AI tool for processing, analysis, or exfiltration.

The collection-to-exfiltration pattern is one of the highest-confidence indicators in the AI threat model. While users routinely access files or use AI services independently, the temporal correlation of mass file access followed by AI uploads represents a behavioral sequence that has few innocent explanations -- especially when the file access volume exceeds the configured threshold (default: 100 files) and the AI upload volume exceeds 1 MB.

The detection uses a dual-source approach: it queries the `Endpoint.Filesystem` data model for file read/access/open events within 1-hour spans, then correlates with AI service uploads from the `Web` data model in the same window. The elevated risk scores (40 user + 20 system) reflect the strong signal quality of this multi-phase behavioral correlation. This detection is a key input to the AI-RISK-003 kill chain correlation rule.

## Detection Logic

### Plain English

1. Query the `Endpoint.Filesystem` data model for file access events (read, accessed, opened) within 1-hour spans, grouped by user and source endpoint.
2. Load the data access spike threshold from `ai_detection_config.csv` (default: 100 file accesses).
3. Filter for users exceeding the file access threshold.
4. Query the `Web` data model for AI service traffic within the same 1-hour spans, grouped by user and source.
5. Enrich web domains against `ai_provider_domains.csv` via `ai_domains_filter`.
6. Join the two datasets by user and source, keeping only records where both file access spikes and AI web uploads exist.
7. Calculate the AI upload volume in megabytes.
8. Filter for correlations where AI uploads exceed 1 MB.
9. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
10. Tag the event with `detection_id=AI-043`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User with file access spike followed by AI upload | `dkim@corp.com` |
| `src` | Source endpoint | `10.1.8.42` |
| `provider` | AI service receiving uploaded data | `Anthropic` |
| `file_accesses` | Number of files accessed in the spike | `342` |
| `ai_mb` | Megabytes uploaded to AI service | `15.8` |
| `ai_bytes` | Total bytes uploaded to AI service | `16580608` |
| `accessed_files` | Sample of file paths accessed | `/data/customers/*.csv` |
| `ai_domains` | AI domains data was sent to | `api.anthropic.com` |

### Thresholds

| Parameter | Default Value | Config Lookup Key | Notes |
|---|---|---|---|
| Data access spike count | 100 files | `data_access_spike_count` | Minimum file accesses to qualify as a spike |
| Minimum AI upload | 1 MB | N/A | Hardcoded; filters incidental AI browsing |
| Time window | 1 hour | N/A | File access and AI uploads must fall within same hour |

## Triage Steps

> **Estimated triage time:** 15-25 minutes

### Step 1: Validate the alert and note the correlation details

```spl
index=risk search_name="AI RBA - AI-043: AI Usage Following Data Access Spike" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the file access count and AI upload volume.
- Very high file access counts (500+) with significant AI uploads (10+ MB) are critical priority.

### Step 2: Identify the user and their data access authorization

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- What data does this user have authorized access to?
- Is the user in a role that routinely accesses large numbers of files (e.g., backup admin, data analyst)?

### Step 3: Examine the files accessed during the spike

```spl
| tstats summariesonly=t count values(Filesystem.file_path) as file_paths values(Filesystem.file_name) as file_names
  from datamodel=Endpoint.Filesystem where Filesystem.user="$user$" Filesystem.dest="$src$"
  Filesystem.action IN ("read","accessed","opened") earliest=-4h
| rename Filesystem.* as *
| stats count as total_accesses values(file_paths) as paths values(file_names) as files
```

- What types of files were accessed? Source code, database exports, customer data, financial records?
- Are the files from a single directory (bulk access) or scattered across the filesystem?
- Are there sensitive file types (.csv, .xlsx, .sql, .json, .parquet)?

### Step 4: Assess file sensitivity by directory pattern

```spl
| tstats summariesonly=t count from datamodel=Endpoint.Filesystem
  where Filesystem.user="$user$" Filesystem.dest="$src$"
  Filesystem.action IN ("read","accessed","opened") earliest=-4h
  by Filesystem.file_path
| rename Filesystem.* as *
| rex field=file_path "^(?<directory>.+[\\\\/])"
| stats count as accesses by directory
| sort -accesses
| head 20
```

- Which directories had the most file accesses?
- Directories containing sensitive data (customer records, financials, source code) increase the severity.
- A single directory with hundreds of accesses suggests bulk data collection.

### Step 5: Examine the AI upload traffic details

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" Web.src="$src$" earliest=-4h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as total_bytes sum(count) as requests values(urls) as urls values(user_agents) as agents by provider
| eval mb_out=round(total_bytes/1024/1024,2)
```

- What AI provider received the data?
- Is the upload proportional to the file access volume?
- API access (suggesting automated exfiltration) vs. web interface (suggesting manual copy-paste)?

### Step 6: Analyze the temporal sequence

```spl
| tstats summariesonly=t count from datamodel=Endpoint.Filesystem
  where Filesystem.user="$user$" Filesystem.dest="$src$"
  Filesystem.action IN ("read","accessed","opened") earliest=-4h by _time span=5m
| eval event_type="file_access"
| append [| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.user="$user$" Web.src="$src$" earliest=-4h by Web.url_domain _time span=5m
  | rename Web.* as *
  | eval ai_domain=lower(url_domain)
  | lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
  | where enabled=1
  | eval event_type="ai_upload"]
| timechart span=5m count by event_type
```

- Did file accesses precede AI uploads? This sequence confirms the collection-then-exfiltration pattern.
- Simultaneous activity suggests an automated pipeline.
- File access spike with AI upload delay (hours later) may indicate staging.

### Step 7: Check for DLP or other corroborating signals

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-043 + AI-020 (DLP violation) = confirmed sensitive data in the exfiltration pipeline.
- AI-043 + AI-025 (copy-paste correlation) = file access, clipboard staging, AI upload chain.
- AI-043 + AI-007 (high volume) = significant data volume matching the file access spike.
- AI-043 + AI-008 (scripted CLI) = automated collection-to-exfiltration pipeline.

### Step 8: Make a determination

- **True Positive:** User collected a large number of files and then uploaded data to an AI service, indicating a deliberate data exfiltration workflow.
- **Benign True Positive:** Data analyst routinely processes large datasets and uses an approved AI tool for analysis.
- **False Positive:** File indexing, backup, or antivirus scanning generates high file access counts that coincide with normal AI browsing.

---

## True Positive Indicators

- File access spike involves sensitive directories (customer data, source code, financial records).
- AI upload volume is proportional to the file access volume, suggesting the accessed data was uploaded.
- File access occurs just before AI upload in temporal sequence.
- API-based AI access with automation user agents suggests scripted exfiltration.
- User's role does not justify mass file access.
- Combined with DLP violations or clipboard correlation in the same window.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| File indexing, backup agent, or antivirus scan inflating file access counts | High | Exclude known system processes from filesystem telemetry or increase the access threshold |
| Data analyst legitimately processing large datasets and using AI for analysis | Medium | Allowlist approved data analysts with provider scope |
| IDE or build tool accessing many source files during compilation while user browses AI | Medium | Exclude IDE process file accesses or raise the threshold for developer roles |
| Application startup loading many config/library files coinciding with AI web access | Low | Filter for user-initiated file access actions only |
| Automated report generation accessing many files, user happens to use AI at same time | Medium | Correlate file types with upload content; non-overlapping data is less suspicious |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** Block the user's AI service access immediately. If file access was to sensitive data repositories, restrict the user's file system access pending investigation.
2. **Notify:** Escalate to the incident response team and the data governance team. Notify the user's manager with the evidence of the collection-to-upload correlation.
3. **Investigate:** Determine the full scope of files accessed and cross-reference with the AI upload content. Assess the sensitivity and business impact of the exposed data.
4. **Remediate:** Implement file access monitoring alerts for the affected data repositories. Deploy DLP policies that inspect content destined for AI services. Consider an AI gateway with content filtering.
5. **Document:** Record the complete kill chain timeline: file access spike, data staging, AI upload. Include file paths, data sensitivity assessment, and upload volumes.

### If Benign True Positive

1. If the user's workflow legitimately involves bulk file access and AI tool usage, document the business justification.
2. Add the user to `ai_sanctioned_entities.csv` with appropriate provider scope and expiration.
3. Consider raising the file access threshold for the user's role group.
4. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Increase the `data_access_spike_count` from 100 to 250 if file indexing or build processes generate frequent high-count events.
- Exclude specific process names from filesystem telemetry that are known to generate bulk file access (antivirus, backup agents, search indexers).
- Add minimum AI upload thresholds (e.g., 5 MB instead of 1 MB) to reduce false correlations with casual AI browsing.

### Measuring file access patterns in your environment

```spl
| tstats summariesonly=t count from datamodel=Endpoint.Filesystem
  where Filesystem.action IN ("read","accessed","opened") earliest=-7d latest=now
  by Filesystem.user Filesystem.dest _time span=1h
| rename Filesystem.* as *
| stats perc50(count) as p50 perc90(count) as p90 perc95(count) as p95 perc99(count) as p99 max(count) as max_count by user
| stats avg(p90) as avg_p90 avg(p95) as avg_p95 avg(p99) as avg_p99
```

Set the `data_access_spike_count` threshold above the p95 to avoid alerting on normal file access patterns.

### Adjusting thresholds

Update `lookups/ai_detection_config.csv`:
```csv
control_name,value,notes
data_access_spike_count,250,Adjusted from 100 to reduce false positives from build processes
```
