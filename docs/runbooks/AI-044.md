# AI-044 - Gradual Low-And-Slow Data Exfiltration Via AI

> **Detection ID:** AI-044
> **Status:** Active
> **Risk Score:** 45 (user) + 20 (system)
> **MITRE ATT&CK:** T1567.002 - Exfiltration Over Web Service: Exfiltration to Cloud Storage (Exfiltration)
> **Data Model:** Web
> **Schedule:** `0 6 * * *` (daily at 6:00 AM)
> **Lookback Window:** `-30d@d` to `now` (analyzes 7-day rolling window within 30-day dataset)
> **Kill Chain Phase:** Exfiltration
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies users who have cumulatively uploaded more than a configurable threshold (default: 50 MB) of data to AI services over a rolling 7-day window. It is specifically designed to catch slow-drip exfiltration that evades per-session volume thresholds like AI-007.

Traditional data exfiltration detections focus on large single-session transfers. A sophisticated insider or attacker will avoid this by uploading data in small, inconspicuous batches -- 5 MB today, 8 MB tomorrow, 3 MB the day after. No individual session exceeds the AI-007 threshold, but the cumulative total over a week reveals a pattern of systematic data extraction. This is the "low-and-slow" technique commonly used by insiders who understand their organization's monitoring capabilities.

The detection runs daily (at 6:00 AM by default) rather than every 5 minutes because it needs to evaluate cumulative trends. It queries 30 days of Web data model events, computes daily upload totals per user/source/provider, then applies a rolling 7-day window sum. Users whose 7-day cumulative upload exceeds the threshold generate a risk event. The detection only fires on the most recent day to avoid re-alerting on historical data.

Both the lookback window (default 7 days) and the cumulative threshold (default 50 MB) are configurable via `ai_detection_config.csv`. Organizations with heavy AI usage may need to raise the threshold, while those with strict data controls may lower it.

This detection complements AI-007 (single-session volume), AI-017 (statistical anomaly), and AI-043 (data access + AI correlation). Together, they provide layered coverage across different exfiltration timescales and techniques.

## Detection Logic

### Plain English

1. Query the `Web` data model for all web traffic with `bytes_out`, aggregating by user, source, and URL domain in daily (1-day) buckets.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter` to identify AI provider traffic.
3. Compute daily upload totals (`daily_bytes`, `daily_requests`) per user/source/provider.
4. Load the lookback window (default 7 days) and cumulative threshold (default 50 MB) from `ai_detection_config.csv`.
5. Filter events to only include the configured lookback window (last 7 days).
6. Use `eventstats` to compute the rolling window total (`window_bytes`, `window_requests`) across all days in the window per user/source/provider.
7. Convert to megabytes and filter to users exceeding the cumulative threshold.
8. Only emit results for the most recent day (to avoid re-alerting on historical data).
9. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
10. Tag the event with `detection_id=AI-044`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User performing the uploads | `azhang@corp.com` |
| `src` | Source endpoint | `10.1.4.88` |
| `provider` | AI service provider | `OpenAI` |
| `window_mb` | Total MB uploaded in the rolling window | `67.3` |
| `window_requests` | Total requests in the rolling window | `1,245` |
| `daily_mb` | Today's upload contribution | `8.2` |
| `daily_bytes` | Today's raw bytes uploaded | `8,598,323` |
| `lookback_days` | Configured lookback window | `7` |
| `cumulative_threshold_mb` | Configured threshold | `50` |

### Thresholds

| Parameter | Default Value | Config Lookup Key | Notes |
|---|---|---|---|
| Lookback window | 7 days | `low_slow_days` | Rolling window for cumulative calculation |
| Cumulative threshold | 50 MB | `low_slow_cumulative_mb` | Total upload within the window to trigger alert |
| Suppress period | 24 hours | N/A (alert config) | Suppressed per user/src/provider |

## Triage Steps

> **Estimated triage time:** 20-30 minutes

### Step 1: Validate the alert and understand the cumulative pattern

```spl
index=risk search_name="AI RBA - AI-044*" risk_object="$user$" earliest=-30d
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the total window_mb and the lookback period.
- Check if this user has triggered this detection on previous days (indicates a persistent pattern).

### Step 2: Identify the user and their role

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit
```

- Data scientists and ML engineers may legitimately upload large amounts of data over time to AI APIs.
- Developers using AI coding assistants can accumulate significant upload volume through normal usage.
- Non-technical users with high cumulative uploads to AI services are more concerning.
- Users who have recently given notice or are on a performance improvement plan warrant elevated scrutiny.

### Step 3: Examine the daily upload trend

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out count from datamodel=Web where Web.user="$user$" Web.url_domain=* earliest=-14d by Web.url_domain _time span=1d
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| eval daily_mb=round(bytes_out/1024/1024,2)
| timechart span=1d sum(daily_mb) as daily_mb_total by provider
```

- Is the upload volume consistent day-over-day (steady drip), or did it spike recently?
- A steady daily pattern of 7-10 MB suggests intentional pacing to stay below per-session thresholds.
- A sudden increase from near-zero to daily uploads suggests a new behavior pattern.
- Compare the 7-day window to the previous 7-day window to identify acceleration.

### Step 4: Identify which AI services are receiving the data

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out count values(Web.url) as urls values(Web.http_user_agent) as user_agents from datamodel=Web where Web.user="$user$" earliest=-7d by Web.url_domain
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| eval mb_out=round(bytes_out/1024/1024,2)
| stats sum(mb_out) as total_mb sum(count) as requests values(urls) as urls values(user_agents) as user_agents by provider
| sort -total_mb
```

- Is the data going to a single provider or spread across multiple?
- Is it going to a web interface (chatgpt.com) or API endpoint (api.openai.com)?
- What user agent is being used? API clients (python-requests, curl) suggest programmatic uploads.
- Multiple providers receiving data simultaneously may indicate automated exfiltration across services.

### Step 5: Compare to the user's historical baseline

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out from datamodel=Web where Web.user="$user$" Web.url_domain=* earliest=-30d by Web.url_domain _time span=7d
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| timechart span=7d sum(eval(round(bytes_out/1024/1024,2))) as weekly_mb
```

- Compare the current 7-day window to previous weeks.
- A sudden jump from 10 MB/week to 60 MB/week is much more concerning than a user who consistently uploads 50+ MB/week.
- Consider whether anything changed in the user's role (new project, new tools) that would explain the increase.

### Step 6: Correlate with file access patterns

```spl
| tstats summariesonly=t count values(Filesystem.file_name) as files from datamodel=Endpoint.Filesystem where Filesystem.user="$user$" Filesystem.action IN ("read","accessed") earliest=-7d by Filesystem.file_path _time span=1d
| rename Filesystem.* as *
| search file_path IN ("*confidential*","*proprietary*","*secret*","*restricted*","*trade_secret*","*financial*","*customer*","*pii*")
| stats sum(count) as access_count values(files) as sensitive_files by file_path _time
| sort -access_count
```

- Is the user accessing sensitive files in patterns that correlate with the upload days?
- High file access on days with high AI uploads strongly suggests data is being staged and exfiltrated.
- Check for access to code repositories, databases, customer data, or financial records.

### Step 7: Check for DLP violations in the same window

```spl
| tstats summariesonly=t count values(DLP.file_name) as files values(DLP.dlp_type) as dlp_types from datamodel=Data_Loss_Prevention where DLP.user="$user$" earliest=-7d by _time span=1d
| rename DLP.* as *
| where count>0
| table _time files dlp_types count
| sort -_time
```

- DLP violations in the same window as cumulative uploads is a strong indicator of data exfiltration.
- Note which DLP policy categories were triggered (PII, source code, financial data, trade secrets).

### Step 8: Correlate with other risk events

```spl
index=risk risk_object="$user$" earliest=-7d
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-044 + AI-007 (single-session volume) = user is exfiltrating both in bulk and via slow drip.
- AI-044 + AI-017 (usage anomaly) = statistical confirmation of abnormal behavior.
- AI-044 + AI-043 (data access + AI) = confirmed access-then-upload pattern.
- AI-044 + AI-020 (DLP violation) = confirmed sensitive data in uploads.
- AI-044 + AI-015 (after-hours) = uploads during off-hours to avoid observation.

### Step 9: Make a determination

- **True Positive:** User systematically uploading data to AI services in small daily batches with no business justification, especially if correlated with sensitive file access.
- **Benign True Positive:** Power user of AI coding tools with legitimate high-volume usage (developer uploading code for AI review, data scientist using AI APIs for model training).
- **False Positive:** Cumulative HTTP overhead (headers, keepalives) inflating bytes_out over the 7-day window. Possible but less likely than with AI-007 because the threshold is 50 MB.

## True Positive Indicators

- Cumulative upload volume significantly exceeds the user's historical weekly average.
- Upload pattern shows consistent daily activity (pacing behavior) rather than sporadic bursts.
- The user has accessed sensitive files or databases in the same time window.
- DLP violations correlate with upload days.
- Uploads go to a personal AI account rather than a corporate-provisioned service.
- The user is in a sensitive role (finance, HR, legal, executive assistant) with access to confidential data.
- The user has recently given notice, been placed on a PIP, or has other HR risk indicators.
- Uploads occur via API (automation) rather than web interface (manual).
- Multiple AI providers are receiving data simultaneously.

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Developer using AI coding assistant (Copilot, Cursor, Claude) with high daily usage | High | Allowlist development teams; set higher thresholds for engineering |
| Data scientist uploading training data to approved AI APIs over multiple days | Medium | Validate the project; add to sanctioned entities with provider scope |
| User uploading large documents to AI for summarization or analysis as part of normal workflow | Medium | Check the user's role and the nature of the documents |
| Proxy log overhead accumulating over 7 days to exceed threshold | Low | Validate proxy data accuracy; check if bytes_out includes non-content overhead |
| Automated CI/CD pipelines with AI integration accumulating API calls | Medium | Allowlist service accounts used by CI/CD systems |

## Response Actions

### If confirmed True Positive

1. **Assess data exposure:** Calculate the total data uploaded during the exfiltration window. Use the daily breakdown to estimate what types of data were uploaded on each day.
2. **Determine content type:** If DLP content inspection is available, review what was uploaded. Was it source code, documents, customer data, financial records, or trade secrets?
3. **Block access:** Suspend the user's access to AI services via proxy policy. If API keys were used, revoke them.
4. **Investigate the user's AI account:** If possible, work with the AI provider or your AI gateway to review the user's conversation/API history for evidence of the exfiltrated data.
5. **Check for staging behavior:** Review the user's file access patterns in the weeks before the exfiltration started. Was data being collected and staged before the upload pattern began?
6. **Notify stakeholders:** Inform the user's manager, legal, HR, and the CISO. For confirmed data exfiltration, this is a data breach that may trigger notification obligations.
7. **Preserve evidence:** Capture web proxy logs, file access logs, DLP events, and any available AI service interaction logs for the full exfiltration window.
8. **Assess intentionality:** Determine whether the exfiltration was deliberate (insider threat) or unintentional (user unaware of policy). Interview the user with HR present.

### If Benign True Positive

1. Add the user to `ai_sanctioned_entities.csv` with the specific provider and an expiration date.
2. If the user is a power user of AI tools, consider setting a custom threshold for their account.
3. Document the legitimate business use and the expected weekly upload volume.
4. Recommend that the user use a corporate-provisioned AI service with content controls rather than personal accounts.
5. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Increase the `low_slow_cumulative_mb` threshold if your developer population regularly exceeds 50 MB/week through legitimate AI coding assistant usage.
- Allowlist development and data science teams at the group level.
- Consider separate thresholds for API-based AI access (higher) vs. web-based AI access (lower), as API usage tends to generate more bytes_out.

### Measuring your environment's weekly upload distribution

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out from datamodel=Web where Web.url_domain=* earliest=-30d latest=now by Web.user Web.url_domain _time span=1d
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as daily_bytes by user provider _time
| eval _time_week=strftime(_time,"%Y-W%U")
| stats sum(daily_bytes) as weekly_bytes by user provider _time_week
| eval weekly_mb=round(weekly_bytes/1024/1024,2)
| stats perc50(weekly_mb) as p50_mb perc90(weekly_mb) as p90_mb perc95(weekly_mb) as p95_mb perc99(weekly_mb) as p99_mb max(weekly_mb) as max_mb by provider
```

Set `low_slow_cumulative_mb` above the p95 to avoid alerting on normal usage. For strict environments, set it at p90.

### Adjusting thresholds

Update `lookups/ai_detection_config.csv`:
```csv
control_name,value,notes
low_slow_cumulative_mb,100,Adjusted from 50 to 100 MB based on developer baseline
low_slow_days,14,Extended from 7 to 14 days for longer-window detection
```
