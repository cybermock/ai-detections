# AI-038: Sanctioned AI DLP Violation

## Overview
| Field | Value |
|-------|-------|
| **Detection ID** | AI-038 |
| **MITRE ATT&CK** | T1567.002 — Exfiltration Over Web Service |
| **Kill Chain Phase** | Exfiltration |
| **Risk Score** | User: 45 (multiplied), System: 20 |
| **Data Model** | Data_Loss_Prevention |
| **Cron** | */5 * * * * |

## Description
Detects DLP policy violations involving sensitive data uploaded to sanctioned (approved) AI providers. Even when an AI provider is officially approved for use, sensitive data categories such as PII, source code, financial data, and trade secrets should not be uploaded. This detection complements AI-020 (which catches DLP violations to unsanctioned providers) by monitoring approved channels for data handling policy violations.

## Alert Rationale
Organizations that sanction AI tools often create a false sense of security — employees assume "approved" means "no restrictions." This detection closes that gap by identifying cases where approved AI tools are being fed sensitive data that violates DLP policies, regardless of the provider's sanctioned status.

## Triage Steps

### 1. Identify the User and Context
```spl
index=risk search_name="AI RBA - AI-038*" earliest=-24h
| stats count sum(risk_score) as total_risk values(provider) as providers values(src) as systems by risk_object
| lookup identity_lookup_expanded identity as risk_object OUTPUT department priority
```

### 2. Review DLP Violation Details
```spl
| from datamodel:Data_Loss_Prevention
| search user="$user$" dest IN ("$ai_domains$")
| stats count values(file_name) as files values(dlp_type) as policy_types by user src dest
```

### 3. Assess Data Sensitivity
- What DLP policy was triggered? (PII, source code, financial, HIPAA, etc.)
- What files were involved?
- What is the total volume of data sent?
- Is this a one-time event or a pattern?

### 4. Check User's AI Usage History
```spl
index=risk search_name="AI RBA*" risk_object="$user$" earliest=-7d
| stats count by search_name
| sort -count
```

## Known False Positives
- DLP policies that flag general business documents as sensitive (overly broad DLP rules)
- Users uploading non-sensitive code snippets that trigger source code DLP policies
- Automated integrations between approved AI tools and internal systems that trigger DLP on metadata
- DLP alerts on AI-generated content being sent back to the AI provider

## Tuning Recommendations
- Exclude specific DLP policy types that are known to over-fire (e.g., generic "code" detection)
- Adjust risk multiplier via `ai_department_sensitivity` lookup for departments with legitimate high-volume AI use
- Consider adding a minimum violation count threshold if DLP events are too noisy
- Create sanctioned entity entries for automated AI integrations

## Escalation Criteria
- **Critical**: DLP policy involves PII, PHI, financial data, or trade secrets
- **High**: Source code or internal documentation uploaded to AI service
- **Medium**: General business data flagged by DLP but low sensitivity
- **Low**: False positive from overly broad DLP policy

## Response Actions
1. Contact the user to understand the business context
2. If sensitive data was confirmed uploaded, initiate incident response per data classification policy
3. Review and revoke API keys or sessions if source code or credentials were exposed
4. Consider requesting data deletion from the AI provider (most have data deletion policies)
5. Update DLP policies or AI acceptable use policy if gaps are identified
