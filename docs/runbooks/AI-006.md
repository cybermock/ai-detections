> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Actions on Objectives

## Overview

This detection identifies API calls to AI providers from endpoints that are not classified as development assets. AI API access from non-development endpoints is a higher-risk indicator because legitimate API usage typically occurs from developer workstations, build servers, or authorized applications. API traffic from a finance laptop, HR desktop, or shared kiosk suggests either shadow AI usage, compromised credentials, or unauthorized automation.

The detection relies on Splunk ES's `asset_lookup_by_str` to determine whether the source IP belongs to a "dev/developer/engineering/build" category. If the asset is unclassified or belongs to a non-dev category, the alert fires.

## Detection Logic

### Plain English

1. Query the `Web` data model for traffic to known AI API domains (api.openai.com, api.anthropic.com, etc.).
2. Enrich via `ai_domains_filter` to identify the AI provider and confirm `usage_type="api"`.
3. Look up the source IP in `asset_lookup_by_str` to determine the endpoint's asset category.
4. Filter to events where the asset category is null or does not match "dev", "developer", "engineering", or "build".
5. Exclude sanctioned entities via `ai_unsanctioned_filter`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User making the API call | `jsmith@corp.com` |
| `src` | Source endpoint IP | `10.5.2.100` |
| `provider` | AI provider | `OpenAI` |
| `src_category` | Asset category (if classified) | `null` or `finance` |
| `url_domain` | API domain accessed | `api.openai.com` |
| `bytes_out` | Bytes uploaded | `45000` |
| `http_user_agent` | User agent string | `python-requests/2.31.0` |

## Triage Steps

> **Estimated triage time:** 5-10 minutes

1. **Validate the alert:** Confirm the source IP and user.
2. **Check the asset classification:**
   ```spl
   | inputlookup asset_lookup_by_str
   | search ip="$src$" OR dns="$src$"
   | table ip dns nt_host category priority bunit owner
   ```
3. **Identify the user:** Determine if they are a developer whose asset is simply unclassified.
4. **Review the API traffic:**
   ```spl
   | tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.http_user_agent) as ua
     from datamodel=Web where Web.src="$src$" Web.url_domain IN ("api.openai.com","api.anthropic.com","api.claude.ai") earliest=-24h
   | table count bytes_out ua
   ```
5. **Check what user agent is making the requests.** Programming language libraries (python-requests, node-fetch) confirm API usage.

---

## True Positive Indicators

- Source endpoint is classified as non-technical (finance, HR, executive).
- User agent indicates programmatic access from a language not typically used by the endpoint owner.
- API calls include large payloads (high bytes_out).
- The user does not have developer credentials or API keys in their role.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Developer endpoint not yet classified in asset_lookup_by_str | High | Update asset classifications; allowlist the user |
| IT-managed testing or evaluation from a shared endpoint | Medium | Allowlist the endpoint with appropriate scope |
| Browser extensions making API calls (proxied through web traffic) | Medium | Correlate with AI-021 (browser extension) |

---

## Response Actions

### If confirmed True Positive

1. Investigate what data is being sent via the API.
2. Determine how the user obtained API credentials (personal account vs. corporate).
3. Notify the user's manager.
4. Consider blocking API domains at the proxy for non-dev endpoints.

### If Benign True Positive

1. Update `asset_lookup_by_str` to correctly classify the endpoint, or add the user to the allowlist.
