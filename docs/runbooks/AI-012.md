# AI-012 - DNS Queries To AI Domains

> **Detection ID:** AI-012
> **Status:** Active
> **Risk Score:** 12 (system only)
> **MITRE ATT&CK:** T1071.004 - Application Layer Protocol: DNS (Command and Control)
> **Data Model:** Network_Resolution
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Reconnaissance
> **Last Updated:** 2026-02-13

---

## Overview

This detection identifies DNS resolution queries for known AI service domains. It provides the broadest coverage of AI usage indicators because DNS queries occur regardless of whether the traffic is subsequently proxied, encrypted, or blocked. This makes it valuable as a system-level risk contributor even when user attribution is not available (the user field is set to "unknown" since DNS logs typically lack user context).

The risk score is intentionally low (12 points, system only) because DNS queries alone do not confirm active usage -- they may result from background processes, browser prefetch, or blocked connections.

## Detection Logic

### Plain English

1. Query the `Network_Resolution` data model for DNS queries matching a comprehensive list of known AI domains (50+ domains).
2. Enrich via `ai_domains_filter` to identify the AI provider.
3. Exclude sanctioned entities at the system level.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `src` | System making the DNS query | `10.1.2.50` |
| `query` / `ai_domain` | Domain being resolved | `api.openai.com` |
| `provider` | AI provider | `OpenAI` |
| `record_types` | DNS record types queried | `A, AAAA` |

---

## Triage Steps

> **Estimated triage time:** 3-5 minutes

1. **Check which AI domain was resolved** and whether it is a high-risk provider.
2. **Identify the source system:**
   ```spl
   | inputlookup asset_lookup_by_str | search ip="$src$"
   | table ip dns nt_host category owner bunit
   ```
3. **Check for corresponding web traffic** (AI-006 and AI-028) or process execution (AI-004, AI-005) from the same source.
4. **If no other detections fired, this is likely background/incidental activity.**

---

## True Positive Indicators

- DNS query for an AI API domain (api.openai.com) from a non-dev endpoint.
- Multiple AI domains resolved from the same system in a short window.
- Corresponding web traffic or process execution detected from the same source.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Browser DNS prefetch for AI domains from visited pages or ads | High | Treat as informational; rely on web/process detections for confirmation |
| DNS cache population from network security tools scanning AI domains | Medium | Allowlist security scanning infrastructure |
| Shared DNS resolver (all client queries appear from one IP) | High | If using a central resolver, this detection may need tuning or disabling |

---

## Response Actions

### If confirmed True Positive

1. Investigate the source system for other AI-related detections.
2. DNS-only alerts rarely require immediate response. Escalate only if combined with other detections.

### If Benign True Positive

1. Allowlist the source system if it is a known infrastructure component generating expected DNS queries.
