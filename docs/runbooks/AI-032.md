# AI-032 - Suspicious Model File Exfiltration From ML Infrastructure

> **Detection ID:** AI-032
> **Status:** Active
> **Risk Score:** 50 (user) + 30 (system)
> **MITRE ATT&CK:** T1567.002 - Exfiltration Over Web Service: Exfiltration to Cloud Storage (Exfiltration)
> **Data Model:** Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Exfiltration
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies data uploads from ML/data science infrastructure to external AI service providers. Machine learning infrastructure -- including GPU training clusters, Jupyter notebook servers, and data science workstations -- contains some of the most valuable intellectual property in a modern organization: trained model weights, proprietary training datasets, model architectures, and experimental results. When data flows from these systems to external AI services, it may indicate theft of proprietary AI/ML assets.

The risk scores for this detection are the highest in the shadow AI category (50 user + 30 system) because the combination of ML infrastructure as the source and an external AI provider as the destination creates a near-certain indicator of proprietary model or data exfiltration. Legitimate AI service usage typically originates from general-purpose workstations, not from dedicated ML infrastructure.

The detection works by enriching the source IP against `asset_lookup_by_str` and filtering for assets categorized as ML, data science, GPU, training, Jupyter, or notebook infrastructure. When these systems upload 5+ MB of data to any AI provider domain, the detection fires. The 5 MB threshold filters out incidental browsing or small API queries while catching model weight files, dataset exports, and bulk training data uploads that represent genuine IP theft.

## Detection Logic

### Plain English

1. Query the `Web` data model for all web traffic, aggregating by user, source IP, and URL domain.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter` to identify AI provider traffic.
3. Filter for traffic where a provider was identified.
4. Look up the source IP against `asset_lookup_by_str` to retrieve the asset category.
5. Filter for assets categorized as ML, data science, GPU, training, Jupyter, or notebook infrastructure.
6. Calculate outbound data volume in megabytes.
7. Filter for uploads of 5+ MB.
8. Aggregate request counts, total MB, AI domains, and source categories per user, source, and provider.
9. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
10. Tag the event with `detection_id=AI-032`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User performing the upload from ML infrastructure | `dsci_admin@corp.com` |
| `src` | ML infrastructure endpoint | `10.5.10.20` |
| `provider` | AI service receiving the data | `OpenAI` |
| `total_mb` | Total megabytes uploaded | `245.8` |
| `request_count` | Number of HTTP requests | `18` |
| `ai_domains` | AI domains data was sent to | `api.openai.com` |
| `src_categories` | Asset category of the source | `gpu, training` |

### Thresholds

| Parameter | Default Value | Notes |
|---|---|---|
| Minimum upload size | 5 MB | Filters incidental browsing from genuine data transfers |
| Asset category match | `ml\|data.science\|gpu\|training\|jupyter\|notebook` | Matched against `asset_lookup_by_str` category |

## Triage Steps

> **Estimated triage time:** 15-25 minutes

### Step 1: Validate the alert and note the source infrastructure

```spl
index=risk search_name="AI RBA - AI-032: Suspicious Model File Exfiltration From ML Infrastructure" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the ML infrastructure source, upload volume, and AI provider.
- This is a high-severity detection -- prioritize accordingly.

### Step 2: Identify the user and their ML infrastructure access

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Is the user a data scientist, ML engineer, or researcher with legitimate access to ML infrastructure?
- Check if the user's access to ML infrastructure is current and authorized.

### Step 3: Examine the source asset in detail

```spl
| inputlookup asset_lookup_by_str
| search ip="$src$" OR dns="$src$"
| table ip dns nt_host owner category is_expected priority bunit
```

- Confirm the source is genuinely ML infrastructure (not a misclassified general workstation).
- What type of ML infrastructure? A GPU training cluster vs. a Jupyter notebook server have different risk profiles.
- Who is the registered owner of this asset?

### Step 4: Examine the upload traffic details

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" Web.src="$src$" earliest=-4h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as total_bytes sum(count) as requests values(urls) as urls values(user_agents) as agents by provider
| eval mb_out=round(total_bytes/1024/1024,2)
| sort -mb_out
```

- What AI provider received the data?
- Are uploads going to API endpoints (programmatic transfer) or web interfaces?
- User agents indicating Python, curl, or automation frameworks suggest scripted exfiltration.
- Very large uploads (100+ MB) likely contain model weights or training datasets.

### Step 5: Check what models or datasets were recently accessed

```spl
| tstats summariesonly=t count values(Filesystem.file_path) as file_paths values(Filesystem.file_name) as file_names
  from datamodel=Endpoint.Filesystem where Filesystem.user="$user$" Filesystem.dest="$src$"
  Filesystem.action IN ("read","accessed","opened") earliest=-4h
| rename Filesystem.* as *
| where match(lower(file_name),"\.gguf$|\.ggml$|\.safetensors$|\.pt$|\.pth$|\.bin$|\.h5$|\.onnx$|\.pkl$|\.csv$|\.parquet$|\.json$|\.jsonl$")
| stats count as accesses values(file_paths) as paths values(file_names) as files
```

- Were model files (GGUF, SafeTensors, PyTorch, ONNX) accessed before the upload?
- Were training datasets (CSV, Parquet, JSONL) accessed?
- File access followed by AI upload strongly indicates exfiltration.

### Step 6: Review network traffic from the ML infrastructure

```spl
| tstats summariesonly=t count sum(All_Traffic.bytes_out) as bytes_out from datamodel=Network_Traffic
  where All_Traffic.src="$src$" earliest=-4h by All_Traffic.dest All_Traffic.dest_port
| rename All_Traffic.* as *
| sort -bytes_out
| head 20
| eval mb_out=round(bytes_out/1024/1024,2)
```

- Are there unusual outbound connections from the ML infrastructure?
- Large data transfers to cloud storage (S3, GCS, Azure Blob) alongside AI uploads compound the risk.

### Step 7: Correlate with other detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-032 + AI-007 (high volume) = confirmed massive data transfer from ML infra.
- AI-032 + AI-008 (scripted CLI) = automated exfiltration from ML infrastructure.
- AI-032 + AI-020 (DLP) = DLP-confirmed sensitive data from ML systems.

### Step 8: Make a determination

- **True Positive:** Proprietary model weights, training data, or ML intellectual property is being exfiltrated to an external AI service.
- **Benign True Positive:** Data scientist using an external AI API for legitimate model evaluation or comparison research.
- **False Positive:** ML infrastructure asset category is too broad, capturing general development workstations.

---

## True Positive Indicators

- Upload volume exceeds 50 MB, consistent with model weight or dataset files.
- Model files (GGUF, SafeTensors, PT, ONNX) were accessed on the ML infrastructure before the upload.
- API endpoint access with automation user agents (Python requests, curl, wget).
- User does not have authorization to share proprietary ML assets externally.
- Upload destination is a competitor's AI platform or an unknown service.
- Activity occurs outside normal data science working hours.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Data scientist using external AI API for approved model benchmarking | Medium | Allowlist approved ML team members with specific provider scope |
| ML infrastructure category is too broad and includes general dev workstations | Medium | Audit and refine asset categories in `asset_lookup_by_str` |
| Automated ML pipeline sending telemetry or metrics to an AI service | Low | Exclude known automated pipeline service accounts |
| User browsing AI documentation from an ML workstation (5+ MB of web content) | Low | Increase the MB threshold or add request count minimums |
| Model registry sync between internal and external platforms | Low | Allowlist specific model registry sync processes and destinations |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** Immediately block the ML infrastructure's access to external AI services. Isolate the source system for forensic investigation. Revoke the user's access to ML infrastructure.
2. **Notify:** Escalate to the CISO and the ML/AI team lead. This is a potential IP theft incident requiring executive notification.
3. **Investigate:** Determine exactly what data was uploaded by analyzing the file access timeline, model registry logs, and network captures. Assess the commercial value of the exfiltrated models or data.
4. **Remediate:** Implement network segmentation that prevents ML infrastructure from directly accessing external AI services. Deploy DLP policies for ML file types. Require all external AI interactions from ML systems to route through an approved AI gateway.
5. **Document:** Record the full scope of the exfiltration including file types, volumes, destination, and intellectual property valuation.

### If Benign True Positive

1. Add the user/system combination to `ai_sanctioned_entities.csv` with specific provider scope, time limit, and documented business justification.
2. Ensure the ML team has approved AI service access through proper channels.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Refine ML infrastructure asset categories in `asset_lookup_by_str` to be specific (e.g., `gpu_training`, `jupyter_server`) rather than broad.
- Increase the 5 MB threshold if your ML teams routinely interact with AI APIs for legitimate evaluation.
- Allowlist specific ML team service accounts used for approved external API interactions.

### Auditing ML infrastructure asset classifications

```spl
| inputlookup asset_lookup_by_str
| where match(lower(category),"ml|data.science|gpu|training|jupyter|notebook")
| table ip dns nt_host owner category bunit priority
| sort category
```

Review this list to ensure only genuine ML infrastructure is categorized appropriately.

### Measuring ML infrastructure outbound traffic patterns

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out count from datamodel=Web
  where Web.url_domain=* earliest=-7d latest=now by Web.src Web.url_domain _time span=1d
| rename Web.* as *
| lookup asset_lookup_by_str ip as src OUTPUTNEW category as src_category
| where match(lower(coalesce(src_category,"")),"ml|data.science|gpu|training|jupyter|notebook")
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| eval mb_out=round(bytes_out/1024/1024,2)
| stats sum(mb_out) as total_mb sum(count) as requests by src src_category provider
| sort -total_mb
```
