## Overview

This detection uses statistical z-score analysis to identify users whose daily AI request count or upload volume significantly deviates from their personal historical baseline. It computes per-user mean and standard deviation over a 30-day window and flags days where the z-score exceeds a configurable threshold (default: 3.0). This catches gradual escalation patterns that fixed-threshold detections might miss.

The detection runs once daily at 3:30 AM to minimize performance impact, as it processes 30 days of data per user.

## Detection Logic

### Plain English

1. Query the `Web` data model for all AI-related web traffic over 30 days, bucketed by day.
2. Enrich via `ai_domains_filter`.
3. Calculate daily request counts and bytes out per user/provider.
4. Compute per-user historical averages and standard deviations using `eventstats`.
5. Load minimum baseline days (default: 14) and z-score threshold (default: 3.0) from `ai_detection_config.csv`.
6. Require at least 14 days of baseline data before alerting.
7. Calculate z-scores for both request count and bytes out.
8. Filter to users where either z-score exceeds the threshold.
9. Filter to only the most recent day's data.
10. Exclude sanctioned entities.

### Thresholds

| Parameter | Default Value | Config Lookup Key |
|---|---|---|
| Z-score threshold | 3.0 | `volume_anomaly_zscore` |
| Minimum baseline days | 14 | `volume_anomaly_min_days` |

## Triage Steps

> **Estimated triage time:** 5-10 minutes

1. **Review the anomaly details:** What is the z-score? How many requests did the user make vs. their average?
2. **Check what caused the spike:**
   ```spl
   | tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web
     where Web.user="$user$" earliest=-7d by Web.url_domain _time span=1d
   | rename Web.* as *
   | eval ai_domain=lower(url_domain)
   | lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
   | where enabled=1
   | timechart span=1d sum(count) as requests sum(bytes_out) as bytes by provider
   ```
3. **Determine if the spike is due to a new project, tool adoption, or unusual activity.**
4. **Check for other detections** that may explain the volume increase.

## True Positive Indicators

- Z-score is very high (> 5), indicating extreme deviation from baseline.
- Volume spike is concentrated in uploads (high bytes_out z-score).
- User has no documented project change that would explain increased AI usage.
- Other detections (AI-007, AI-020) also fired.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| User started a new project requiring heavy AI usage | High | Monitor for a few days; adjust if persistent |
| User adopted a new AI tool (first-seen + increased volume) | Medium | Correlate with AI-009 (first-seen) |
| Baseline period is short (user is new or has sparse history) | Medium | Increase `volume_anomaly_min_days` to 21+ |

---

## Response Actions

### If confirmed True Positive

1. Investigate the cause of the volume spike.
2. If data exfiltration is suspected, follow AI-007 and AI-020 runbooks.
3. Document the anomaly for the user's risk profile.

### If Benign True Positive

1. No allowlist action needed; the baseline will naturally adjust over time.
2. If the user's new normal is consistently higher, the z-score will normalize.

## Tuning Guidance

### Calibrating the z-score threshold

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.url_domain=* earliest=-30d by Web.user Web.url_domain _time span=1d
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(count) as daily_requests sum(bytes_out) as daily_bytes by _time user
| eventstats avg(daily_requests) as mean stdev(daily_requests) as stdev by user
| eval z_score=if(stdev>0, round((daily_requests-mean)/stdev, 2), 0)
| stats perc90(z_score) as p90 perc95(z_score) as p95 perc99(z_score) as p99 max(z_score) as max
```

Set the threshold above p95 to reduce noise while catching true outliers.
