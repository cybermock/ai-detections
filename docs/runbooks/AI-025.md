# AI-025 - Copy-Paste Correlation To AI Service

> **Detection ID:** AI-025
> **Status:** Active
> **Risk Score:** 25 (user) + 12 (system)
> **MITRE ATT&CK:** T1115 - Clipboard Data (Collection)
> **Data Model:** Data Loss Prevention + Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Collection
> **Last Updated:** 2026-02-22

---

## Overview

This detection correlates clipboard copy events with subsequent uploads to AI services within a short time window. The copy-paste workflow is the most common method employees use to transfer sensitive data into AI tools: copying source code from an IDE, database query results from a terminal, or confidential document text from a PDF viewer, then pasting it into a ChatGPT or Claude conversation.

Unlike file uploads which DLP tools can often inspect, clipboard-based data transfers are ephemeral and difficult to monitor. By the time the data reaches the AI service via an HTTP POST, the clipboard content may have been overwritten, leaving no forensic trace on the endpoint. This detection bridges that gap by correlating DLP clipboard events with temporally adjacent AI web traffic, creating a composite signal that neither data source would produce alone.

The detection uses a dual-source approach: it queries the `Data_Loss_Prevention` data model for clipboard/copy/paste events and the `Web` data model for AI service traffic, then joins them by user and source within 5-minute time windows. When both clipboard activity and AI uploads occur in the same window, the correlation fires. The moderate risk scores (25 user + 12 system) reflect that while clipboard usage alone is normal, its pairing with AI service uploads warrants investigation.

## Detection Logic

### Plain English

1. Query the `Data_Loss_Prevention` data model for clipboard-related events (copy, paste, cut) within 5-minute windows, grouped by user and source.
2. Query the `Web` data model for AI service traffic within the same 5-minute windows, grouped by user and source.
3. Enrich web domains against `ai_provider_domains.csv` via `ai_domains_filter`.
4. Join the two datasets by user and source, keeping only records where both clipboard events and AI web requests exist in the same window.
5. Calculate the correlation count and outbound data volume.
6. Filter for correlations with at least 2 combined events.
7. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
8. Tag the event with `detection_id=AI-025`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User performing copy-paste to AI | `klee@corp.com` |
| `src` | Source endpoint | `10.2.3.15` |
| `provider` | AI service receiving pasted data | `OpenAI` |
| `clipboard_count` | Number of clipboard events in window | `8` |
| `ai_count` | Number of AI web requests in window | `12` |
| `mb_out` | Megabytes uploaded to AI service | `2.3` |
| `ai_domains` | AI domains accessed | `chatgpt.com` |
| `correlation_count` | Combined clipboard + AI event count | `20` |

### Thresholds

| Parameter | Default Value | Notes |
|---|---|---|
| Minimum correlation count | 2 | At least 1 clipboard event + 1 AI request |
| Time window | 5 minutes | Clipboard and AI events must fall within same 5-minute span |

## Triage Steps

> **Estimated triage time:** 15-20 minutes

### Step 1: Validate the alert and note the correlation strength

```spl
index=risk search_name="AI RBA - AI-025: Copy-Paste Correlation To AI Service" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the clipboard count versus AI request count. High clipboard counts (10+) with corresponding AI uploads indicate repeated copy-paste cycles.
- Check if this is a recurring pattern or a one-time event.

### Step 2: Identify the user and their access context

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Developers may routinely copy-paste code snippets into AI tools for assistance.
- Users in sensitive roles (finance, legal, HR) copy-pasting into AI tools is more concerning.

### Step 3: Examine the DLP clipboard events

```spl
| tstats summariesonly=t count values(DLP.dlp_type) as dlp_types values(DLP.file_name) as file_names
  from datamodel=Data_Loss_Prevention where DLP.user="$user$" DLP.src="$src$" earliest=-4h
| rename DLP.* as *
| search dlp_type="*clipboard*" OR dlp_type="*copy*" OR dlp_type="*paste*"
| table _time dlp_type file_name count
```

- What type of clipboard events were detected?
- Are file names associated with the clipboard events? These indicate the source application.
- Look for patterns suggesting bulk data staging (many copy events in rapid succession).

### Step 4: Examine the AI upload traffic

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" Web.src="$src$" earliest=-4h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as total_bytes sum(count) as requests values(urls) as urls by provider
| eval mb_out=round(total_bytes/1024/1024,2)
```

- What AI providers received the data after the clipboard events?
- Is the upload volume proportional to clipboard activity?
- What URLs were accessed -- web chat interface or API endpoints?

### Step 5: Assess what applications the user was copying from

```spl
| tstats summariesonly=t count values(Processes.process_name) as processes from datamodel=Endpoint.Processes
  where Processes.user="$user$" Processes.dest="$src$" earliest=-4h
  by Processes.process_name _time span=5m
| rename Processes.* as *
| stats count by process_name
| sort -count
```

- Active applications during the correlation window reveal the data source.
- Database clients, IDE editors, file viewers, or email clients indicate what type of data was copied.

### Step 6: Check for related detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-025 + AI-007 (high volume) = clipboard staging with bulk upload.
- AI-025 + AI-020 (DLP violation) = confirmed sensitive data copied to AI.
- AI-025 + AI-043 (data access spike) = file access spike followed by copy-paste to AI.

### Step 7: Review the temporal correlation closely

```spl
| tstats summariesonly=t count from datamodel=Data_Loss_Prevention
  where DLP.user="$user$" DLP.src="$src$" earliest=-2h by _time span=1m
| eval event_type="clipboard"
| append [| tstats summariesonly=t count from datamodel=Web
  where Web.user="$user$" Web.src="$src$" earliest=-2h by Web.url_domain _time span=1m
  | rename Web.* as *
  | eval ai_domain=lower(url_domain)
  | lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
  | where enabled=1
  | eval event_type="ai_request"]
| timechart span=1m count by event_type
```

- Visualize the minute-by-minute correlation. Clipboard events closely followed by AI requests (within 1-2 minutes) strongly indicate copy-paste-submit workflows.

### Step 8: Make a determination

- **True Positive:** User systematically copied sensitive data and pasted it into an AI service.
- **Benign True Positive:** Developer copying non-sensitive code snippets for AI coding assistance.
- **False Positive:** DLP clipboard events coincidentally overlap with unrelated AI browsing activity.

---

## True Positive Indicators

- High clipboard event count (10+) tightly correlated with AI uploads.
- Clipboard events originate from sensitive applications (database clients, financial tools, HR systems).
- Upload volume is proportional to clipboard activity volume.
- User does not have an approved AI usage exception.
- Multiple copy-paste-submit cycles visible in temporal analysis.
- Combined with AI-020 DLP violations confirming sensitive content.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Developer copying code snippets for AI coding assistance | High | Allowlist development teams for approved AI providers |
| Clipboard monitoring software generating events for routine copy-paste | Medium | Tune DLP clipboard policy to exclude low-sensitivity clipboard types |
| User copying non-sensitive public information into AI for summarization | Medium | Assess data sensitivity based on source application, not just clipboard count |
| DLP events and AI traffic coincide but are unrelated activities | Low | Tighten the time window from 5 minutes to 2 minutes to reduce false correlations |
| Automated clipboard sync tools generating noise | Low | Exclude known clipboard manager process names from DLP events |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** Restrict the user's AI service access via proxy policy. If sensitive data was confirmed uploaded, initiate incident response procedures.
2. **Notify:** Alert the user's manager and the data governance team. Include details of the source application and data type.
3. **Investigate:** Determine the specific data that was copied using endpoint telemetry, DLP content inspection logs, and the user's recent application activity.
4. **Remediate:** Deploy clipboard monitoring policies that alert on or block clipboard content transfers to AI services. Consider deploying an AI gateway with content inspection.
5. **Document:** Record the correlation timeline, source applications, data sensitivity assessment, and upload volumes.

### If Benign True Positive

1. Add the user to `ai_sanctioned_entities.csv` with provider scope and expiration.
2. Document the business justification for clipboard-to-AI workflows.
3. Close the notable as "Benign True Positive."

---

## Tuning Guidance

### Reducing false positives

- Tighten the correlation time window from 5 minutes to 2 minutes if you are seeing too many coincidental overlaps.
- Exclude specific DLP clipboard event types that are low-fidelity (e.g., system clipboard sync, password manager fills).
- Allowlist development teams who routinely use AI coding assistants.

### Measuring clipboard-to-AI correlation rates

```spl
| tstats summariesonly=t count from datamodel=Data_Loss_Prevention
  where (DLP.dlp_type="*clipboard*" OR DLP.dlp_type="*copy*" OR DLP.dlp_type="*paste*")
  earliest=-7d latest=now by DLP.user _time span=5m
| rename DLP.* as *
| eventstats dc(user) as total_clipboard_users
| stats count as clipboard_windows dc(user) as clipboard_users by _time
```

Compare clipboard event frequency against AI upload frequency to understand the baseline correlation rate.
