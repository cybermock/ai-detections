# AI-042 - New AI Provider Adoption Velocity

> **Detection ID:** AI-042
> **Status:** Active
> **Risk Score:** 20 (user) + 10 (system)
> **MITRE ATT&CK:** T1071.001 - Application Layer Protocol: Web Protocols (Command and Control)
> **Data Model:** Web
> **Schedule:** `0 5 * * *` (daily at 5:00 AM)
> **Lookback Window:** `-30d@d` to `now`
> **Kill Chain Phase:** Reconnaissance
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies users who rapidly adopt multiple new AI service providers within a configurable time window (default: 3 or more new providers within 7 days). While exploring a single new AI service may reflect curiosity or legitimate evaluation, rapidly adopting several new providers in quick succession suggests either systematic "AI shopping" to find the least monitored service, or a compromised account being used to stage data across multiple external platforms.

Rapid AI provider adoption is a reconnaissance-phase indicator that often precedes more serious exfiltration activity. An insider threat actor may test multiple AI services to determine which one is not blocked by the corporate proxy, which offers the largest file upload limits, or which provides the least logging. Similarly, an external attacker who has compromised credentials may rapidly test multiple AI platforms to find the most effective exfiltration channel.

The detection works by calculating the first-seen date for each user-provider combination over a 30-day window, then counting how many providers have a first-seen date within the configurable velocity window (default: 7 days). When a user exceeds the velocity count threshold (default: 3 new providers), the detection fires. The low risk scores (20 user + 10 system) reflect that this is a reconnaissance indicator -- it gains significance primarily when combined with other behavioral signals.

## Detection Logic

### Plain English

1. Query the `Web` data model for all AI service traffic over the past 30 days, grouped by user, source, and URL domain.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter`.
3. Calculate the first-seen date for each user-provider combination.
4. Load velocity window (default: 7 days) and velocity count (default: 3) from `ai_detection_config.csv`.
5. Filter for providers with first-seen dates within the velocity window (relative to now).
6. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
7. Count distinct new providers per user.
8. Filter for users exceeding the velocity count threshold.
9. Tag the event with `detection_id=AI-042`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User rapidly adopting new AI providers | `nwong@corp.com` |
| `src` | Source endpoint | `10.1.7.33` |
| `new_providers` | Count of new providers in velocity window | `5` |
| `providers` | Names of the new providers | `OpenAI, Anthropic, DeepSeek, Mistral, Groq` |
| `provider` | Comma-separated provider list | `OpenAI,Anthropic,DeepSeek,Mistral,Groq` |

### Thresholds

| Parameter | Default Value | Config Lookup Key | Notes |
|---|---|---|---|
| Velocity window | 7 days | `new_provider_velocity_days` | Lookback window for first-seen providers |
| Velocity count | 3 providers | `new_provider_velocity_count` | Minimum new providers to trigger detection |

## Triage Steps

> **Estimated triage time:** 10-15 minutes

### Step 1: Validate the alert and note the new providers

```spl
index=risk search_name="AI RBA - AI-042: New AI Provider Adoption Velocity" risk_object="$user$" earliest=-7d
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note the number of new providers and their names.
- Are the new providers mainstream (OpenAI, Anthropic) or obscure (DeepSeek, Groq, smaller services)?

### Step 2: Identify the user

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit department
```

- Is the user in a role that evaluates AI tools (product manager, engineer, researcher)?
- Is the user in a department with no known AI evaluation activity?

### Step 3: Examine the first-seen dates for each provider

```spl
| tstats summariesonly=t count min(_time) as first_seen max(_time) as last_seen from datamodel=Web
  where Web.user="$user$" earliest=-30d latest=now by Web.url_domain
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats min(first_seen) as first_seen max(last_seen) as last_seen sum(count) as total_requests by provider
| eval first_seen=strftime(first_seen,"%Y-%m-%d %H:%M"), last_seen=strftime(last_seen,"%Y-%m-%d %H:%M")
| sort first_seen
```

- Were the new providers accessed in rapid succession (same day) or spread across the week?
- Same-day exploration of 5+ providers is more suspicious than gradual exploration over 7 days.

### Step 4: Assess the depth of engagement with each provider

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out values(Web.url) as urls
  from datamodel=Web where Web.user="$user$" earliest=-7d latest=now by Web.url_domain
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(count) as requests sum(bytes_out) as total_bytes values(urls) as urls by provider
| eval mb_out=round(total_bytes/1024/1024,2)
| sort -requests
```

- Shallow engagement (1-2 requests per provider) suggests browsing or testing availability.
- Deep engagement (50+ requests, significant uploads) suggests active data transfer.
- Data uploads to multiple providers is concerning -- it suggests the user is distributing data across services.

### Step 5: Check if the user has approval for AI evaluation

```spl
| inputlookup ai_sanctioned_entities
| search entity="$user$"
| table entity provider scope expires owner reason enabled
```

- Does the user have an approved AI evaluation scope that covers multiple providers?
- Has the user been asked to evaluate AI tools for their team?

### Step 6: Correlate with other detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-042 + AI-009 (first-seen provider) = confirms new provider access.
- AI-042 + AI-007 (high volume) = new provider exploration with significant data uploads.
- AI-042 + AI-018 (multiple services in session) = concentrated multi-provider access.
- AI-042 + AI-040 (peer group anomaly) = usage pattern anomalous relative to department peers.

### Step 7: Make a determination

- **True Positive:** User is systematically testing multiple AI services to identify an exfiltration path or the least-monitored service.
- **Benign True Positive:** Product manager or engineer evaluating AI tools for potential organizational adoption.
- **False Positive:** User clicked links in an AI comparison article, briefly visiting multiple provider homepages without interaction.

---

## True Positive Indicators

- Multiple new providers accessed on the same day with data uploads to each.
- Providers include obscure or foreign-operated services (not just mainstream providers).
- User's role does not include AI tool evaluation responsibilities.
- Data uploads (bytes_out) to multiple new providers suggest data distribution.
- Combined with high volume upload (AI-007) to one or more of the new providers.
- User has not requested approval for AI evaluation.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Product team evaluating multiple AI providers for organizational adoption | High | Allowlist users with documented AI evaluation projects |
| Developer testing AI APIs as part of product integration work | Medium | Allowlist engineering teams with approved AI integration scope |
| User reading AI comparison articles that redirect to multiple provider pages | Medium | Increase minimum request count per provider to 3+ |
| New employee exploring available AI tools during onboarding week | Medium | Exclude users in their first 14 days; document onboarding exploration |
| Industry conference or hackathon where multiple AI tools are demonstrated | Low | Cross-reference with calendar events or known conference dates |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** If data was uploaded to multiple new providers, restrict the user's AI service access broadly.
2. **Notify:** Contact the user to understand the business purpose. If none exists, escalate to the user's manager.
3. **Investigate:** Determine if sensitive data was uploaded to any of the new providers. Check bytes_out for each provider.
4. **Remediate:** If unauthorized, enforce AI usage policy. If the user needs to evaluate AI tools, channel the evaluation through an approved process.
5. **Document:** Record the providers accessed, engagement depth, and investigation findings.

### If Benign True Positive

1. If the user has an approved AI evaluation project, document it and add to `ai_sanctioned_entities.csv` with multi-provider scope.
2. If the evaluation is complete, remove the sanctioning when the project concludes.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Increase the `new_provider_velocity_count` from 3 to 5 if your organization has many users evaluating AI tools.
- Shorten the `new_provider_velocity_days` from 7 to 3 to focus on rapid same-week adoption.
- Add a minimum request count per provider (e.g., 5+) to exclude drive-by page visits.

### Monitoring provider adoption patterns across the organization

```spl
| tstats summariesonly=t count min(_time) as first_seen from datamodel=Web
  where Web.url_domain=* earliest=-30d latest=now by Web.user Web.url_domain
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats min(first_seen) as first_seen by user provider
| where first_seen>=relative_time(now(),"-7d")
| stats dc(provider) as new_providers values(provider) as providers by user
| sort -new_providers
| head 25
```

This shows which users are adopting the most new providers, helping identify both evaluation activity and potential threats.

### Adjusting thresholds

Update `lookups/ai_detection_config.csv`:
```csv
control_name,value,notes
new_provider_velocity_days,7,Window for measuring new provider adoption
new_provider_velocity_count,5,Increased from 3 to accommodate evaluation teams
```
