# AI-009 - First-Seen AI Provider For User

> **Detection ID:** AI-009
> **Status:** Active
> **Risk Score:** 15 (user) + 8 (system) = 23 total
> **MITRE ATT&CK:** T1071.001 - Application Layer Protocol: Web Protocols (Command and Control)
> **Data Model:** Web
> **Schedule:** `*/15 * * * *` (every 15 minutes)
> **Lookback Window:** `-30d@d` to `now`
> **Kill Chain Phase:** Reconnaissance
> **Last Updated:** 2026-02-13

---

## Overview

This detection identifies the first time a user accesses a new AI service provider within a 30-day lookback window. First-seen provider activity is a behavioral anomaly indicator: a user who has never accessed DeepSeek suddenly visiting it may be exploring new exfiltration channels, responding to a phishing link, or simply trying a new tool. On its own the risk is low (23 points), but when combined with other detections it contributes to a meaningful risk picture.

## Detection Logic

### Plain English

1. Query the `Web` data model for all traffic to known AI domains over the past 30 days.
2. Enrich via `ai_domains_filter` to map domains to providers.
3. Aggregate to find the earliest access time per user/source/provider.
4. Filter to users whose first access to the provider occurred within the last 15 minutes (current run window).
5. Exclude sanctioned entities.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User accessing the new provider | `jsmith@corp.com` |
| `src` | Source endpoint | `10.1.2.50` |
| `provider` | Newly accessed AI provider | `DeepSeek` |
| `first_seen` | Timestamp of first access | `2026-02-13T10:30:00` |
| `total_requests` | Total requests to the provider | `5` |

---

## Triage Steps

> **Estimated triage time:** 5 minutes

1. **Check if the provider is significant:** Some providers are more concerning than others (e.g., DeepSeek, Perplexity vs. Microsoft Copilot).
2. **Review user history:**
   ```spl
   | tstats summariesonly=t count from datamodel=Web where Web.user="$user$" earliest=-30d by Web.url_domain
   | rename Web.* as *
   | eval ai_domain=lower(url_domain)
   | lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
   | where enabled=1
   | stats sum(count) as requests by provider
   ```
3. **Check for related detections:** If AI-007 or AI-020 also fired, escalate priority.
4. **Determine if the user is exploring or actively using the service.**

## True Positive Indicators

- User accesses a provider with higher risk (foreign-owned, less regulated).
- First-seen provider access combined with high upload volume.
- User has no documented reason to try new AI services.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| User casually tries a new AI service after reading about it | High | Low-risk on its own; monitor for escalation |
| Browser redirect or ad click lands on an AI domain | Medium | Check request count; single requests may be incidental |
| New employee whose 30-day baseline is not established yet | Medium | Suppress for users with < 7 days of activity |

---

## Response Actions

### If confirmed True Positive

1. Monitor the user for additional AI detections over the next 24-48 hours.
2. If the provider is unsanctioned, remind the user of the AI usage policy.
3. No immediate containment needed for this detection alone.

### If Benign True Positive

1. No action needed if the user is exploring approved tools. Consider allowlisting if expected to recur.
