# AI-033 - AI API Key Exposure In Code/DLP

> **Detection ID:** AI-033
> **Status:** Active
> **Risk Score:** 55 (user) + 25 (system)
> **MITRE ATT&CK:** T1552.001 - Unsecured Credentials: Credentials In Files (Credential Access)
> **Data Model:** Raw search (sourcetype-based)
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Staging
> **Last Updated:** 2026-02-22

---

## Overview

This detection identifies AI provider API keys exposed in code repositories, DLP alerts, secret scanner findings, and log files. It uses regex patterns to match 11 distinct API key formats from major AI providers including OpenAI, Anthropic, Google Gemini, HuggingFace, Replicate, Together, Groq, Fireworks, xAI, and OpenRouter.

Exposed API keys are a critical security risk for several reasons. First, AI API keys provide direct access to paid services, and attackers can run up substantial charges within minutes. Second, an exposed key may grant access to conversation histories, fine-tuned models, or organizational data stored with the AI provider. Third, attackers can use stolen keys to make API calls that appear to originate from the victim organization, enabling social engineering or data exfiltration under the organization's identity.

The detection searches across multiple sourcetypes (GitHub/GitLab push events, DLP scans, secret scanners, code review systems) to catch key exposures regardless of where they surface. Each matched key is tagged with a pattern ID (AK-001 through AK-015), the provider name, and a severity level (critical or high) to help analysts prioritize response.

Because API key exposure requires immediate rotation regardless of whether the key was actively exploited, this detection carries a relatively high risk score of 55 for the user entity. The time between exposure and key rotation is the critical metric -- every minute a key remains exposed increases the risk of unauthorized use.

## Detection Logic

### Plain English

1. Search across code repository, DLP, code review, and secret scanner sourcetypes for raw events matching any of 11 AI API key regex patterns.
2. For each match, tag the event with a pattern ID (AK-001 through AK-015), the AI provider name, and a severity level.
3. Expand multi-value matched patterns and extract the pattern ID, provider, and severity into separate fields.
4. Aggregate results by user, source host, and provider, collecting the exposure count, source types, severity levels, file paths, and matched patterns.
5. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
6. Tag the event with `detection_id=AI-033`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User who committed or created the file | `jdoe@corp.com` |
| `src` | Source host where key was found | `github-runner-01` |
| `provider` | AI provider whose key was exposed | `OpenAI` |
| `exposure_count` | Number of key exposures found | `3` |
| `sources` | Sourcetypes that detected the key | `github:enterprise:audit` |
| `severities` | Severity levels of exposed keys | `critical` |
| `file_paths` | File paths where keys were found | `src/config/api_client.py` |
| `matched_patterns` | Pattern IDs that matched | `AK-001, AK-002` |

## Triage Steps

> **Estimated triage time:** 15-25 minutes (includes key rotation coordination)

### Step 1: Validate the alert and assess severity

```spl
index=risk search_name="AI RBA - AI-033*" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score risk_message
| sort -_time
```

- Note which provider's key was exposed and the severity level.
- Critical severity (OpenAI, Anthropic, Google) means immediate rotation is required.
- Check if this user has had prior key exposure events.

### Step 2: Identify the user and their role

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit
```

- Developers and engineers are the most likely source of accidental key commits.
- Non-technical users with exposed keys may indicate a broader compromise or credential sharing.
- Note the user's manager for notification purposes.

### Step 3: Determine the exposure source and scope

```spl
sourcetype IN ("*github*","*gitlab*","*bitbucket*","*dlp*","*code_review*","*secret_scan*") user="$user$" earliest=-24h
| where match(_raw, "sk-[A-Za-z0-9]|sk-proj-|sk-ant-|AIzaSy|hf_[A-Za-z]|r8_[A-Za-z]|tog-|gsk_|fw_[A-Za-z]|xai-|sk-or-")
| table _time sourcetype source host user _raw
| sort -_time
```

- **GitHub/GitLab push event:** Key was committed to a repository. Check if the repo is public or internal.
- **DLP scan:** Key was found in a document or file by the DLP system. Check the file path and type.
- **Secret scanner:** A dedicated tool (e.g., GitLeaks, TruffleHog) found the key. Check if it was pre-commit or post-commit.
- **Code review:** Key appeared in a pull request or code review. Check if the review was approved.

### Step 4: Assess if the key is live or already rotated

```spl
sourcetype IN ("*github*","*gitlab*","*bitbucket*","*secret_scan*") earliest=-7d
| where match(_raw, "$matched_key_prefix$")
| stats earliest(_time) as first_seen latest(_time) as last_seen count by sourcetype source
| eval exposure_hours=round((last_seen-first_seen)/3600,1)
```

- Determine how long the key has been exposed. Keys exposed in git history may have been available for days or weeks.
- Contact the user to ask if they have already rotated the key.
- Check the AI provider's dashboard for recent usage patterns that might indicate unauthorized access.

### Step 5: Check for unauthorized key usage

```spl
| tstats summariesonly=t count sum(Web.bytes_out) as bytes_out from datamodel=Web where Web.url_domain IN ("api.openai.com","api.anthropic.com","generativelanguage.googleapis.com") earliest=-7d by Web.src Web.user Web.url_domain _time span=1h
| rename Web.* as *
| where user!="$user$"
| stats sum(count) as requests sum(bytes_out) as total_bytes by user src url_domain
| sort -requests
```

- Look for API calls to the exposed provider from unusual sources or users.
- If the key was used from an IP outside your organization, the key has been compromised.

### Step 6: Check git history for exposure duration

```spl
sourcetype="*github*" OR sourcetype="*gitlab*" action="push" user="$user$" earliest=-30d
| search "sk-" OR "sk-proj-" OR "sk-ant-" OR "AIzaSy" OR "hf_" OR "r8_" OR "tog-" OR "gsk_" OR "fw_" OR "xai-" OR "sk-or-"
| table _time source action user repository ref
| sort _time
```

- Identify the first commit that introduced the key.
- Calculate total exposure duration.
- Check if subsequent commits attempted to remove the key (it remains in git history regardless).

### Step 7: Correlate with other risk events

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-033 + AI-006 (API access from non-dev endpoint) = possible key theft and lateral use.
- AI-033 + AI-007 (high volume upload) = key may be used for bulk data exfiltration.
- AI-033 + AI-RISK-004 (privilege escalation + AI) = elevated concern for key misuse.

### Step 8: Make a determination

- **True Positive:** Live API key exposed in a code repository, DLP scan, or log file.
- **Benign True Positive:** Key was already rotated before the detection fired, or the key is a test/sandbox key with no billing or data access.
- **False Positive:** Example key in documentation, test key pattern that matches regex but is not a real key, or a security tool report about OTHER organizations' exposures.

---

## True Positive Indicators

- The matched key format has the correct length and character composition for the provider (not a truncated or partial match).
- The key was found in a code file (`.py`, `.js`, `.env`, `.yaml`, `.json`) rather than documentation.
- The sourcetype is a git push event, indicating the key was committed to version control.
- The repository is public or accessible to a broad audience.
- The AI provider's usage dashboard shows activity that the key owner cannot account for.
- Multiple keys from different providers are exposed in the same commit (indicates systemic credential hygiene issues).
- The exposure was found by a post-commit scanner (GitLeaks, TruffleHog) rather than a pre-commit hook, meaning it already reached the repository.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Example or placeholder keys in documentation (e.g., `sk-proj-example123...`) | High | Check key length and entropy; real keys have specific lengths and high entropy |
| Test/sandbox API keys with no billing or data access | Medium | Verify with user; still recommend rotation as best practice |
| Security tool (TruffleHog, GitLeaks) reporting on other repositories' exposures | Medium | Filter by repository owner; exclude third-party scan reports |
| Rotated keys that were already invalidated before detection | Medium | Verify rotation with provider dashboard; close as benign if confirmed |
| Regex partial match on non-key strings (e.g., `sk-` prefix in unrelated context) | Low | The regex patterns are specific enough that partial matches are rare |
| Key in an encrypted or access-controlled secrets vault being audited | Low | Check if the sourcetype is a vault audit log rather than a plain-text exposure |

## Response Actions

### If confirmed True Positive

1. **IMMEDIATE: Rotate the key.** Contact the user and/or the AI provider to rotate or revoke the exposed key. This is the highest-priority action. Every minute the key remains active is a window for unauthorized use.
2. **Check provider dashboard:** Log into the AI provider's console and review usage for the exposed key. Look for unexpected API calls, new fine-tuned models, or data access from unknown IPs.
3. **Scan git history:** If the key was committed to a repository, use `git log -S "key_prefix"` or a secret scanner to determine the full exposure window. Keys remain in git history even after deletion.
4. **Block the key:** Work with the AI provider to permanently revoke the exposed key, not just rotate it. If the provider supports IP allowlisting, implement it for the replacement key.
5. **Remediate the source:** Remove the key from the codebase. Implement pre-commit hooks (e.g., GitLeaks, detect-secrets) to prevent future commits containing API keys. Move keys to a secrets management system (Vault, AWS Secrets Manager).
6. **Assess data exposure:** If the key was used for unauthorized access, determine what data the attacker could have accessed through the AI provider's API.
7. **Notify:** Inform the user's manager, the security team, and the AI service account owner. For critical-severity keys with evidence of unauthorized use, escalate to IR.

### If Benign True Positive

1. Confirm the key has been rotated or is a test key with no production access.
2. Document the finding and the user's response.
3. If the key was in documentation, recommend replacing it with a clearly fake placeholder.
4. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Exclude known documentation repositories or paths that contain example keys.
- Add regex refinements if certain patterns generate false matches on non-key strings in your environment.
- If your secret scanner generates reports that include keys from external repositories, filter those sourcetypes to only include your organization's repositories.

### Measuring your key exposure baseline

```spl
sourcetype IN ("*github*","*gitlab*","*bitbucket*","*dlp*","*code_review*","*secret_scan*") earliest=-30d
| where match(_raw, "sk-[A-Za-z0-9]|sk-proj-|sk-ant-|AIzaSy|hf_[A-Za-z]|r8_[A-Za-z]|tog-|gsk_|fw_[A-Za-z]|xai-|sk-or-")
| stats count by sourcetype
| sort -count
```

Use this to understand which data sources are generating the most key exposure events and prioritize tuning efforts.

### Adding provider-specific key patterns

If your organization uses AI providers not covered by the current 11 patterns, add new regex patterns to the detection search. Follow the AK-XXX naming convention and assign an appropriate severity level.

### Suppressing known test environments

If you have dedicated test environments where placeholder keys are routinely used:
```
entity_type,entity,provider,scope,expires,owner,reason,enabled,allow_key
system,dev-sandbox-01,*,system,2026-12-31,devops,test_environment_keys,1,dev-sandbox-01|*
```
