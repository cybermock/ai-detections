# AI-007 - High Volume Upload To AI Service

> **Detection ID:** AI-007
> **Status:** Active
> **Risk Score:** 20 / 40 / 60 / 80 (user, tiered) + 20 (system, flat)
> **MITRE ATT&CK:** T1567.002 - Exfiltration Over Web Service: Exfiltration to Cloud Storage (Exfiltration)
> **Data Model:** Web
> **Schedule:** `*/5 * * * *` (every 5 minutes)
> **Lookback Window:** `-15m@m` to `now`
> **Kill Chain Phase:** Actions on Objectives
> **Last Updated:** 2026-02-13

---

## Overview

This detection identifies when a user uploads an unusually high volume of data to AI service providers within a 15-minute window. It uses a tiered risk scoring model that assigns progressively higher risk scores as the upload volume increases, reflecting the escalating probability of significant data exfiltration.

Data exfiltration to AI services is a growing concern because employees may paste or upload large amounts of proprietary data (source code repositories, database exports, documents) into AI chat interfaces or via API calls. Unlike traditional exfiltration vectors that security tools readily flag, AI services are perceived as productivity tools, making uploads appear routine.

The tiered design means low-volume uploads (1-5 MB) generate modest risk that only becomes notable when combined with other detections, while critical-tier uploads (50+ MB) generate enough risk on their own to warrant immediate investigation. This approach reduces alert fatigue while ensuring truly dangerous uploads are surfaced quickly.

## Detection Logic

### Plain English

1. Query the `Web` data model for all web traffic, aggregating by user, source IP, and URL domain.
2. Enrich domains against `ai_provider_domains.csv` via `ai_domains_filter` to identify AI provider traffic.
3. Aggregate bytes_out across all AI domains per user/source/provider combination.
4. Convert bytes_out to megabytes.
5. Load the four tier thresholds from `ai_detection_config.csv` (defaults: 1 MB, 5 MB, 10 MB, 50 MB).
6. Filter out users below the lowest tier (< 1 MB).
7. Assign an upload tier label (low, medium, high, critical) based on which threshold was exceeded.
8. Assign a dynamic risk score based on the tier (20, 40, 60, 80).
9. Exclude sanctioned users/systems via `ai_unsanctioned_filter`.
10. Tag the event with `detection_id=AI-007`.

### Key Fields

| Field | Description | Example Value |
|---|---|---|
| `user` | User performing the upload | `jsmith@corp.com` |
| `src` | Source endpoint | `10.1.2.50` |
| `provider` | AI service provider | `OpenAI` |
| `bytes_out_mb` | Total megabytes uploaded | `12.5` |
| `upload_tier` | Tier classification | `high` |
| `risk_score` | Dynamically assigned risk score | `60` |
| `request_count` | Number of HTTP requests | `145` |
| `ai_domains` | AI domains data was sent to | `api.openai.com` |

### Thresholds

| Parameter | Default Value | Config Lookup Key | Notes |
|---|---|---|---|
| Tier 1 (low) | 1 MB | `upload_tier1_mb` | Risk score 20. Normal conversational usage can reach this. |
| Tier 2 (medium) | 5 MB | `upload_tier2_mb` | Risk score 40. Indicates substantial content submission. |
| Tier 3 (high) | 10 MB | `upload_tier3_mb` | Risk score 60. Likely file uploads or bulk data pasting. |
| Tier 4 (critical) | 50 MB | `upload_tier4_mb` | Risk score 80. Almost certainly intentional bulk data upload. |

## Triage Steps

> **Estimated triage time:** 10-20 minutes

### Step 1: Validate the alert and note the tier

```spl
index=risk search_name="AI RBA - High Volume Upload To AI Service" risk_object="$user$" earliest=-24h
| table _time risk_object risk_score search_name
| sort -_time
```

- Note the risk score to understand the severity tier (20=low, 40=medium, 60=high, 80=critical).
- Check if this user has triggered multiple upload alerts.

### Step 2: Identify the user

```spl
| inputlookup identity_lookup_expanded
| search identity="$user$"
| table identity first_name last_name email managedBy category priority bunit
```

- Determine the user's role and department.
- Users in engineering or data science roles may legitimately upload more data.
- Users in HR, Legal, or Finance uploading large volumes to AI services is more concerning.

### Step 3: Examine the upload details

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out count values(Web.url) as urls values(Web.http_user_agent) as user_agents
  from datamodel=Web where Web.user="$user$" earliest=-1h
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as total_bytes sum(count) as requests values(urls) as urls values(user_agents) as user_agents by provider
| eval mb_out=round(total_bytes/1024/1024,2)
| sort -mb_out
```

- What AI provider is receiving the data?
- Are the uploads going to a web interface (chatgpt.com) or API endpoint (api.openai.com)?
- API uploads suggest programmatic/automated data transfer, which is higher risk.
- What user agent is being used? Curl/Python/scripts indicate automation.

### Step 4: Assess the data sensitivity

```spl
| datamodel Data_Loss_Prevention search
| search DLP.user="$user$" earliest=-24h
| table _time DLP.dlp_type DLP.file_name DLP.dest DLP.action
```

- Check if any DLP policies were triggered for this user around the same time.
- If DLP data is not available, assess based on the user's role and access patterns.

### Step 5: Check for related detections

```spl
index=risk risk_object="$user$" earliest=-24h
| stats sum(risk_score) as total_risk values(search_name) as detections by risk_object
```

- AI-007 + AI-020 (DLP) = confirmed sensitive data upload.
- AI-007 + AI-016 (privileged) = privileged user uploading data.
- AI-007 + AI-015 (after-hours) = potential evasion.
- AI-007 + AI-008 (scripted CLI) = automated exfiltration.

### Step 6: Compare to user's historical baseline

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.user="$user$" earliest=-30d latest=now by Web.url_domain _time span=1d
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| timechart span=1d sum(bytes_out) as daily_bytes
| eval daily_mb=round(daily_bytes/1024/1024,2)
```

- Is today's upload volume abnormal compared to the user's history?
- A sudden spike is more concerning than a consistent pattern.

### Step 7: Make a determination

- **True Positive:** Unsanctioned user uploading significant data volumes to AI services with no business justification.
- **Benign True Positive:** Authorized user with legitimate need (developer using Copilot API, researcher testing models).
- **False Positive:** Bytes_out includes non-content overhead (headers, keepalives) inflating the count. Rare but possible with certain proxy configurations.

---

## True Positive Indicators

- Upload tier is "high" or "critical" (10+ MB in 15 minutes).
- User is not in engineering or a role that would justify large AI interactions.
- API endpoint access rather than web interface.
- User agent indicates automation (curl, python-requests, scripts).
- DLP violations fired for the same user (AI-020).
- Upload occurred outside business hours.
- User recently accessed sensitive file shares, databases, or code repositories.

---

## False Positive Scenarios

| Scenario | Likelihood | Mitigation |
|---|---|---|
| Developer using an AI coding assistant (Copilot, Cursor) with large context windows | High | Allowlist approved developers for the specific provider |
| Data scientist uploading training data to an approved AI API | Medium | Add to allowlist with provider-specific scope |
| Proxy log counting HTTP overhead inflating bytes_out | Low | Validate proxy data accuracy; adjust tier thresholds if needed |
| User uploading images/media to a multimodal AI (Midjourney, DALL-E) | Medium | Consider separate thresholds for media-focused providers |

---

## Response Actions

### If confirmed True Positive

1. **Contain:** Block the user's access to the AI service via proxy policy. If API keys are involved, work with IT to revoke them.
2. **Notify:** Inform the user's manager and data governance team. For critical-tier uploads, notify the CISO.
3. **Investigate:** Determine what data was uploaded. Check recent file access, database queries, and clipboard history if endpoint telemetry is available.
4. **Remediate:** Enforce AI usage training. Implement DLP policies that inspect content destined for AI services. Consider deploying an approved AI gateway with content filtering.
5. **Document:** Record the incident with volume details, tier classification, and data exposure assessment.

### If Benign True Positive

1. Add the user to `ai_sanctioned_entities.csv` with appropriate provider scope and expiration.
2. Document the business justification.
3. Close the notable as "Benign True Positive."

## Tuning Guidance

### Reducing false positives

- Increase tier thresholds if your environment has many developers using AI coding assistants.
- Allowlist specific development teams at the group level for their approved AI providers.
- Consider adjusting the tier 1 threshold from 1 MB to 3-5 MB in environments with heavy AI API usage.

### Measuring current upload volumes to set thresholds

```spl
| tstats summariesonly=t sum(Web.bytes_out) as bytes_out from datamodel=Web
  where Web.url_domain=* earliest=-7d latest=now by Web.user Web.url_domain _time span=15m
| rename Web.* as *
| eval ai_domain=lower(url_domain)
| lookup ai_provider_domains domain as ai_domain OUTPUTNEW provider enabled
| where enabled=1
| stats sum(bytes_out) as window_bytes by user provider _time
| eval window_mb=round(window_bytes/1024/1024,2)
| stats perc50(window_mb) as p50_mb perc90(window_mb) as p90_mb perc95(window_mb) as p95_mb perc99(window_mb) as p99_mb max(window_mb) as max_mb by provider
```

Set tier 1 above the p90 to avoid alerting on normal usage. Set tier 2 at p95, tier 3 at p99, and tier 4 well above the max observed during normal operations.

### Adjusting thresholds

Update `lookups/ai_detection_config.csv`:
```csv
control_name,value,notes
upload_tier1_mb,3,Adjusted tier 1 from 1 to 3 MB based on environment baseline
upload_tier2_mb,10,Adjusted tier 2 from 5 to 10 MB
upload_tier3_mb,25,Adjusted tier 3 from 10 to 25 MB
upload_tier4_mb,100,Adjusted tier 4 from 50 to 100 MB
```

### Adding allowlist entries

To allowlist a development team for all AI providers:
```
entity_type,entity,provider,scope,expires,owner,reason,enabled,allow_key
group,engineering,*,group,2026-12-31,ciso,approved_ai_usage_for_engineering,1,engineering|*
```
